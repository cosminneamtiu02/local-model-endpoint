## Short description

Pydantic models for `Message`, `ModelParams`, the request envelope (`messages`, `model`, `params`, `metadata`), and the response envelope; these models are the single source of truth that drives FastAPI's auto-generated OpenAPI schema.

## Problem

The schemas are the public contract that consumer projects program against. They are how G1 (consumer projects decoupled from the inference backend) actually manifests — the consumer never sees Ollama-specific fields, only the service-owned envelope. They are also the home for G4's `metadata` pass-through field, the placeholder for future per-project attribution (project_id, request_id, trace_id) that v1 reserves space for without implementing. Without this feature, every other feature in the project lacks a contract surface to validate against, encode into FastAPI's OpenAPI, or translate to and from Ollama's API.

*Stakeholder:* Cosmin in his consumer-side-developer role. *Need:* a Python-level type-safe interface that consumer projects can program against, that survives a future backend swap unchanged, and that reserves space for v1's-out-of-scope-but-v2's-needed per-project controls.

## Scope

- **`Message` Pydantic model.** `role: Literal["user", "assistant", "system"]` and `content: str | list[ContentPart]` where `ContentPart` is a discriminated union covering text, image, and audio parts (since the planned Gemma 4 E2B registry entry has all three capability flags). Multipart content supports multimodal inputs from day one.
- **`ModelParams` Pydantic model.** Typed sampling fields per the disambiguated idea: `temperature: float | None`, `top_p: float | None`, `top_k: int | None`, `max_tokens: int | None`, `stop: list[str] | None`, `seed: int | None`, `think: bool = False`. All optional except `think` (default `False`) so `model_dump(exclude_unset=True)` lets the orchestrator merge consumer-supplied values over registry defaults at merge time (E002-F002 owns the merging).
- **`InferenceRequest` envelope.** `messages: list[Message]`, `model: str` (logical name resolved by the registry, never a backend tag), `params: ModelParams = ModelParams()`, `metadata: dict[str, Any] = {}` (pass-through, no schema validation on contents in v1).
- **`InferenceResponse` envelope.** `content: str` (the generated assistant text) plus a structured response-side `metadata` field with eight fields: `model: str` (logical name used), `prompt_tokens: int`, `completion_tokens: int`, `request_id: str` (echoed from request middleware), `latency_ms: int` (full request wall time), `queue_wait_ms: int` (time spent waiting on the semaphore), `finish_reason: Literal["stop", "length", "timeout"]`, `backend: str` (e.g., `"ollama"`).
- **FastAPI OpenAPI exposure.** The schemas are referenced via `response_model=InferenceResponse` on the (future) inference endpoint, so FastAPI's auto-generated `/openapi.json` accurately describes the wire format. F001 ships the schemas; F002 wires them into the route.

## Out of scope

- **Validation against registry capability flags** — e.g., rejecting an audio Message when the model doesn't support audio. That is E002-F003 (capability-flag exposure) used by the orchestrator (E001-F002).
- **The actual HTTP endpoint / FastAPI router** — E001-F002.
- **Sampling-default merging logic** — E002-F002.
- **Streaming response** — out of v1 project scope per the disambiguated idea's Project Boundary.
- **Tool / function calling fields** — out of v1 project scope; envelope intentionally has no `tools` field.
- **Backend-specific (Ollama) request fields** — `template`, `system`, `format`, `keep_alive`, `raw`, etc. live in the adapter (E003-F002), not in the service envelope.
- **Response caching** — out of v1 project scope.

## Acceptance criteria

- `Message(role="user", content="hello")` constructs without error; `model_dump()` returns the expected dict shape.
- `Message(role="user", content=[TextContent(text="…"), ImageContent(url="…")])` accepts the multipart-list form for multimodal inputs.
- `Message(role="invalid", content="x")` raises Pydantic `ValidationError` (role outside the Literal).
- `ModelParams()` constructs with all sampling fields as `None`, `think=False`. `model_dump(exclude_unset=True)` is empty (or near-empty), signalling "consumer overrode nothing" to the orchestrator's merge logic in E002-F002.
- `ModelParams(temperature=2.5)` raises `ValidationError` (out of range 0.0–2.0); `temperature=1.0` succeeds.
- `ModelParams(top_p=1.5)` raises `ValidationError` (out of range 0.0–1.0); `top_p=0.95` succeeds.
- `ModelParams(top_k=0)` raises `ValidationError` (must be `> 0`); `top_k=40` succeeds.
- `ModelParams(max_tokens=0)` raises `ValidationError` (must be `> 0`); `max_tokens=512` succeeds.
- `ModelParams(seed=-1)` raises `ValidationError` (must be `>= 0`); `seed=42` succeeds.
- `InferenceRequest(messages=[], model="x")` raises `ValidationError` (`min_length=1` on `messages`); empty conversations rejected at the boundary.
- `InferenceRequest(messages=[Message(...)], model="x", metadata={"project_id": "abc"})` accepts arbitrary metadata-dict contents (no schema validation on `metadata`'s keys/values in v1).
- `InferenceRequest(messages=[…], model="x", stream=True)` raises `ValidationError` because `extra="forbid"` rejects unknown fields like `stream` (which is out of v1 scope).
- `InferenceResponse` requires all eight response-side metadata fields; constructing one with any field missing raises `ValidationError`.
- `InferenceResponse(content="…", metadata=ResponseMetadata(..., finish_reason="bad"))` raises `ValidationError` — `finish_reason` validates against `Literal["stop", "length", "timeout"]`.
- The auto-generated OpenAPI schema served at `/openapi.json` (verified via `task dev` plus a curl) contains `Message`, `ModelParams`, `ContentPart` (and its three variants), `InferenceRequest`, `InferenceResponse` as named components in `components.schemas`, with `Message.content` rendered as a JSON Schema `oneOf` discriminated union.

## Technical constraints

Per CLAUDE.md's "one class per file" Sacred Rule and the LIP feature's repurposed feature-slice semantics (`model/` for Pydantic value-objects, `schemas/` for wire envelopes), the feature slug is **`inference`** and the file layout is:

```
apps/backend/app/features/inference/
├── model/
│   ├── message.py            # Message
│   ├── text_content.py       # TextContent variant
│   ├── image_content.py      # ImageContent variant
│   ├── audio_content.py      # AudioContent variant
│   ├── content_part.py       # ContentPart discriminated-union type alias
│   └── model_params.py       # ModelParams
└── schemas/
    ├── inference_request.py  # InferenceRequest envelope
    ├── inference_response.py # InferenceResponse envelope (content + metadata: ResponseMetadata)
    └── response_metadata.py  # ResponseMetadata (8 response-side fields)
```

Pydantic v2 config:

- `model_config = ConfigDict(extra="forbid")` on all envelope schemas — unknown fields raise `ValidationError` instead of being silently ignored. Catches consumer bugs at the boundary.
- `Literal[...]` for the `role` enum (`user|assistant|system`) and `finish_reason` enum (`stop|length|timeout`).
- Strict numeric bounds via `Field`: `temperature: float | None = Field(default=None, ge=0.0, le=2.0)`, `top_p: float | None = Field(default=None, gt=0.0, le=1.0)`, `top_k: int | None = Field(default=None, gt=0)`, `max_tokens: int | None = Field(default=None, gt=0)`, `seed: int | None = Field(default=None, ge=0)`.
- `messages: list[Message] = Field(min_length=1)` on `InferenceRequest`.
- `ContentPart` as a discriminated union over `TextContent | ImageContent | AudioContent` using a `type` field discriminator (e.g., `type: Literal["text"|"image"|"audio"]`).

## Open questions

- *[UNRESOLVED]* **ContentPart serialization format for image and audio.** What does `ImageContent` carry — a URL, a base64-encoded blob, a data URI, or several of these? Same for `AudioContent`. The choice depends on what Ollama's Gemma 4 E2B accepts in its multimodal request format, which the translation layer (E003-F002) bridges. Resolvable by checking Ollama API docs during F001 implementation; if multiple formats are reasonable, the schema models them as an internal union.
- *[UNRESOLVED]* **Whether `assistant` and `system` role messages support multipart content.** OpenAI's API restricts multipart content to `user` messages only; `assistant` is always text. LIP could follow that convention (separate Pydantic models per role, with multipart only on `user`) or allow all roles to use either form (simpler, lets backends ignore unsupported combinations). Default leaning: all roles support both forms; role/multimodal compatibility handled in the adapter (E003-F002).

## Unit test scenarios

- Constructing `Message(role="user", content="hello")` succeeds; `model_dump()` returns `{"role": "user", "content": "hello"}` with no extra keys.
- Constructing `Message(role="user", content=[TextContent(text="A"), ImageContent(...)])` accepts the multipart-list form and `model_dump()` returns the discriminated-union JSON shape with each part's `type` field populated.
- Constructing `Message(role="invalid", content="x")` raises `ValidationError`; the error message names `role` and lists the allowed `Literal` values.
- Constructing `ModelParams()` produces an instance where `model_dump(exclude_unset=True)` returns `{}` (or `{"think": False}` only if `think` is treated as set-by-default per Pydantic semantics).
- `ModelParams(temperature=2.5)` raises `ValidationError`; `ModelParams(temperature=2.0)` succeeds; `ModelParams(temperature=0.0)` succeeds; `ModelParams(temperature=-0.1)` raises.
- `ModelParams(top_p=1.5)` raises; `top_p=1.0` succeeds; `top_p=0.0` raises (must be `gt=0`); `top_p=0.001` succeeds.
- `ModelParams(top_k=0)` raises (must be `gt=0`); `top_k=1` succeeds; `top_k=-1` raises.
- `ModelParams(max_tokens=0)` raises; `max_tokens=1` succeeds.
- `ModelParams(seed=-1)` raises (must be `ge=0`); `seed=0` succeeds; `seed=42` succeeds.
- `ModelParams(temperature=None)` succeeds and `model_dump(exclude_unset=True)` returns `{"temperature": None}` (explicit None is "set" per Pydantic v2 `__pydantic_fields_set__`).
- `InferenceRequest(messages=[], model="x")` raises `ValidationError` on `min_length=1`.
- `InferenceRequest(messages=[Message(role="user", content="hi")], model="x")` succeeds with `params=ModelParams()` and `metadata={}` defaulted.
- `InferenceRequest(messages=[...], model="x", metadata={"project_id": "abc", "trace_id": 42})` accepts arbitrary metadata-dict keys/values without validating their content.
- `InferenceRequest(messages=[...], model="x", stream=True)` raises `ValidationError` because `extra="forbid"` rejects the unknown `stream` field.
- `InferenceResponse` requires all 8 metadata fields (`model`, `prompt_tokens`, `completion_tokens`, `request_id`, `latency_ms`, `queue_wait_ms`, `finish_reason`, `backend`); omitting any raises `ValidationError` naming the missing field.
- `InferenceResponse(...metadata=ResponseMetadata(..., finish_reason="bad"))` raises `ValidationError` because `finish_reason` is `Literal["stop", "length", "timeout"]`.
- `ContentPart` discriminated-union: a JSON dict `{"type": "text", "text": "x"}` parses as `TextContent`; `{"type": "image", ...}` parses as `ImageContent`; `{"type": "video", ...}` raises.
- All envelope schemas reject unknown fields (`extra="forbid"`) — verified by sending an unknown field on each: `Message`, `ModelParams`, `InferenceRequest`, `InferenceResponse`, `ResponseMetadata`.

## Contract test scenarios

- `/openapi.json` contains `Message` as a named component schema with `role` enumerated and `content` as a `oneOf` discriminated union.
- `/openapi.json` lists `TextContent`, `ImageContent`, `AudioContent` as component schemas with their `type` discriminator values.
- `/openapi.json` contains `InferenceRequest` as a component with `messages: array(min: 1)`, `model: string`, `params: $ref(ModelParams)`, `metadata: object`.
- `/openapi.json` contains `InferenceResponse` as a component with `content: string` and `metadata: $ref(ResponseMetadata)` listing all 8 metadata fields with their types.
- `/openapi.json` declares `additionalProperties: false` on `InferenceRequest`, `Message`, `ModelParams`, and `ResponseMetadata` (mirroring `extra="forbid"`).
- `/openapi.json` declares `finish_reason: enum["stop", "length", "timeout"]` on `ResponseMetadata`.
- Schemathesis fuzz of envelope schemas via the `/openapi.json` shape produces no examples that the schemas accept but the typed Pydantic models reject (consistency between OpenAPI shape and Pydantic enforcement).
