## Short description

A `POST` endpoint that accepts the envelope, validates it via schemas, looks up the logical model in the registry, calls the adapter with the resolved tag and merged params, and returns a buffered JSON response.

## Problem

F002 is the project's HTTP surface. Without it: F001's envelope schemas exist but no route accepts them; the registry (E002-F001), merge function (E002-F002), and capability check (E002-F003) have no caller; the adapter's `chat()` method (E003-F002) and its failure mapping (E003-F003) are dead code; the DomainError hierarchy (E004-F004) defines exceptions nothing raises. F002 is the orchestration unit that wires every other detailed feature into a single user-visible behavior — POST envelope → JSON response.

This is the strongest expression of the project's purpose: G1 (consumers see only the service-owned envelope, never Ollama's API), G2 (per-model defaults applied transparently inside the orchestrator), G3 (single observability point — every inference passes through this one method, where future tracing and structured logging will hook), G4 (envelope's `metadata` field passes through unchanged, reserving space for future per-project controls). Every other E001/E002/E003/E004 feature exists *to support* this one route.

*Stakeholder:* Cosmin (consumer-side dev hat). *Need:* one stable HTTP endpoint to POST inference requests against, returning typed JSON responses. The contract that consumer projects program against, surviving future backend swaps unchanged.

## Scope

- **Endpoint:** `POST /v1/inference`. Path locked to `/v1/inference` (not `/v1/chat`) because the project's domain language is "inference" (per Project Boundary and Epic Description), and the path stays accurate even when non-chat endpoints (`/api/embeddings` equivalents) are added later. Versioned `/v1` prefix establishes URL-level forward compatibility for future contract changes.
- **`InferenceService` class** in `apps/backend/app/features/inference/service/inference_service.py`. One public method: `async def run(envelope: InferenceRequest, request_id: str) -> InferenceResponse`. Constructor receives `ModelRegistry` and `OllamaClient` instances via DI; class itself holds no mutable state.
- **Orchestration sequence inside `run()`:**
  1. **Capture start time** via `time.perf_counter()` for `latency_ms`.
  2. **Registry lookup** — `model_info = self._registry.get(envelope.model)`. Raises `RegistryNotFoundError` (HTTP 404) if logical name not registered.
  3. **Capability check** — `check_envelope_capabilities(envelope, model_info)`. Raises `ModelCapabilityNotSupportedError` (HTTP 422) on first missing capability.
  4. **Param merge** — `merged = merge_sampling_params(envelope.params, model_info.sampling_defaults)`.
  5. **Adapter call** — `result = await self._client.chat(model_tag=model_info.backend_tag, messages=envelope.messages, params=merged)`. Returns `OllamaChatResult` on success; raises `AdapterConnectionFailureError` (HTTP 502) on any Ollama-specific failure (E003-F003 owns the catch).
  6. **Compute `latency_ms`** = `int((time.perf_counter() - start) * 1000)`.
  7. **Compose response** — assemble `InferenceResponse` with `result.content` and an 8-field `ResponseMetadata` (`model=envelope.model` — *logical* name, not backend tag; `prompt_tokens=result.prompt_tokens`; `completion_tokens=result.completion_tokens`; `request_id`; `latency_ms`; `queue_wait_ms=0` (v1; E004-F001 will replace); `finish_reason=result.finish_reason`; `backend="ollama"`).
- **Router** in `apps/backend/app/features/inference/router/inference_router.py`. One handler, `async def inference(envelope, request, service)`. Body is exactly `return await service.run(envelope, request_id=request.state.request_id)`. Zero business logic in the handler per CLAUDE.md.
- **`OllamaClient` lifecycle: long-lived singleton** constructed in FastAPI's `lifespan` context manager and stored at `app.state.ollama_client`. Closed via `await client.close()` at shutdown. `get_ollama_client()` Depends factory reads from `request.app.state`. This preserves httpx's connection pool and keep-alive across requests; per-request construction would defeat F001-of-E003's pool sizing and bring noticeable connect-handshake overhead per call.
- **`InferenceService` Depends factory** in `app/api/deps.py`: `get_inference_service(registry=Depends(get_registry), client=Depends(get_ollama_client)) -> InferenceService` — constructs a fresh `InferenceService` per request, but the underlying registry and client are both singletons. The service object is stateless; per-request construction is pure overhead-free instantiation.
- **OpenAPI integration:** `@router.post("/inference", response_model=InferenceResponse, responses={404: {"model": ProblemDetails}, 422: {"model": ProblemDetails}, 502: {"model": ProblemDetails}, 500: {"model": ProblemDetails}})`. The `response_model` ensures `/openapi.json` accurately documents the success body; the `responses` dict documents every error body F002 can produce, so consumer-side codegen sees the full contract.
- **Router registration** — `app.include_router(inference_router)` in `app/main.py`'s factory, alongside the existing health router.

## Acceptance criteria

- `POST /v1/inference` with body `{"messages": [{"role": "user", "content": "hi"}], "model": "default-task"}` returns HTTP 200 with `Content-Type: application/json` and a body matching `InferenceResponse`'s schema (`content: str` + `metadata` with all 8 fields populated).
- The 200 response's `metadata.model` equals the *logical* name passed in (`"default-task"`), never the backend tag (`"gemma4:e2b"`). G1 sealed at the response boundary.
- The 200 response's `metadata.backend` equals `"ollama"`.
- The 200 response's `metadata.request_id` matches the `X-Request-ID` response header (both originate from `RequestIdMiddleware`).
- The 200 response's `metadata.latency_ms` is a non-negative integer ≥ 1 (any successful Gemma-4-E2B inference takes ≥ 1 ms wall-clock; the type is `int`, computed from `time.perf_counter()`).
- The 200 response's `metadata.queue_wait_ms` is `0` in v1.
- The 200 response's `metadata.finish_reason` is one of `"stop" | "length"` post-translation (never `"timeout"` in v1 — E004-F003 owns that).
- `POST /v1/inference` with body containing `"model": "nonexistent"` returns HTTP 404, `Content-Type: application/problem+json`, body code `"REGISTRY_NOT_FOUND"`, params `{"model": "nonexistent"}`.
- `POST /v1/inference` with multimodal content where the model lacks the capability (e.g., text-only registry entry receiving an image part) returns HTTP 422, `application/problem+json`, code `"MODEL_CAPABILITY_NOT_SUPPORTED"`, params naming the model and the missing capability.
- `POST /v1/inference` with `messages: []` (empty list) returns HTTP 422, code `"VALIDATION_FAILED"` from F001's `min_length=1` constraint.
- `POST /v1/inference` with an unknown top-level field (e.g., `"stream": true`) returns HTTP 422, code `"VALIDATION_FAILED"` (F001 envelope's `extra="forbid"`).
- `POST /v1/inference` with the Ollama daemon stopped returns HTTP 502, code `"ADAPTER_CONNECTION_FAILURE"`, params `{"backend": "ollama", "reason": "connection_refused"}` — F003-of-E003 mapping fires through unchanged.
- The route handler in `inference_router.py` consists of one statement (`return await service.run(...)`). No `if`/`else`, no `try`/`except`, no logging, no field manipulation in the router.
- `app.include_router` is called for the inference router in `app/main.py`'s factory.
- `app.state.ollama_client` is constructed once during `lifespan` startup and `await ollama_client.close()` runs once during shutdown — not once per request.
- `/openapi.json` lists `POST /v1/inference` with `requestBody` referencing `InferenceRequest`, success `200` referencing `InferenceResponse`, and error `404`, `422`, `502`, `500` each referencing `ProblemDetails`.
- The route is `async def`, never sync. No `run_in_executor`. No mixing of sync and async paths.
- Server bind address is loopback or LAN-only. The startup configuration loads `host` from `Settings.bind_host` (default `127.0.0.1`); binding to `0.0.0.0` is rejected at startup by a Pydantic field validator that raises `ValueError` unless the dedicated escape-hatch setting `allow_public_bind: bool = False` (env var `LIP_ALLOW_PUBLIC_BIND`) is explicitly flipped to `true` (which is never set in v1). Sealed at startup so consumer projects on the `192.168.1.x` LAN reach the service while the public-internet bind is structurally impossible (per `docs/disambiguated-idea.md:148` "local-network only; no `0.0.0.0` binding"; narrative-only in `graphs/LIP/LIP.md:32` until this AC).

**Service implementation pattern:**

```python
# service/inference_service.py
import time

from app.features.inference.repository.ollama_client import OllamaClient
from app.features.inference.schemas.inference_request import InferenceRequest
from app.features.inference.schemas.inference_response import InferenceResponse
from app.features.inference.schemas.response_metadata import ResponseMetadata
from app.features.inference.service.capability_check import check_envelope_capabilities
from app.features.inference.service.model_registry import ModelRegistry
from app.features.inference.service.sampling_merge import merge_sampling_params


class InferenceService:
    def __init__(self, registry: ModelRegistry, client: OllamaClient) -> None:
        self._registry = registry
        self._client = client

    async def run(
        self,
        envelope: InferenceRequest,
        request_id: str,
    ) -> InferenceResponse:
        start = time.perf_counter()
        model_info = self._registry.get(envelope.model)
        check_envelope_capabilities(envelope, model_info)
        merged = merge_sampling_params(envelope.params, model_info.sampling_defaults)
        result = await self._client.chat(
            model_tag=model_info.backend_tag,
            messages=envelope.messages,
            params=merged,
        )
        latency_ms = int((time.perf_counter() - start) * 1000)
        return InferenceResponse(
            content=result.content,
            metadata=ResponseMetadata(
                model=envelope.model,
                prompt_tokens=result.prompt_tokens,
                completion_tokens=result.completion_tokens,
                request_id=request_id,
                latency_ms=latency_ms,
                queue_wait_ms=0,
                finish_reason=result.finish_reason,
                backend="ollama",
            ),
        )
```

**Lifespan + Depends wiring:**

```python
# app/main.py — lifespan addition
from contextlib import asynccontextmanager
from collections.abc import AsyncIterator

from fastapi import FastAPI

from app.api.health_router import router as health_router
from app.core.config import get_settings
from app.features.inference.repository.ollama_client import OllamaClient
from app.features.inference.router.inference_router import router as inference_router


@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncIterator[None]:
    settings = get_settings()
    client = OllamaClient(base_url=settings.ollama_host)
    app.state.ollama_client = client
    try:
        yield
    finally:
        await client.close()


def create_app() -> FastAPI:
    app = FastAPI(title="Local Inference Provider", lifespan=lifespan)
    app.include_router(health_router)
    app.include_router(inference_router)
    # middleware, exception handlers — unchanged
    return app
```

# app/api/deps.py — additions
from fastapi import Depends, Request

from app.features.inference.repository.ollama_client import OllamaClient
from app.features.inference.service.inference_service import InferenceService

**Why lifespan-managed singleton over `Depends(yield)` per request:** httpx's `AsyncClient` keeps a connection pool and keep-alive connections to Ollama. Per-request construction would tear down the pool every call, defeating both pooling and keep-alive — measurable overhead on each inference (TCP handshake to localhost is cheap but non-zero, and it grows with TLS or any future remote backend). One client per process is the documented httpx pattern.

**Async-only route handler.** Per CLAUDE.md "Never write a sync `def` route handler" and "Never use `run_in_executor`". The orchestrator awaits the adapter; everything in the path is non-blocking.

**Pre-existing scaffolding (project-bootstrap, not part of this feature):**

- `RequestIdMiddleware` lives at `apps/backend/app/api/request_id_middleware.py` and sets `request.state.request_id` to either the client-provided `X-Request-ID` header (if it's a valid UUID) or a freshly minted `uuid.uuid4()`. The handler at `inference_router.py` reads from this attribute; the middleware is wired into the FastAPI app via `configure_middleware(app)` (also pre-existing). F002 does not introduce or modify the middleware — it relies on it as a precondition.

**Error pipeline (CLAUDE.md mandate):** the codes `REGISTRY_NOT_FOUND` (HTTP 404), `MODEL_CAPABILITY_NOT_SUPPORTED` (HTTP 422), and `ADAPTER_CONNECTION_FAILURE` (HTTP 502) are defined in [packages/error-contracts/errors.yaml](packages/error-contracts/errors.yaml) under E004-F004's contract. Per CLAUDE.md "Never add an error code without editing errors.yaml, running task errors:generate," before implementing F002 confirm entries exist for these three codes in `errors.yaml`, run `task errors:generate` to regenerate `app/exceptions/_generated/`, and run `task check:errors` to verify the regenerated files match source control. F002 imports the typed exception classes from `app.exceptions` — never from `app.exceptions._generated` directly.

**Vertical-slice import boundary** (mechanically enforced by `architecture/import-linter-contracts.ini`):

- `inference_router` imports from `inference_service` ✓ (router → service).
- `inference_service` imports from `ollama_client`, `model_registry`, `sampling_merge`, `capability_check` ✓ (service → repository/service-helpers).
- `ollama_client`, `model_registry` import from `model/` files only ✓ (repository/service → model).
- `inference_service` imports from `schemas/inference_request`, `schemas/inference_response`, `schemas/response_metadata` ✓ (service composes the response envelope).
- No imports across feature boundaries (the `inference` feature is self-contained except for `app.api.deps`, `app.exceptions`, `app.schemas` — all cross-cutting layers).

The contracts file gets one new entry whose layers list `inference_router → inference_service → (ollama_client | model_registry | sampling_merge | capability_check)`.

**Error propagation discipline:** F002's `run()` does no `try/except`. Every failure mode is already typed by the time it reaches `run()`:

- `RegistryNotFoundError` from `registry.get()` (E002-F001).
- `ModelCapabilityNotSupportedError` from `check_envelope_capabilities` (E002-F003).
- `AdapterConnectionFailureError` from `client.chat()` (E003-F003 wraps every httpx exception).
- `Pydantic ValidationError` is raised before `run()` is even called — FastAPI's request-body parsing produces it from F001's envelope schemas; the F004 exception handler maps it to `VALIDATION_FAILED`.

The orchestrator's purity (no error handling) is load-bearing for testing — every error path is exercised at the dep level, not the orchestrator level.
