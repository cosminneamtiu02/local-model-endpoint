## Short description

A data structure holding logical-name-to-tag mappings plus per-model metadata (sampling defaults, capability flags, context max); ships with exactly one entry (Gemma 4 E2B) and is expandable only by code change.

## Problem

The registry is the single in-process source of truth for per-model knowledge. It is the mechanism behind G1 (consumer projects decoupled from the inference backend) — without it, consumers would embed backend-specific tags like `"gemma4:e2b"` in their code, or the orchestrator would inline per-model defaults inside its routing logic. The registry is also the home for G2 (per-model knowledge in the service) — sampling defaults, capability flags, and context-window limits live in one place so consumer projects do not duplicate or drift on Gemma-specific facts.

Without F001: every consumer must know `"gemma4:e2b"` specifically; every consumer must know Gemma's sampling sweet spots and would silently diverge over time; every consumer must know Gemma's 128K context limit and supported modalities; a future backend swap (Ollama → vLLM / MLX / cloud) requires changing every consumer.

*Stakeholder:* Cosmin (consumer-side developer + maintainer hats). *Need:* a single in-process data structure that maps logical model names to backend-specific tags AND carries the per-model knowledge (sampling defaults, capabilities, context max) the orchestrator + adapter need to do their jobs without consumers caring.

## Scope

- **`ModelInfo` Pydantic model** — one immutable record per registry entry. Fields: `logical_name: str`, `backend: Literal["ollama"]`, `backend_tag: str`, `sampling_defaults: ModelParams` (re-uses F001's envelope type for trivial merging in F002), `capabilities: frozenset[Literal["text", "image", "audio"]]`, `context_max: int = Field(gt=0)`. Pydantic config `frozen=True, extra="forbid"`.
- **`ModelRegistry` class** — holds the in-process map and provides typed lookup. Methods:
    - `__init__(self, entries: dict[str, ModelInfo])` — validates no duplicate `logical_name` and that every dict key matches its entry's `logical_name` field. Failures raise `ValueError` (programmer error caught at startup).
    - `get(self, logical_name: str) -> ModelInfo` — returns the entry or raises `RegistryNotFoundError(model=logical_name)` from F004's DomainError hierarchy.
    - `list_models(self) -> list[ModelInfo]` — returns all entries; used by E006-F002 for state inspection.
- **Initial `default-task` entry for Gemma 4 E2B:**
    ```python
    ModelInfo(
        logical_name="default-task",
        backend="ollama",
        backend_tag="gemma4:e2b",
        sampling_defaults=ModelParams(
            temperature=0.0,    # fully deterministic — cognitive-task workload
            top_p=None,         # irrelevant under greedy decoding
            top_k=None,         # irrelevant under greedy decoding
            max_tokens=None,    # let consumer or backend default decide
            stop=None,
            seed=None,          # not needed at temp=0
            think=False,        # opt-in per request
        ),
        capabilities=frozenset({"text", "image", "audio"}),
        context_max=131072,     # 128K tokens
    )
    ```
- **Logical name format: flat** (no slashes / namespaces). `"default-task"`, not `"text/default-task"`. Locks Epic open question #2.
- **Wired via FastAPI `Depends()` with a per-process singleton** via `@lru_cache(maxsize=1)` on the `get_registry()` factory, mirroring the existing `get_settings()` pattern in `apps/backend/app/api/deps.py`. Per CLAUDE.md, this is the project's *one* allowed DI mechanism.
- **Adding new entries is a code change.** v1 ships exactly one entry. New entries are added by editing `default_models.py` and redeploying. No runtime config; no admin API.

## Out of scope

- **Default-merging logic** (consumer-supplied `params` over registry defaults) — that is E002-F002. F001 only *provides* `sampling_defaults` as a `ModelParams` instance; F002 implements the merge.
- **Capability-flag-based envelope validation** (rejecting an audio Message when the model lacks the audio capability) — that is E002-F003. F001 exposes `capabilities`; F003 implements the check + raises `MODEL_CAPABILITY_NOT_SUPPORTED`.
- **Runtime registry editing / management API** — out of v1; adding a new entry is a code change.
- **Persistent storage of registry state** — registry is in-process; no DB, no files.
- **Multi-model loaded-state tracking** — registry is *just metadata*; tracking which model Ollama currently has loaded in RAM is E006-F002 (state-inspection endpoints).
- **Aliases / multiple logical names per backend tag** — v1 ships one logical name per entry. If aliases are needed later, multiple `ModelInfo` entries can share a `backend_tag`.
- **Versioned models / model rollback** — v1 has no versioning concept. A new Gemma version becomes a new logical_name entry.

## Acceptance criteria

- `ModelInfo(logical_name="x", backend="ollama", backend_tag="y", sampling_defaults=ModelParams(), capabilities=frozenset({"text"}), context_max=131072)` constructs without error.
- `ModelInfo(...)` with `backend="vllm"` raises `ValidationError` (rejected by `Literal["ollama"]`).
- `ModelInfo(...)` with `context_max=0` or negative raises `ValidationError` (rejected by `Field(gt=0)`).
- `ModelInfo(...)` with `capabilities=frozenset({"text", "video"})` raises `ValidationError` (`"video"` is not a member of the Literal).
- `ModelInfo` instances are frozen — assigning to a field after construction raises `ValidationError` (Pydantic `frozen=True`).
- `ModelInfo` rejects unknown fields (`extra="forbid"`) — typos caught at construction.
- `ModelRegistry({"default-task": ModelInfo(logical_name="default-task", ...)}).get("default-task")` returns the entry.
- `ModelRegistry(...).get("nonexistent")` raises `RegistryNotFoundError(model="nonexistent")` (typed error from F004).
- Constructing a `ModelRegistry` with two entries that share a `logical_name` raises a constructor `ValueError` (deduplication validated at startup).
- Constructing a `ModelRegistry` with a dict key that does not match the entry's `logical_name` raises a constructor `ValueError` (key/value consistency).
- `ModelRegistry(...).list_models()` returns all entries.
- The shipped `default-task` entry has sampling defaults `temperature=0.0`, all other `ModelParams` fields `None` except `think=False`; `capabilities=frozenset({"text", "image", "audio"})`; `context_max=131072`; `backend="ollama"`; `backend_tag="gemma4:e2b"`.
- `get_registry()` Depends() factory returns the same `ModelRegistry` singleton across calls within a single FastAPI app instance (one construction per process startup, not per request).

## Dependencies

Two outgoing dependencies surfaced during thickening (the stub list `[]` is replaced with `[LIP-E001-F001, LIP-E004-F004]`):

- **`LIP-E001-F001`** (envelope schemas) — `ModelInfo.sampling_defaults` is typed as `ModelParams`, defined in F001's envelope. Re-using the type makes F002's merge logic trivial: both registry-default and consumer-supplied are `ModelParams` instances merged via `model_dump(exclude_unset=True)`.
- **`LIP-E004-F004`** (DomainError hierarchy) — `ModelRegistry.get()` raises `RegistryNotFoundError`, a typed DomainError defined in F004. Without F004, the registry would have to raise a Python-native `KeyError` and let the orchestrator catch-and-rewrap, which adds boilerplate without value.

Five incoming-dependency features (unchanged from `requirements-elicitation`):

- `LIP-E001-F002` (orchestration) — calls `registry.get(envelope.model)` to resolve the logical name.
- `LIP-E002-F002` (sampling default merging, sibling) — reads `entry.sampling_defaults` and merges with consumer-supplied `params`.
- `LIP-E002-F003` (capability flag exposure, sibling) — reads `entry.capabilities` to validate the request envelope.
- `LIP-E005-F001` (startup warm-up) — calls `registry.get(default_logical_name)` to find the model to warm up at startup.
- `LIP-E007-F001` (README) — documents the registry conceptual model for consumer projects.

## Technical constraints

**File layout** (per CLAUDE.md "one class per file" and repurposed feature-slice semantics):

```
apps/backend/app/features/inference/
├── model/
│   └── model_info.py             # ModelInfo Pydantic value-object
└── service/
    ├── model_registry.py         # ModelRegistry class
    └── default_models.py         # DEFAULT_MODELS: dict[str, ModelInfo] constant (v1 Gemma 4 E2B entry)

apps/backend/app/api/
└── deps.py                       # add get_registry() -> ModelRegistry factory (existing file)
```

`ModelInfo` lives in `model/` (Pydantic value-object). `ModelRegistry` lives in `service/` (provides an API, holds state, encapsulates lookup logic — service-layer behavior). The `DEFAULT_MODELS` constant lives next to the registry that consumes it. The `get_registry()` factory lives in the existing `api/deps.py` alongside `get_settings()`, matching the existing pattern.

**Pydantic config:**

- `ModelInfo`: `model_config = ConfigDict(frozen=True, extra="forbid")` — entries immutable post-construction; unknown fields rejected.
- `ModelInfo.context_max`: `Field(gt=0)` for safety.
- `ModelInfo.capabilities`: `frozenset[Literal["text", "image", "audio"]]` — set semantics in Python; serializes to a JSON array with `uniqueItems: true` in OpenAPI.
- `ModelInfo.backend`: `Literal["ollama"]` — narrow type; future backends (`"vllm"`, `"mlx"`) are deliberate Literal extensions, not arbitrary strings.

**Registry construction invariants (validated at startup):**

- No duplicate `logical_name` across entries.
- Every dict key matches its entry's `logical_name` field.
- Failures raise `ValueError` (programmer error caught at startup, not a runtime DomainError consumers see).

**Singleton wiring (FastAPI Depends pattern):**

```python
# in app/api/deps.py
from functools import lru_cache

@lru_cache(maxsize=1)
def get_registry() -> ModelRegistry:
    return ModelRegistry(DEFAULT_MODELS)
```

`@lru_cache(maxsize=1)` provides a per-process singleton without a global mutable variable. Mirrors the existing `get_settings()` pattern.

**Determinism note (per Q2 lock):**

The `default-task` entry's `sampling_defaults.temperature = 0.0`. Per Ollama's documented behavior (verified via the Context7-fetched Ollama docs during this thickening), `temperature=0` produces fully greedy decoding — same input yields byte-identical output. `top_p`, `top_k`, and `seed` are all `None` because they are mathematically irrelevant under greedy decoding (the argmax token is always picked regardless of nucleus or top-k filtering). If the model ever produces different outputs for the same input under `temp=0`, that is a backend determinism issue (Ollama version mismatch, GPU non-determinism), not a sampling-config issue.
