## Short description

Service-side merge logic where consumer-supplied `params` override the registry's per-model defaults, leaving any fields the consumer did not set filled in from the registry's defaults.

## Problem

The orchestrator (LIP-E001-F002) receives a consumer's `InferenceRequest` with `params: ModelParams` (most fields likely `None` — for cognitive-task consumers, defaults are usually fine). It also has the registry entry's `sampling_defaults: ModelParams` (carrying the per-model recommended values, e.g., `temperature=0.0` for the `default-task` entry). To call the adapter with a single coherent set of sampling parameters, the orchestrator needs *one* `ModelParams` that combines both sides — consumer overrides where set, registry defaults elsewhere.

Without F002:

- The orchestrator inlines the merge logic, conflating routing with parameter resolution.
- Or every consumer has to send a fully-populated `params` (defeats the registry's "single source of truth for defaults" property).
- Or the adapter does the merging (puts per-model knowledge in the adapter, defeating G2's encapsulation).

*Stakeholder:* Cosmin in the consumer-side-developer hat — defaults work without him having to know each model's sampling sweet spots, while explicit overrides are honored when needed. *Need:* a pure, type-safe function that takes consumer + registry params and produces the resolved per-request `ModelParams` the adapter actually receives.

## Scope

- **`merge_sampling_params(consumer: ModelParams, registry_defaults: ModelParams) -> ModelParams` function** — pure function in `apps/backend/app/features/inference/service/sampling_merge.py`. No state, no side effects, deterministic.
- **Merge semantics: consumer-set wins; consumer-unset falls through to registry; both-unset stays unset.** Implementation uses Pydantic v2's `consumer.model_dump(exclude_unset=True)` to identify only the fields the consumer explicitly passed; those override the full `registry_defaults.model_dump()` picture.
- **Override semantics for list-typed fields (`stop`):** Consumer's list fully *replaces* registry's list. Registry `stop=["</answer>"]` plus consumer `stop=["DONE"]` produces merged `stop=["DONE"]`. No concatenation. (A consumer who wants both must pass both: `stop=["DONE", "</answer>"]`.) This matches OpenAI/Anthropic/Ollama convention.
- **Explicit `None` from consumer = deliberate opt-out.** A consumer passing `temperature=None` explicitly produces `temperature=None` in the merged result, regardless of registry's value. This falls out automatically from Pydantic's `exclude_unset` semantics — explicit `None` is "set", consumer-omitted is "unset".
- **Pure function — no input mutation.** Returns a fresh `ModelParams` instance. Consumer and registry instances are unchanged; safe to call concurrently from multiple async tasks.
- **Type-safe boundary.** Inputs are two typed `ModelParams`; output is one typed `ModelParams`. Internal dict-merging is an implementation detail, not the API surface — callers never see a raw `dict`.

## Out of scope

- **Bound validation** (rejecting `temperature=2.5`, etc.) — already enforced at envelope construction in `LIP-E001-F001` via Pydantic `Field(ge=0.0, le=2.0)`. F002 operates on already-validated `ModelParams` instances and trusts both inputs.
- **Capability validation** (rejecting consumer's `image` content if the model doesn't support it) — that is `LIP-E002-F003`. F002 only merges sampling params, not message content.
- **Backend-specific param translation** — Ollama may use different field names. Translation lives in `LIP-E003-F002`. F002 produces `ModelParams` (service-internal type), not backend-specific dicts.
- **Concatenation of `stop` sequences** — full override is the documented semantics; concatenation is rejected.
- **`think` mode special handling** — `think` is a regular boolean field that merges like any other (consumer-set wins). No opt-in/opt-out logic.
- **Registry lookup** — F002 takes `registry_defaults` as a parameter. Looking up the right entry by `logical_name` is the orchestrator's job (calls `registry.get(model_name)` from F001).
- **Per-request param reflection** — F002 does not expose what params were used in the response. The response metadata's `model: str` (logical name) is set by the orchestrator, not by F002.
- **Caching of merged results** — every call does a fresh merge. The merge is sub-microsecond; caching is premature optimization for the project's actual scale (≤4 serial consumers, hundreds of requests per session at most).

## Acceptance criteria

- `merge_sampling_params(ModelParams(), ModelParams(temperature=0.0))` returns a `ModelParams` instance with `temperature=0.0` (consumer overrode nothing → registry default falls through).
- `merge_sampling_params(ModelParams(temperature=0.5), ModelParams(temperature=0.0))` returns `temperature=0.5` (consumer override wins).
- `merge_sampling_params(ModelParams(temperature=None), ModelParams(temperature=0.0))` returns `temperature=None` (explicit consumer-`None` overrides registry's 0.0 — opt-out semantics).
- `merge_sampling_params(ModelParams(top_p=0.9), ModelParams(temperature=0.0))` returns merged with `temperature=0.0` and `top_p=0.9` (multi-field — consumer wins where set, registry fills elsewhere).
- `merge_sampling_params(ModelParams(stop=["DONE"]), ModelParams(stop=["</answer>"]))` returns `stop=["DONE"]` (full replace, not concatenation).
- `merge_sampling_params(ModelParams(), ModelParams())` returns a `ModelParams` with all sampling fields `None` and `think=False` (both empty → defaults all the way down).
- `merge_sampling_params(ModelParams(think=True), ModelParams(think=False))` returns `think=True` (consumer override wins for boolean field).
- The function does not mutate either input — consumer's `__pydantic_fields_set__` is unchanged after the call; the returned object is a fresh instance, not an alias of either input.
- The merged result satisfies all `ModelParams` field bounds. Pydantic re-validation on `ModelParams(**merged)` construction would raise `ValidationError` if the merged dict somehow violated bounds; in practice both inputs are pre-validated so the result is always valid, but the re-validation is a free safety net.

## Technical constraints

**File layout** (per CLAUDE.md "one class per file" — note: this is a *function*, not a class):

```
apps/backend/app/features/inference/service/
└── sampling_merge.py         # merge_sampling_params(consumer, registry_defaults) -> ModelParams
```

`sampling_merge.py` exports a single public function `merge_sampling_params`. No class definition; the file holds the function plus imports.

**Implementation pattern:**

```python
from app.features.inference.model.model_params import ModelParams


def merge_sampling_params(
    consumer: ModelParams,
    registry_defaults: ModelParams,
) -> ModelParams:
    """Merge consumer-supplied params over registry defaults.

    Consumer-set fields (whether to a value or explicitly to None) override
    registry defaults. Consumer-unset fields fall through to the registry's
    value. List-typed fields (e.g., stop) follow replace semantics, not
    concatenation.
    """
    consumer_overrides = consumer.model_dump(exclude_unset=True)
    registry_dict = registry_defaults.model_dump()
    merged = {**registry_dict, **consumer_overrides}
    return ModelParams(**merged)
```

**Pydantic semantics relied on:**

- `model_dump(exclude_unset=True)` returns only fields the consumer *explicitly passed* (whether to a value or to `None`). Pydantic v2 tracks "set vs unset" via `__pydantic_fields_set__`, making this distinction exact and reliable.
- Full `model_dump()` on `registry_defaults` includes all fields with their current values (including explicit `None`s).
- Re-construction via `ModelParams(**merged)` re-runs Pydantic field bounds — if the merged result somehow violated bounds, it would raise `ValidationError`.

**Pure-function discipline:**

- No global state read or written.
- No mutation of inputs. The `model_dump → dict-merge → fresh construct` pattern guarantees this — Pydantic's `model_dump` returns a new dict copy, the dict-merge produces a new dict, and `ModelParams(**merged)` constructs a new instance.
- No I/O. No logging in the merge body itself.
- Deterministic — same inputs always produce identical output.

## Unit test scenarios

- `merge_sampling_params(ModelParams(), ModelParams(temperature=0.0))` returns `temperature=0.0` (registry default falls through; consumer overrode nothing).
- `merge_sampling_params(ModelParams(temperature=0.5), ModelParams(temperature=0.0))` returns `temperature=0.5` (consumer override wins).
- `merge_sampling_params(ModelParams(temperature=None), ModelParams(temperature=0.0))` returns `temperature=None` (explicit consumer-`None` overrides registry value — opt-out semantics via Pydantic `exclude_unset=True`).
- `merge_sampling_params(ModelParams(top_p=0.9), ModelParams(temperature=0.0))` returns `temperature=0.0, top_p=0.9` (multi-field merge: each field independent).
- `merge_sampling_params(ModelParams(stop=["DONE"]), ModelParams(stop=["</answer>"]))` returns `stop=["DONE"]` (full replace; not concatenation).
- `merge_sampling_params(ModelParams(), ModelParams())` returns a `ModelParams` with all sampling fields `None`, `think=False`.
- `merge_sampling_params(ModelParams(think=True), ModelParams(think=False))` returns `think=True`.
- The function does not mutate either input — verified by checking `consumer.__pydantic_fields_set__` before and after, plus verifying `id(result) != id(consumer) and id(result) != id(registry_defaults)`.
- The merged result re-validates: passing the merged dict back through `ModelParams(**merged)` does not raise.
- Calling `merge_sampling_params` with `consumer=None` (positional) raises `TypeError` at the function-signature level (Pyright strict catches at build time; runtime test verifies the type-check produces a meaningful error).
