## Short description

Each registry entry's text/image/audio capability flags are readable by the orchestrator so the service can reject envelopes that request modalities the model does not support.

## Problem

The orchestrator (LIP-E001-F002) needs to reject inference requests whose content the model cannot actually serve. Example: a consumer sends a `Message` with an `ImageContent` part, but the registry entry for the requested model only declares `capabilities=frozenset({"text"})`. Without F003, the orchestrator either (a) sends the request to Ollama anyway, getting a confusing backend-specific error mid-inference; (b) has no awareness of capabilities and the mismatch surfaces as a generic 5xx; or (c) reimplements the capability scan inline.

F003 provides the capability check at the service boundary, producing a structured `MODEL_CAPABILITY_NOT_SUPPORTED` error response (defined by F004) with typed params naming both the model and the requested-but-unsupported capability. Maps to G2 (per-model knowledge in the service — capabilities are part of that knowledge) and G3 (structured error responses are the consumer-visible part of observability).

*Stakeholder:* Cosmin in consumer-side-developer hat. *Need:* when a consumer sends content the model cannot process, the response is a typed structured error at the service boundary — fast, predictable, programmable. Not a confusing backend error mid-inference.

## Scope

- **`check_envelope_capabilities(envelope: InferenceRequest, model_info: ModelInfo) -> None` function** — pure function in `apps/backend/app/features/inference/service/capability_check.py`. Returns `None` on success; raises `ModelCapabilityNotSupportedError` (from F004) if the envelope requires capabilities the model lacks.
- **Required-capability scan over the envelope.** Internal helper `_required_capabilities(envelope) -> Iterator[str]` walks `envelope.messages → message.content` and yields required capability names (`"text"`, `"image"`, `"audio"`) in envelope-encounter order. If `message.content` is a `str` (the simple case), yields `"text"`. If `list[ContentPart]` (multimodal), yields each part's `type` discriminator field per the F001 envelope's discriminated-union schema.
- **Set-membership check against `model_info.capabilities`.** For each yielded capability, check `if capability not in model_info.capabilities` — short-circuit raise on the first mismatch.
- **First-encountered ordering.** When multiple capabilities are missing, the error reports the first one detected in envelope-scan order. This matches how the consumer constructed the request, making "fix-and-retry" debugging intuitive — the consumer sees the first thing in their request that the model can't serve.
- **Pure function — no input mutation.** Reads `envelope` and `model_info`; raises or returns. No side effects, no state.

## Out of scope

- **Exposure of capabilities for non-validation purposes** — e.g., a `/state` endpoint listing what each model can do is `LIP-E006-F002`'s territory (state-inspection endpoints). F003 only does per-request validation.
- **Validation of message structure** (role validity, ContentPart shape) — that is `LIP-E001-F001`'s envelope-validation territory; capability validation only checks the *types* of content, not the validity of any individual ContentPart's fields.
- **Multi-error collection** — F003 raises on the first missing capability and short-circuits. Doesn't accumulate all missing capabilities into a single error response.
- **Capability negotiation** ("this model can't do audio but here's a different model that can") — out of v1.
- **Tool / function-calling capability** — out of project scope.
- **Streaming capability** — out of project scope.
- **Conditional / fine-grained capabilities** ("supports image but only ≤1024 px", "supports audio but only WAV") — out of v1; capabilities are coarse-grained set membership.
- **Mutability of capabilities at runtime** — `ModelInfo.capabilities` is `frozenset` on a `frozen=True` model; F003 only reads.
- **Pre-computing required-capabilities at envelope-construction time** — the scan happens at request time, not at envelope validation. Optimization-only choice; deferred unless profiling shows it matters.

## Acceptance criteria

- `check_envelope_capabilities(envelope, model_info)` with `envelope.messages=[Message(role="user", content="hello")]` and `model_info.capabilities=frozenset({"text"})` returns `None`.
- Same call with `model_info.capabilities=frozenset()` raises `ModelCapabilityNotSupportedError(model=<logical_name>, requested_capability="text")` — model has zero capabilities; text content is not supportable.
- With `envelope.messages=[Message(content=[ImageContent(...)])]` and `capabilities=frozenset({"text"})`, raises `requested_capability="image"`.
- With `content=[TextContent, ImageContent]` and `capabilities=frozenset({"text"})`, raises `requested_capability="image"` — text passes, image is first missing.
- With `content=[ImageContent, AudioContent]` and `capabilities=frozenset({"text"})`, raises `requested_capability="image"` — image first-encountered; audio surfaces on retry after consumer fixes image.
- With `content=[ImageContent, AudioContent]` and `capabilities=frozenset({"text", "image", "audio"})`, returns `None`.
- Multi-message envelope: msg 0 = text only, msg 1 = audio; model `capabilities=frozenset({"text", "image"})` → raises `requested_capability="audio"` (audio encountered in msg 1; first missing in scan order).
- `check_envelope_capabilities` does not mutate either input — `envelope.messages`, `model_info.capabilities`, and all nested objects are unchanged after the call.

## Technical constraints

**File layout:**

```
apps/backend/app/features/inference/service/
└── capability_check.py        # check_envelope_capabilities + private _required_capabilities helper
```

**Implementation pattern:**

```python
from collections.abc import Iterator

from app.exceptions import ModelCapabilityNotSupportedError
from app.features.inference.model.model_info import ModelInfo
from app.features.inference.schemas.inference_request import InferenceRequest


def check_envelope_capabilities(
    envelope: InferenceRequest,
    model_info: ModelInfo,
) -> None:
    """Raise ModelCapabilityNotSupportedError if envelope requires capabilities
    the model lacks. Returns None on success.
    """
    for capability in _required_capabilities(envelope):
        if capability not in model_info.capabilities:
            raise ModelCapabilityNotSupportedError(
                model=model_info.logical_name,
                requested_capability=capability,
            )


def _required_capabilities(envelope: InferenceRequest) -> Iterator[str]:
    """Yield required capabilities in envelope-encounter order (no dedup)."""
    for msg in envelope.messages:
        if isinstance(msg.content, str):
            yield "text"
        else:
            for part in msg.content:
                yield part.type   # ContentPart.type discriminator: "text" | "image" | "audio"
```

**Iterator-based scan with short-circuit:** the helper yields capabilities one at a time; the check loop raises on the first missing. No unnecessary work after a violation is detected. Deduplication isn't needed — the same capability appearing twice still passes/fails the same `in` check.

**Reading `part.type` (discriminator) over `isinstance(part, ImageContent)`:** when F001-E001 ever extends `ContentPart` with new variants (e.g., a future `VideoContent` via `Literal[..., "video"]` extension), F003 picks up the new variant automatically by reading the discriminator. `isinstance` would silently miss new variants until updated.

**Pure-function discipline:** no I/O, no mutation of inputs, deterministic. Same shape as F002.

**Error pipeline (CLAUDE.md mandate):** the code `MODEL_CAPABILITY_NOT_SUPPORTED` (HTTP 422) is defined in [packages/error-contracts/errors.yaml](packages/error-contracts/errors.yaml) under E004-F004's contract with params `{model: str, requested_capability: str}`. Per CLAUDE.md "Never add an error code without editing errors.yaml, running task errors:generate," before implementing F003 confirm the entry exists in `errors.yaml`, run `task errors:generate`, and run `task check:errors`. F003 imports `ModelCapabilityNotSupportedError` from `app.exceptions` — never from `app.exceptions._generated` directly.

## Unit test scenarios

- `check_envelope_capabilities(envelope, model_info)` with a single text-only message and a model with `capabilities={"text"}` returns `None`.
- Same call with `model_info.capabilities=frozenset()` raises `ModelCapabilityNotSupportedError(model=<logical_name>, requested_capability="text")`.
- A multipart envelope with a single `ImageContent` part and `capabilities={"text"}` raises with `requested_capability="image"`.
- A multipart envelope with `[TextContent, ImageContent]` and `capabilities={"text"}` raises with `requested_capability="image"` (text passes; image is the first missing).
- A multipart envelope with `[ImageContent, AudioContent]` and `capabilities={"text"}` raises with `requested_capability="image"` (image first-encountered).
- Same envelope with `capabilities={"text", "image", "audio"}` returns `None`.
- A two-message envelope where msg 0 = text and msg 1 = audio with `capabilities={"text", "image"}` raises with `requested_capability="audio"` (audio in msg 1 is first missing in scan order).
- Mutation invariants: `check_envelope_capabilities(envelope, model_info)` does not modify either input. Verified by deep-comparing serialized forms before and after the call.
- Empty messages list (caught earlier by F001 of E001's `min_length=1`, but defensive test): does not raise from F003 because the iterator over zero messages yields nothing.
- The internal `_required_capabilities` iterator is short-circuit; verified by mocking the iterator and asserting the loop stops at the first mismatch (no further iteration).
- Future `Literal` extension safety: adding a hypothetical `"video"` ContentPart variant produces `requested_capability="video"` from the discriminator without code changes to `_required_capabilities`.
