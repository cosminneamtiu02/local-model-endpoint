## Short description

`OLLAMA_HOST` setting (default `http://localhost:11434`) drives an `httpx.AsyncClient` whose lifecycle matches the FastAPI app's lifespan, replacing any hardcoded host string.

## Problem

The adapter epic (LIP-E003) is the seam where Ollama-specific knowledge is concentrated. F001 is its foundation: a configured `httpx.AsyncClient` that talks to Ollama, with its lifecycle managed by FastAPI's lifespan. Without F001:

- Each sibling (F002 translation, F003 failure mapping) would construct its own `httpx.AsyncClient` — wasteful (one client per app is the right shape for connection pooling and lifecycle).
- Or each sibling would import a global module-level client — forbidden by CLAUDE.md's "no global singletons" rule.
- Or the orchestrator would pass an httpx client through layers it shouldn't know about — leaks the HTTP-library choice up the stack.

F001 provides one well-managed `httpx.AsyncClient` per app instance, configured at startup, closed at shutdown, exposed via FastAPI `Depends()`. Maps to G1 (configurable backend host enables future swaps and test isolation) and G6 (lifespan-managed lifecycle aligns with the on-demand service posture — client comes up at startup, comes down at shutdown).

*Stakeholder:* Cosmin (operator + maintainer hats). *Need:* a single well-managed httpx client per app instance, with the host configurable so test fixtures can point at a stub Ollama and future deployments can target a different daemon (e.g., when scaling out to a beefier Mac on the same LAN).

## Scope

- **`OLLAMA_HOST` Settings field** — confirmed and locked. The `ollama_host: str = "http://localhost:11434"` field on `Settings` already exists post-bootstrap; F001 spec-locks the contract: type `str`, default the local Ollama port, no validator beyond pydantic-settings defaults. Invalid URLs fail at httpx construction time, not at settings validation.
- **`OllamaClient` class** in `apps/backend/app/features/inference/repository/ollama_client.py`:
    - `__init__(self, base_url: str, timeout: httpx.Timeout = DEFAULT_TIMEOUT)` — constructs an internal `httpx.AsyncClient` with the given base URL and timeout.
    - `async def close(self) -> None` — closes the underlying httpx client; must be called at shutdown.
    - Async context-manager support: `async def __aenter__(self) -> Self` and `async def __aexit__(...)` so test code can use `async with OllamaClient(...) as client: ...` for guaranteed cleanup.
    - Internal `_client: httpx.AsyncClient` (single underscore — module-private). Sibling features should use the `_request` method below; direct `_client` access is permitted only when the wrapper genuinely needs more httpx surface than `_request` provides.
    - `async def _request(self, method: str, path: str, *, json: dict[str, Any] | None = None) -> httpx.Response` — thin wrapper forwarding to `self._client.request(method, path, json=json)`. The only point siblings touch httpx at; F002's typed methods are built on top of this.
- **App lifespan integration** in `apps/backend/app/main.py`:
    - At lifespan startup: construct `OllamaClient(base_url=settings.ollama_host)` and store at `app.state.ollama_client`.
    - At lifespan shutdown: `await app.state.ollama_client.close()` inside a `finally:` so it runs even if the app body raises.
    - Replaces the current no-op lifespan body. The "warm-up dummy inference" comment stays — that work is layered on top in `LIP-E005-F001`.
- **`get_ollama_client()` Depends factory** in `apps/backend/app/api/deps.py`:
    - Reads `app.state.ollama_client` set by lifespan (cannot use `@lru_cache` like `get_registry()` because the client is per-app-instance, not per-process).
    - Mirrors the existing factory pattern in `deps.py`.
- **Default `httpx.Timeout` configuration:** `httpx.Timeout(connect=5.0, read=None, write=None, pool=None)`. 5-second connect timeout surfaces dead daemons fast; no read/write/pool timeout because `asyncio.wait_for(180s)` in `LIP-E004-F003` is the *actual* per-request cap. Setting httpx read timeout would create two competing caps; cleaner to let the orchestrator's `wait_for` be the only one.
- **Default `httpx.Limits` configuration:** httpx defaults (`max_connections=100`, `max_keepalive_connections=20`). LIP's serial-inference profile (1 active + ≤4 waiters) doesn't stress these; tuning them is premature.

## Out of scope

- **API-specific methods** (`chat`, `generate`, `embeddings`, `tags`) — `LIP-E003-F002`'s territory. F001 only provides low-level `_request`.
- **Failure mapping** (httpx exceptions → DomainError) — `LIP-E003-F003`. F001 lets httpx exceptions propagate as-is.
- **Per-request timeout enforcement** — `LIP-E004-F003`'s `asyncio.wait_for`. F001's httpx transport-layer timeout is just connect-level.
- **Backend swapping** — F001 is Ollama-specific by name and file path. Future backends (vLLM, MLX, cloud) become new sibling files in `repository/` (e.g., `vllm_client.py`), not modifications to F001.
- **Authentication / TLS / HTTPS** — Ollama runs locally on `http://`, no auth in v1. Local-network trust per disambiguated-idea constraints.
- **Test-connection / liveness ping at startup** — F001 doesn't ping Ollama at lifespan startup. The warm-up dummy inference (`LIP-E005-F001`) is the right place because it both tests connectivity *and* loads the model into Ollama's memory.
- **Connection retry / backoff logic** — F001 doesn't retry. Connection failures propagate as httpx exceptions; F003 maps them to `ADAPTER_CONNECTION_FAILURE`.
- **Pool tuning beyond httpx defaults** — defaults are fine for LIP's serial-inference profile.
- **Observability hooks** (httpx event hooks for request/response logging) — out of v1 per the disambiguated idea's "structured logging emission deferred."

## Acceptance criteria

- Setting `OLLAMA_HOST=http://192.168.1.50:11434` env var causes the constructed `OllamaClient`'s internal httpx base URL to match.
- With `OLLAMA_HOST` unset, the client's base URL is `http://localhost:11434` (the pydantic-settings default).
- `OllamaClient(base_url, timeout).close()` properly closes the underlying httpx client — no `unclosed AsyncClient` warning at GC time.
- `async with OllamaClient(base_url, timeout) as client: ...` works as an async context manager — `close()` runs on exit, including on exceptions raised inside the `with` body.
- `OllamaClient._request("GET", "/api/version")` returns an `httpx.Response` and forwards directly to httpx (no custom code paths).
- `OllamaClient._request("POST", "/api/chat", json={...})` sends the JSON body and returns an `httpx.Response`.
- App startup constructs `OllamaClient(base_url=settings.ollama_host)` and stores it on `app.state.ollama_client`.
- App shutdown calls `await app.state.ollama_client.close()` from a `finally:` block so it runs even if the app body raises an exception.
- `get_ollama_client()` Depends factory returns the same `OllamaClient` instance across multiple requests within the same FastAPI app instance — verified by `id()` equality across two dependency-injected clients.
- `httpx.Timeout` config matches `connect=5.0, read=None, write=None, pool=None`.
- Connection failures (e.g., Ollama daemon not running) raise `httpx.ConnectError` uncaught — F001 lets it propagate; mapping to `ADAPTER_CONNECTION_FAILURE` is F003's job.

## Technical constraints

**File layout:**

```

└── ollama_client.py              # OllamaClient class + DEFAULT_TIMEOUT module constant

apps/backend/app/main.py           # lifespan body updated to manage the client
apps/backend/app/api/deps.py       # get_ollama_client(request: Request) -> OllamaClient added
```

**Implementation pattern:**

```python
# repository/ollama_client.py
from __future__ import annotations

from typing import Any, Self

import httpx

DEFAULT_TIMEOUT = httpx.Timeout(connect=5.0, read=None, write=None, pool=None)


class OllamaClient:
    """Lifecycle-managed httpx.AsyncClient for talking to Ollama.

    Constructed at app lifespan startup; closed at lifespan shutdown.
    Sibling features (LIP-E003-F002 translation, LIP-E003-F003 failure mapping)
    use the low-level _request method; direct _client access is permitted only
    when the wrapper genuinely needs more httpx surface than _request provides.
    """

    def __init__(self, base_url: str, timeout: httpx.Timeout = DEFAULT_TIMEOUT) -> None:
        self._client = httpx.AsyncClient(base_url=base_url, timeout=timeout)

    async def close(self) -> None:
        await self._client.aclose()

    async def __aenter__(self) -> Self:
        return self

    async def __aexit__(self, *_: object) -> None:
        await self.close()

    async def _request(
        self,
        method: str,
        path: str,
        *,
        json: dict[str, Any] | None = None,
    ) -> httpx.Response:
        return await self._client.request(method, path, json=json)
```

```python
# main.py — lifespan body
@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncGenerator[None]:
    settings = get_settings()
    app.state.ollama_client = OllamaClient(base_url=settings.ollama_host)
    try:
        yield
    finally:
        await app.state.ollama_client.close()
```

```python
# deps.py
from fastapi import Request

def get_ollama_client(request: Request) -> OllamaClient:
    return request.app.state.ollama_client
```

**Pyright considerations:** `app.state` is `Any`-typed by default. The `request.app.state.ollama_client` access loses typing inside the factory body, but the type is recovered at call sites via `Annotated[OllamaClient, Depends(get_ollama_client)]` — siblings see properly-typed `OllamaClient`.

**Lifecycle discipline:**

- All public methods are `async`.
- No I/O at module-import time (constructor doesn't ping Ollama; first `_request` is the first network call).
- No exception catching — exceptions from httpx propagate to F003 for typed mapping.

**No retry logic, no test-connect, no health-check, no observability hooks** — all out of scope per the section above.

## Unit test scenarios

- `OllamaClient(base_url="http://localhost:11434")` constructs without error; the internal `httpx.AsyncClient` has its `base_url` set to that host.
- `OllamaClient(base_url=...)` configures `httpx.Timeout(connect=5.0, read=None, write=None, pool=None)` on the underlying client.
- `OllamaClient` is async-context-manager-compatible: `async with OllamaClient(...) as client:` enters returning the client and exits calling `await client.close()`.
- `await client.close()` is idempotent — calling twice does not raise.
- `client._request("GET", "/api/tags", json=None)` constructs the right `httpx` call and returns the response (verified via `httpx.MockTransport`).
- `client._request("POST", "/api/chat", json={"model": "x"})` sends the JSON body.
- The `Settings.ollama_host` field defaults to `"http://localhost:11434"` when no `OLLAMA_HOST` env var is set.
- `OLLAMA_HOST=http://192.168.1.50:11434` env var is reflected in `Settings.ollama_host`.
- The `_request` helper is private (single leading underscore); not in the public class API surface.

## Integration test scenarios

- `httpx.MockTransport` configured to respond to `GET /api/tags` with `{"models": []}` allows `OllamaClient` to round-trip a real `_request` call without a live Ollama. (Validates the `_request` plumbing as a unit + the MockTransport pattern other features will reuse.)
- The lifespan-managed singleton (introduced by F002 of E001) constructs exactly one `OllamaClient` at app startup and `await close()`s it exactly once at shutdown — verified by counting constructor calls and close() calls.
- `app.state.ollama_client` survives across multiple requests within a single `TestClient` lifespan; verified via `id(client)` identity across two route invocations.
