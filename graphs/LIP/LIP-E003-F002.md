## Short description

Translates the service's envelope (with registry-resolved tag and merged params) into Ollama's expected request format on the way out, and translates Ollama's response back into the service-owned response schema on the way back.

## Problem

The translation layer is what makes G1 actually work. Consumer projects send LIP's typed envelope (`InferenceRequest` with `Message[]`, `model`, `params`, `metadata`); Ollama wants its own JSON shape (`{model, messages, options, stream, ...}`). Without F002, the orchestrator either inlines the translation (couples it to one backend), reaches into the adapter's internals (defeats encapsulation), or sends the envelope to Ollama directly (fails because the field shapes do not match).

F002 is the actual translation layer. It sits on F001's `OllamaClient._request` foundation and provides typed methods that consume `InferenceRequest`-shaped inputs and produce `OllamaChatResult`-shaped outputs. When G1 fires (Ollama → vLLM/MLX swap), F002 is the unit replaced. The orchestrator's contract with the adapter stays the same: same input types, same output type, different `chat()` implementation.

*Stakeholder:* Cosmin (consumer-side dev + maintainer hats). *Need:* inference requests reach Ollama in its expected format and Ollama's responses come back in the service's expected format, all in one well-encapsulated translation layer that is the swap point for future backends.

## Scope

- **`OllamaChatResult` Pydantic value-object** — intermediate type returned by `chat()`. Fields: `content: str`, `prompt_tokens: int`, `completion_tokens: int`, `finish_reason: Literal["stop", "length", "timeout"]`. `frozen=True, extra="forbid"`. The orchestrator (E001-F002) composes the full `InferenceResponse` from this plus its own metadata (`request_id`, `latency_ms`, `queue_wait_ms`, `logical_name`, `backend="ollama"`). F002 stays focused on what it can know about.
- **`OllamaClient.chat(*, model_tag, messages, params) -> OllamaChatResult`** async method added to F001's existing class. Locks Epic open question #1: **single high-level `chat()` method**, not multiple per-endpoint methods. Future Ollama endpoints (`/api/embeddings`) become new methods on the same class if/when LIP needs them — `/api/generate` and `/api/embeddings` are out of v1 scope.
- **Outbound translation: service envelope → Ollama `/api/chat` request body:**
    - `model_tag` → Ollama `model` field (already a backend tag because the orchestrator resolved logical_name via the registry).
    - Per `Message`:
        - `role`: pass through (`user|assistant|system`).
        - `content` as `str` → Ollama `content` field directly.
        - `content` as `list[ContentPart]`:
            - All `TextContent.text` values concatenated with `\n\n` separators → Ollama `content` field (a single string).
            - All `ImageContent` payloads as base64 strings → Ollama `images` array on the same message object.
            - All `AudioContent` payloads as base64 strings → Ollama `audios` array on the same message object — consistent with the `images` array convention. Single placement; no per-Ollama-version branching.
    - `params` → Ollama `options` dict (only consumer-set fields, via `model_dump(exclude_unset=True)`):
        - `temperature` → `temperature`
        - `top_p` → `top_p`
        - `top_k` → `top_k`
        - `max_tokens` → **`num_predict`** (Ollama's parameter name differs — likely future-bug source if the rename is forgotten)
        - `stop` → `stop`
        - `seed` → `seed`
        - `think` → placed at `options.think: true` (groups with the other sampling-style controls inside Ollama's `options` dict). Locked to a single placement so the integration tests can assert deterministically.
    - `stream` → always `false` (project boundary excludes streaming).
    - `keep_alive`, `tools`, `format` → not sent (daemon owns keep-alive, tools out of scope, no structured-output v1 spec).
- **Inbound translation: Ollama `/api/chat` response → `OllamaChatResult`:**
    - `response.message.content` → `OllamaChatResult.content`.
    - `response.prompt_eval_count` → `prompt_tokens`.
    - `response.eval_count` → `completion_tokens`.
    - `response.done_reason` → `finish_reason` mapping: `"stop"` → `"stop"`, `"length"` → `"length"`, anything else (`"unload"`, missing, etc.) → `"stop"` by default. `"timeout"` is set by E004-F003, never by F002.
- **`tool_calls` in response: ignored.** Tools/function calling is out of v1 project scope. F002 reads `message.content` only; if the model returns `tool_calls` (Gemma 4 may in some configurations), `content` is treated as the canonical answer (possibly empty — orchestrator handles empty strings as a model behavior, not a translation error).
- **httpx errors propagate raw.** F002 calls `response.raise_for_status()` for HTTP non-2xx (raises `httpx.HTTPStatusError`); any `httpx.RequestError` from `_request` propagates. F003 maps these to `ADAPTER_CONNECTION_FAILURE` at a higher layer.

## Out of scope

- **Failure mapping** (httpx exceptions → DomainError) — `LIP-E003-F003`.
- **Per-request timeout / `asyncio.wait_for`** — `LIP-E004-F003`.
- **Streaming responses** — out of v1 project scope.
- **Tool / function calling** — out of v1 scope; `tools` field never sent, `tool_calls` in response ignored.
- **`/api/generate`, `/api/embeddings`, `/api/tags`** — out of v1 scope. F002 only implements `/api/chat`.
- **`keep_alive` per-request override** — out of v1; the daemon's `KEEP_ALIVE=300s` env var owns model-retention behavior.
- **`format` (structured-output) parameter** — Ollama supports JSON-schema constrained generation, but LIP doesn't expose it in v1.
- **Capability validation** — `LIP-E002-F003`. F002 trusts that the orchestrator already validated capabilities before calling.
- **Sampling-default merging** — `LIP-E002-F002`. F002 receives already-merged `params`.
- **Response metadata fields the orchestrator owns** — `request_id`, `queue_wait_ms`, `latency_ms`, `backend`, logical-name `model`. F002's `OllamaChatResult` has only the four fields it can know about.

## Acceptance criteria

- `OllamaChatResult` Pydantic model has `content: str`, `prompt_tokens: int`, `completion_tokens: int`, `finish_reason: Literal["stop", "length", "timeout"]`; `frozen=True, extra="forbid"`.
- `chat(model_tag="gemma4:e2b", messages=[Message(role="user", content="hi")], params=ModelParams(temperature=0.0))` sends `POST /api/chat` with body `{"model": "gemma4:e2b", "messages": [{"role": "user", "content": "hi"}], "options": {"temperature": 0.0}, "stream": false}`.
- Multimodal: `messages=[Message(role="user", content=[TextContent(text="describe"), ImageContent(base64="iV...")])]` produces Ollama body containing `{"messages": [{"role": "user", "content": "describe", "images": ["iV..."]}], ...}`.
- Multipart text concatenation: `content=[TextContent(text="A"), TextContent(text="B")]` produces Ollama `content="A\n\nB"`.
- `max_tokens` rename: `ModelParams(max_tokens=512)` produces Ollama `options.num_predict=512` (NOT `max_tokens`). Specific AC for the rename to catch a likely future bug.
- Only consumer-set fields appear in `options`: `ModelParams(temperature=0.0)` produces `options={"temperature": 0.0}` only — `top_p`, `top_k`, etc. not included.
- `stream` is always `false`: every Ollama request body produced by F002 includes `"stream": false`.
- Response translation: Ollama `{"message": {"content": "hello"}, "prompt_eval_count": 10, "eval_count": 5, "done_reason": "stop"}` produces `OllamaChatResult(content="hello", prompt_tokens=10, completion_tokens=5, finish_reason="stop")`.
- `done_reason` length: Ollama `"length"` → `OllamaChatResult.finish_reason="length"`.
- `done_reason` fallback: Ollama `"unload"`, missing, or any unrecognized value → `finish_reason="stop"`.
- httpx errors propagate uncaught: `httpx.ConnectError` raised by `_request` propagates up from `chat()` without translation; F003 maps it at a higher layer.
- Forbidden fields never appear: `tools`, `keep_alive`, `format` never appear in any Ollama request body produced by F002.

**Implementation pattern:**

```python
# model/ollama_translation.py
from typing import Any

from app.features.inference.model.message import Message
from app.features.inference.model.model_params import ModelParams
from app.features.inference.model.ollama_chat_result import OllamaChatResult


def translate_message(msg: Message) -> dict[str, Any]:
    """Service Message -> Ollama /api/chat message dict."""
    if isinstance(msg.content, str):
        return {"role": msg.role, "content": msg.content}
    text_parts: list[str] = []
    images: list[str] = []
    audios: list[str] = []
    for part in msg.content:
        if part.type == "text":
            text_parts.append(part.text)
        elif part.type == "image":
            images.append(part.base64)
        elif part.type == "audio":
            audios.append(part.base64)
    out: dict[str, Any] = {"role": msg.role, "content": "\n\n".join(text_parts)}
    if images:
        out["images"] = images
    if audios:
        out["audios"] = audios
    return out


_PARAM_RENAMES = {"max_tokens": "num_predict"}


def translate_params(params: ModelParams) -> dict[str, Any]:
    """ModelParams -> Ollama options dict. Only includes consumer-set fields.

    `think` rides along inside the options dict alongside the sampling-style
    fields, so a single options payload carries every per-request toggle.
    """
    set_fields = params.model_dump(exclude_unset=True)
    return {_PARAM_RENAMES.get(k, k): v for k, v in set_fields.items()}


_OLLAMA_TO_LIP_FINISH = {"stop": "stop", "length": "length"}


def build_chat_result(response_json: dict[str, Any]) -> OllamaChatResult:
    """Ollama /api/chat response -> OllamaChatResult."""
    raw_finish = response_json.get("done_reason", "stop")
    finish_reason = _OLLAMA_TO_LIP_FINISH.get(raw_finish, "stop")
    return OllamaChatResult(
        content=response_json["message"]["content"],
        prompt_tokens=response_json.get("prompt_eval_count", 0),
        completion_tokens=response_json.get("eval_count", 0),
        finish_reason=finish_reason,
    )
```

```python
# repository/ollama_client.py — chat() method added to existing OllamaClient class
from app.features.inference.model.ollama_translation import (
    build_chat_result,
    translate_message,
    translate_params,
)


class OllamaClient:
    # __init__, close, __aenter__, __aexit__, _request unchanged from F001

    async def chat(
        self,
        *,
        model_tag: str,
        messages: list[Message],
        params: ModelParams,
    ) -> OllamaChatResult:
        body: dict[str, Any] = {
            "model": model_tag,
            "messages": [translate_message(m) for m in messages],
            "stream": False,
        }
        options = translate_params(params)
        if options:
            body["options"] = options

        response = await self._request("POST", "/api/chat", json=body)
        response.raise_for_status()
        return build_chat_result(response.json())
```

**Pure helper discipline:** `translate_message`, `translate_params`, `build_chat_result` are pure module-level functions. Easily unit-testable without httpx mocking. The `OllamaClient.chat` method delegates translation to them, keeping `chat()` thin.

**Pydantic config on `OllamaChatResult`:** `frozen=True, extra="forbid"` — immutable, no surprise fields.

**Errors propagate raw:** `chat()` calls `response.raise_for_status()` for HTTP non-2xx (raises `httpx.HTTPStatusError`), and any `httpx.RequestError` from `_request` propagates uncaught. F003 catches and maps to `ADAPTER_CONNECTION_FAILURE`.

## Open questions

None. Both prior `[UNRESOLVED]` markers are now locked:

- *[RESOLVED]* **`AudioContent` payload encoding for Ollama.** Sent as base64 strings in an `audios` array on the message object, mirroring the `images` array convention. Single canonical placement; the `translate_message` helper appends to `audios` for every audio part, the same way it appends to `images` for every image part.
- *[RESOLVED]* **Placement of `think` in the Ollama request body.** Locked to `options.think` — same dict that already carries `temperature`, `top_p`, `top_k`, `num_predict`, `stop`, and `seed`. Keeps all sampling-style toggles in one structured location and makes the integration test scenarios deterministic (no per-version branching).

## Unit test scenarios

- `OllamaChatResult(content="x", prompt_tokens=10, completion_tokens=5, finish_reason="stop")` constructs without error.
- `OllamaChatResult` is frozen — assigning to `.content` after construction raises `ValidationError`.
- `OllamaChatResult` rejects `finish_reason="bad"` (Literal violation).
- `translate_message(Message(role="user", content="hi"))` returns `{"role": "user", "content": "hi"}` exactly.
- `translate_message(Message(role="user", content=[TextContent(text="A"), TextContent(text="B")]))` returns `{"role": "user", "content": "A\n\nB"}` (concatenation with `\n\n` separator).
- `translate_message(Message(role="user", content=[TextContent(text="describe"), ImageContent(base64="iV...")]))` returns `{"role": "user", "content": "describe", "images": ["iV..."]}`.
- `translate_message(Message(role="user", content=[ImageContent(base64="aaa"), ImageContent(base64="bbb")]))` returns `images=["aaa", "bbb"]` in original order.
- `translate_params(ModelParams())` returns `{}` (no consumer-set fields).
- `translate_params(ModelParams(temperature=0.0))` returns `{"temperature": 0.0}` only — no other fields.
- `translate_params(ModelParams(max_tokens=512))` returns `{"num_predict": 512}` (rename verified — single most fragile mapping).
- `translate_params(ModelParams(temperature=0.5, top_p=0.9, top_k=40, max_tokens=512, stop=["X"], seed=42))` returns all six fields with `max_tokens` renamed to `num_predict`.
- `translate_params(ModelParams(think=True))` returns `{"think": True}` — `think` rides inside the options dict alongside the sampling-style fields.
- `build_chat_result({"message": {"content": "hello"}, "prompt_eval_count": 10, "eval_count": 5, "done_reason": "stop"})` returns `OllamaChatResult(content="hello", prompt_tokens=10, completion_tokens=5, finish_reason="stop")`.
- `build_chat_result(...done_reason="length"...)` returns `finish_reason="length"`.
- `build_chat_result(...done_reason="unload"...)` returns `finish_reason="stop"` (fallback).
- `build_chat_result(...{"message": {"content": "x"}})` (no `done_reason` key) returns `finish_reason="stop"`.
- `build_chat_result({"message": {"content": "x"}, "prompt_eval_count": 0, "eval_count": 0})` returns zero-token counts.
- `build_chat_result({})` raises `KeyError` on `message.content` access (failure path; F003 of E003 catches).
- `build_chat_result({"message": {}})` raises `KeyError` on `content`.
- The `_PARAM_RENAMES` dict contains exactly `{"max_tokens": "num_predict"}` — single rename only.
