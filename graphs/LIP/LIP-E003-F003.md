## Short description

Connection refused, HTTP non-2xx responses, malformed JSON, and other backend-specific failures are caught at the adapter boundary and mapped to typed DomainError subclasses so callers above the adapter never see Ollama-specific exceptions.

## Problem

The orchestrator must never see Ollama-specific exception types. Without F003, the orchestrator catches `httpx.ConnectError`, `httpx.HTTPStatusError`, etc. — backend-specific. When G1 fires (Ollama → vLLM / MLX), the orchestrator's exception handlers break because vLLM raises different error types.

F003 is the boundary layer: catches every Ollama-specific exception type that can propagate out of F002's `chat()` call, maps each to `AdapterConnectionFailureError` (defined by F004) with a typed `reason` discriminator naming the failure mode. Post-F003, the orchestrator only ever sees typed DomainError instances — backend-agnostic.

This is the strongest practical expression of G1 alongside F002. F002 isolates the request/response translation; F003 isolates the failure translation. Together they form the complete swap point: replace E003 entirely with a `vllm/` directory and the orchestrator's contract is unchanged.

*Stakeholder:* Cosmin (consumer-side dev + maintainer hats). *Need:* every adapter failure produces a typed `AdapterConnectionFailureError` with a `reason` consumers can program against. No leaking httpx exceptions into orchestrator code.

## Scope

- **F003 modifies `OllamaClient.chat()` (defined by F002) to wrap its body in `try/except`.** Public method signature unchanged. Successful Ollama responses still return `OllamaChatResult`; failures raise `AdapterConnectionFailureError` from F004's hierarchy.
- **Exception → `reason` mapping (5 reasons):**
    - `httpx.ConnectError` → `reason="connection_refused"` (Ollama daemon not reachable)
    - `httpx.ConnectTimeout` → `reason="connect_timeout"` (daemon up but slow to accept; exceeds F001's 5s connect timeout)
    - `httpx.HTTPStatusError` → `reason="non_2xx"` (Ollama returned 4xx or 5xx)
    - `json.JSONDecodeError` or `KeyError` from response-shape access → `reason="malformed_response"` (invalid JSON or missing expected fields)
    - `httpx.RequestError` (catch-all parent class) → `reason="unknown"` (any other transport failure)
- **Order of `except` clauses** is concrete-to-general — `ConnectError` and `ConnectTimeout` are `RequestError` subclasses; `RequestError` must be the last clause or the specific subclasses get masked.
- **Original exception preserved via `raise ... from exc`** for traceback debugging. Production-grade logging of the original exception is deferred to a future structured-logging milestone (out of v1 scope per Project Boundary).
- **Public API unchanged.** No `chat_safe()` wrapper method, no class hierarchy change. F003's modifications are entirely inside `chat()`'s body.

## Out of scope

- **Per-request `asyncio.wait_for` timeout** — `LIP-E004-F003`. F003 catches httpx's connect-level timeout, not the orchestrator's request-level timeout.
- **DomainError class definitions** — `LIP-E004-F004`. F003 imports `AdapterConnectionFailureError`, doesn't define it.
- **Retry / backoff logic** — out of v1; F003 catches-and-re-raises, never retries.
- **Structured-log emission** — out of v1 project scope. The original exception is preserved via `from exc` for traceback debugging; production-grade logging is a future milestone.
- **Capability validation** — `LIP-E002-F003`.
- **Translation logic** — `LIP-E003-F002`.
- **HTTP client configuration** — `LIP-E003-F001`.

## Acceptance criteria

- With the Ollama daemon stopped, `await client.chat(...)` raises `AdapterConnectionFailureError(backend="ollama", reason="connection_refused")`.
- With a slow-to-accept Ollama (simulated via mock or unresponsive endpoint), `client.chat(...)` raises `AdapterConnectionFailureError(backend="ollama", reason="connect_timeout")` after the 5-second connect timeout.
- With Ollama returning HTTP 500, `client.chat(...)` raises `AdapterConnectionFailureError(backend="ollama", reason="non_2xx")`.
- With Ollama returning HTTP 200 + body `"not json{"`, `client.chat(...)` raises `AdapterConnectionFailureError(backend="ollama", reason="malformed_response")`.
- With Ollama returning HTTP 200 + valid JSON `{"foo": "bar"}` (missing `message.content`), `client.chat(...)` raises `AdapterConnectionFailureError(backend="ollama", reason="malformed_response")`.
- The raised `AdapterConnectionFailureError` chains the original httpx (or json / KeyError) exception via `__cause__` — set by `raise ... from exc`.
- Successful 2xx with valid JSON returns `OllamaChatResult` — F003's catch logic does not intercept the happy path.
- `chat()`'s public signature, parameters, and return type are unchanged from F002.

## Technical constraints

**File layout:** F003 adds no new files. Modifies the existing `apps/backend/app/features/inference/repository/ollama_client.py`'s `chat()` method body.

**Implementation pattern (post-F002 `chat()` modified by F003):**

```python
# repository/ollama_client.py — chat() method post-F003
import json

import httpx

from app.exceptions import AdapterConnectionFailureError

# ... other imports unchanged from F001/F002 ...


class OllamaClient:
    # __init__, close, __aenter__, __aexit__, _request unchanged from F001

    async def chat(
        self,
        *,
        model_tag: str,
        messages: list[Message],
        params: ModelParams,
    ) -> OllamaChatResult:
        body: dict[str, Any] = {
            "model": model_tag,
            "messages": [translate_message(m) for m in messages],
            "stream": False,
        }
        options = translate_params(params)
        if options:
            body["options"] = options
        if params.think:
            body["think"] = True

        try:
            response = await self._request("POST", "/api/chat", json=body)
            response.raise_for_status()
            return build_chat_result(response.json())
        except httpx.ConnectTimeout as exc:
            raise AdapterConnectionFailureError(
                backend="ollama", reason="connect_timeout"
            ) from exc
        except httpx.ConnectError as exc:
            raise AdapterConnectionFailureError(
                backend="ollama", reason="connection_refused"
            ) from exc
        except httpx.HTTPStatusError as exc:
            raise AdapterConnectionFailureError(
                backend="ollama", reason="non_2xx"
            ) from exc
        except (json.JSONDecodeError, KeyError) as exc:
            raise AdapterConnectionFailureError(
                backend="ollama", reason="malformed_response"
            ) from exc
        except httpx.RequestError as exc:
            raise AdapterConnectionFailureError(
                backend="ollama", reason="unknown"
            ) from exc
```

**`except` ordering invariant:** `ConnectTimeout` and `ConnectError` must come before `HTTPStatusError` (irrelevant for ordering between those two), and the catch-all `RequestError` must be *last* — otherwise its specific subclasses are masked. This is the most fragile aspect of F003's implementation; AC #1, #2, and #6 collectively verify the ordering.

**Pure exception-translation discipline:** F003's added code does no I/O, no retries, no logging in v1. The exception chain (`__cause__` set by `from exc`) carries the original failure for traceback debugging without F003 needing to log it explicitly.

**Error pipeline (CLAUDE.md mandate):** the code `ADAPTER_CONNECTION_FAILURE` (HTTP 502) is defined in [packages/error-contracts/errors.yaml](packages/error-contracts/errors.yaml) under E004-F004's contract with params `{backend: str, reason: Literal["connection_refused", "connect_timeout", "non_2xx", "malformed_response", "unknown"]}`. Per CLAUDE.md "Never add an error code without editing errors.yaml, running task errors:generate," before implementing F003 confirm the entry exists in `errors.yaml` (and that the `reason` Literal includes all five values F003 produces), run `task errors:generate`, and run `task check:errors`. F003 imports `AdapterConnectionFailureError` from `app.exceptions`.

## Open questions

None. F003's mapping is a complete enumeration of expected exception sources from F001/F002. Future httpx versions adding new exception types fall through to the `httpx.RequestError` catch-all with `reason="unknown"` — a graceful default, not silent corruption.

## Unit test scenarios

- A `chat()` call against an `httpx.MockTransport` that raises `httpx.ConnectError` raises `AdapterConnectionFailureError(backend="ollama", reason="connection_refused")` with `__cause__` set to the original `ConnectError`.
- A `chat()` call where the transport raises `httpx.ConnectTimeout` raises `AdapterConnectionFailureError(reason="connect_timeout")`.
- A `chat()` call where the transport returns HTTP 500 raises `AdapterConnectionFailureError(reason="non_2xx")` (`raise_for_status` raises `HTTPStatusError` first, F003 catches and rewraps).
- A `chat()` call where the transport returns HTTP 404 raises `AdapterConnectionFailureError(reason="non_2xx")` (HTTP 4xx also non-2xx).
- A `chat()` call where the transport returns HTTP 200 + `b"not json{"` raises `AdapterConnectionFailureError(reason="malformed_response")`.
- A `chat()` call where the transport returns HTTP 200 + `{"foo": "bar"}` (no `message.content`) raises `AdapterConnectionFailureError(reason="malformed_response")` via `KeyError`.
- A `chat()` call where the transport raises `httpx.NetworkError` (an arbitrary `RequestError` subclass not in the specific list) raises `AdapterConnectionFailureError(reason="unknown")`.
- The exception ordering invariant holds: a transport raising `httpx.ConnectError` produces `reason="connection_refused"`, NOT `reason="unknown"` — verified that `RequestError`'s catch-all does not mask its specific subclasses.
- The original exception is preserved in `__cause__`: `assert isinstance(raised.__cause__, httpx.ConnectError)` for the connection-refused case.
- A successful 2xx response with valid JSON does NOT raise — F003's catch logic is bypassed.
- F003's modifications do not alter `chat()`'s public signature: `chat(*, model_tag, messages, params)` returns `OllamaChatResult` per F002's contract.

## Integration test scenarios

- With Ollama daemon stopped (real integration test, not mock), `await client.chat(...)` raises `AdapterConnectionFailureError(backend="ollama", reason="connection_refused")` within ~5 seconds (the F001 `connect=5.0` timeout).
