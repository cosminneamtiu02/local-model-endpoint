## Short description

`asyncio.Semaphore(1)` wrapping the orchestrator's inference call, ensuring only one inference executes at a time regardless of HTTP-level concurrency.

## Problem

The 16 GB M4 Mac Mini hosting LIP cannot run two Gemma-4-E2B inferences simultaneously without thrashing memory and degrading both calls' latency below useful thresholds. Without serialization, FastAPI's async transport accepts every incoming request immediately and forwards them in parallel to Ollama, which then either rejects, OOMs, or interleaves token-by-token execution — none of which produce the deterministic per-call latency a cognitive-task consumer needs. Worse, parallel calls don't even speed throughput because Ollama's GPU/CPU is the bottleneck, not its API.

F001 enforces *one inference in flight at a time* at LIP's service layer, so the deployment fact ("Mac Mini, 1 model loaded, sequential is fastest") is enforced by the service rather than left to consumer discipline. This is the foundation feature for E004's QoS story; F002 (bounded waiter count) and F003 (per-request timeout) both wrap this same semaphore.

*Stakeholder:* Cosmin (operator hat). *Need:* deterministic per-request inference latency under any incoming concurrency, regardless of how many consumer projects (up to 4 in v1) happen to issue calls at the same instant. The hardware constraint becomes a service-layer guarantee.

## Scope

- **Process-wide `asyncio.Semaphore(1)` singleton** wired via `@lru_cache(maxsize=1)`-decorated factory `get_inference_semaphore() -> asyncio.Semaphore` in `apps/backend/app/api/deps.py`. Mirrors the existing `get_registry()` / `get_settings()` pattern; produces one semaphore per process for the lifetime of the app.
- **`InferenceService` constructor extended** to receive the semaphore as a third dependency: `__init__(self, registry: ModelRegistry, client: OllamaClient, semaphore: asyncio.Semaphore)`. The service holds a reference; the singleton lives in `lru_cache`.
- **`get_inference_service()` Depends factory updated** to pass the semaphore through: now reads `semaphore: asyncio.Semaphore = Depends(get_inference_semaphore)` alongside the existing registry and client deps.
- **`InferenceService.run()` body modified** — the existing 7-step sequence from F002-of-E001 keeps its order, with the semaphore wrapping *only* the adapter call (steps 5–6, not the pre-checks). New step layout:
    1. **Capture total-start** (`time.perf_counter()`).
    2. Registry lookup. *(May raise `RegistryNotFoundError` — fast-fails before the semaphore.)*
    3. Capability check. *(May raise `ModelCapabilityNotSupportedError` — fast-fails.)*
    4. Param merge.
    5. **Capture wait-start** (`time.perf_counter()`).
    6. **`async with self._semaphore:`** — wait for exclusive access.
    7. Inside the lock: compute `queue_wait_ms = int((perf_counter() - wait_start) * 1000)`, then `await self._client.chat(...)` to produce the `OllamaChatResult`.
    8. After the lock releases, compute `latency_ms` and compose `InferenceResponse` with the *real* `queue_wait_ms` value (replacing F002-of-E001's `queue_wait_ms=0` placeholder).
- **`queue_wait_ms` measurement scope** is *only* the semaphore-wait portion — the time between attempting acquisition and obtaining the lock. Once inside the lock the timer stops; the inference itself contributes to `latency_ms - queue_wait_ms - pre_check_time`. This makes `queue_wait_ms` a clean diagnostic of "how long did this request queue behind others."
- **`async with` guarantees release on exit**, including exception paths. If `client.chat()` raises `AdapterConnectionFailureError` (E003-F003) or any other DomainError, the lock is released before the exception propagates. No leaked permits.
- **Fast-fail policy preserved**: registry-not-found, capability-not-supported, validation errors, and any other failure detected before step 6 short-circuit before queueing. A consumer hitting `model="nonexistent"` gets a 404 in microseconds even when 3 other inference requests are in flight.

## Out of scope

- **Bounded waiter count / queue-full 503** — `LIP-E004-F002`. F001 enforces "one at a time" but doesn't reject the 5th waiter; that's F002's role. F001 alone is "infinite queue," which under realistic ≤4-consumer workloads is fine; F002 caps it.
- **Per-request `asyncio.wait_for` timeout** — `LIP-E004-F003`. F001's `async with self._semaphore` waits forever in the worst case; F003 wraps this with a 180s budget.
- **Per-model semaphores** — v1 ships exactly one Gemma 4 E2B entry; one global semaphore is correct. If/when LIP loads multiple models concurrently, per-model semaphores become a refactor; deferred.
- **Cross-process serialization** — LIP runs as a single process (no Gunicorn worker fanout, no multiprocess hosting); single-process semaphore is sufficient.
- **Priority queue / fairness beyond FIFO** — `asyncio.Semaphore` is FIFO since Python 3.10 (documented behavior). No priority lanes in v1; deferred unless a real workload need surfaces.
- **Rate limiting per consumer project** — out of v1; G4 architectural foundation only. F001 is global serialization, not per-caller.
- **Streaming response semaphore semantics** — out of v1 (no streaming).
- **Backpressure / 503 emission** — `LIP-E004-F002`. F001 just queues silently; F002 adds the rejection path.
- **Idle-shutdown coordination with the semaphore** — `LIP-E005-F002`. The idle timer reads "time since last request finished" from somewhere; F001 doesn't expose that. E005-F002 introduces the timestamp tracking it needs.
- **Health-check observability of the semaphore** — `LIP-E006-F002` (state-inspection endpoints). F001 ships the semaphore; the readback endpoint is later.
- **`pydantic-settings` configurability of the semaphore size** — semaphore is fixed at 1 (architectural constraint from the deployment hardware fact). F002's waiter count and F003's timeout are configurable; F001's "1" is not.

## Acceptance criteria

- A single `POST /v1/inference` succeeds end-to-end with `metadata.queue_wait_ms` ≈ 0 (under 5 ms in practice — the only "wait" is the time `async with self._semaphore` takes when uncontested, which is sub-millisecond).
- Two simultaneous `POST /v1/inference` calls (kicked off via concurrent `httpx.AsyncClient.post` from a test client) both succeed with HTTP 200, but the second one's `metadata.queue_wait_ms` is ≥ the first one's `metadata.latency_ms - metadata.queue_wait_ms` (the wall time of the first inference body, minus the first one's negligible wait). Verified deterministically by mocking the adapter to await a known-duration `asyncio.sleep`.
- Three simultaneous calls all eventually succeed; the third one's `queue_wait_ms` ≈ 2× the first inference's body wall time (queueing behind two predecessors).
- `Ollama.chat` is *never* invoked concurrently. Verified by an integration test using a mock adapter that asserts `call_count == 1` while another call is in flight (uses `asyncio.Event` to suspend mid-call and check from another task).
- A registry-not-found request (HTTP 404) returns *without ever attempting to acquire the semaphore* — verified by ordering: a slow inference is in progress holding the lock; a 404 request issued while the slow one runs returns its 404 within microseconds, not after the slow one completes.
- A capability-not-supported request (HTTP 422 from `MODEL_CAPABILITY_NOT_SUPPORTED`) similarly fast-fails without queueing.
- An adapter failure (Ollama down → `AdapterConnectionFailureError` → HTTP 502) releases the semaphore on exit. Verified by: kill Ollama, fire one request that 502s, then a second request — the second one acquires immediately (semaphore was released, not leaked).
- `get_inference_semaphore()` returns the same `asyncio.Semaphore` object across calls within a single FastAPI app instance (verified via `is` identity check).
- `get_inference_semaphore()._value == 1` at app startup (one permit available) before any request is processed.
- `InferenceService.__init__` accepts the semaphore as a positional/keyword arg; passing `None` or a wrong-type object is a type error caught by Pyright strict at build time, not at runtime.
- Pyright strict passes; `task check` passes.

No `inference_semaphore.py` standalone file — the semaphore is just an `asyncio.Semaphore` instance, not a class. Per CLAUDE.md "one class per file" applies to *classes*, not to module-level factory functions; the factory belongs in `deps.py` alongside its peers.

**`api/deps.py` additions:**

```python
import asyncio
from functools import lru_cache

from fastapi import Depends, Request

from app.features.inference.repository.ollama_client import OllamaClient
from app.features.inference.service.inference_service import InferenceService
from app.features.inference.service.model_registry import ModelRegistry


@lru_cache(maxsize=1)
def get_inference_semaphore() -> asyncio.Semaphore:
    return asyncio.Semaphore(1)


def get_inference_service(
    registry: ModelRegistry = Depends(get_registry),
    client: OllamaClient = Depends(get_ollama_client),
    semaphore: asyncio.Semaphore = Depends(get_inference_semaphore),
) -> InferenceService:
    return InferenceService(registry, client, semaphore)
```

**`InferenceService` modified:**

```python
import asyncio
import time

from app.features.inference.repository.ollama_client import OllamaClient
from app.features.inference.schemas.inference_request import InferenceRequest
from app.features.inference.schemas.inference_response import InferenceResponse
from app.features.inference.schemas.response_metadata import ResponseMetadata
from app.features.inference.service.capability_check import check_envelope_capabilities
from app.features.inference.service.model_registry import ModelRegistry
from app.features.inference.service.sampling_merge import merge_sampling_params


class InferenceService:
    def __init__(
        self,
        registry: ModelRegistry,
        client: OllamaClient,
        semaphore: asyncio.Semaphore,
    ) -> None:
        self._registry = registry
        self._client = client
        self._semaphore = semaphore

    async def run(
        self,
        envelope: InferenceRequest,
        request_id: str,
    ) -> InferenceResponse:
        start = time.perf_counter()
        model_info = self._registry.get(envelope.model)
        check_envelope_capabilities(envelope, model_info)
        merged = merge_sampling_params(envelope.params, model_info.sampling_defaults)

        wait_start = time.perf_counter()
        async with self._semaphore:
            queue_wait_ms = int((time.perf_counter() - wait_start) * 1000)
            result = await self._client.chat(
                model_tag=model_info.backend_tag,
                messages=envelope.messages,
                params=merged,
            )

        latency_ms = int((time.perf_counter() - start) * 1000)
        return InferenceResponse(
            content=result.content,
            metadata=ResponseMetadata(
                model=envelope.model,
                prompt_tokens=result.prompt_tokens,
                completion_tokens=result.completion_tokens,
                request_id=request_id,
                latency_ms=latency_ms,
                queue_wait_ms=queue_wait_ms,
                finish_reason=result.finish_reason,
                backend="ollama",
            ),
        )
```

**`async with self._semaphore` is exception-safe.** Python's context-manager protocol guarantees `__aexit__` runs whether the body completes normally or raises. A leaked permit would mean a deadlock under the next request; the protocol prevents that. No `try/finally` needed; `async with` is the idiom.

**Why a permit count of 1, not configurable:**

Hardware fact: on 16 GB / M4 / Gemma-4-E2B, two simultaneous inferences degrade both calls' latency below useful thresholds (memory thrash; GPU contention). The deployment is single-machine, single-model in v1; "one at a time" is not a tunable parameter, it's the architecturally correct value. A configurable semaphore size in v1 would invite operators to set it to 4 "for throughput" and break the QoS contract. F002 and F003 expose tunables (waiter count, timeout) precisely *because* those are policy choices; semaphore size is a hardware truth.

**`asyncio.Semaphore` outside an event loop:** Since Python 3.10, `asyncio.Semaphore()` constructed outside a running loop binds to the loop only on first `acquire`. The `@lru_cache` factory called by FastAPI's request handling occurs inside the loop, so this is safe.

**FIFO ordering**: `asyncio.Semaphore` is FIFO since Python 3.10, documented in the asyncio module docs. Requests acquire in arrival order; no fairness anomalies. Cosmin's "≤4 consumers" workload is small enough that FIFO is indistinguishable from any other policy.

**Pyright strict typing:** `asyncio.Semaphore` is fully typed in the standard library; `Depends(get_inference_semaphore)` annotates correctly without `# type: ignore`. The constructor's `semaphore: asyncio.Semaphore` annotation is enforced.
