## Short description

Maximum of four waiters on the inference semaphore; the fifth concurrent request returns HTTP 503 with a structured "queue full" DomainError instead of waiting indefinitely.

## Problem

F001 of E004 enforces serial inference but not bounded queueing — a 50-request burst from a misbehaving consumer would queue all 50 with `queue_wait_ms` climbing into the minutes. The 50th caller would eventually receive a real response 10+ minutes later, far past any useful timeout, and the orchestrator's memory footprint would balloon with suspended coroutines holding request bodies.

F002 caps the queue depth at 4 (per the disambiguated idea's Quality Attributes — "≤4 concurrent consumer projects each issuing serial requests" — the tail end of "everyone hits at once"). The 5th waiter receives an immediate HTTP 503 with a structured `QUEUE_FULL` DomainError carrying `{max_waiters, current_waiters}` so the consumer can implement its own retry-with-backoff. This converts the unbounded-queue failure mode (silent latency degradation, memory pressure) into a fast, programmable rejection that consumers can handle deterministically.

*Stakeholder:* Cosmin (operator + consumer-side dev hats). *Need:* the service rejects clearly when its capacity is exceeded, instead of silently degrading. Consumer code can pattern-match `error.code == "QUEUE_FULL"` and either retry, fall back, or surface to the user.

## Scope

- **Configurable `max_waiters` setting** added to `app/core/config.py`'s `Settings` class: `max_inference_waiters: int = Field(default=4, gt=0)`. Pydantic-settings reads from the `MAX_INFERENCE_WAITERS` env var. Default 4 matches the project's stated ≤4-concurrent-consumer Quality Attribute. Hardcoding would be cleaner but conflicts with CLAUDE.md's "Never use os.environ … Use pydantic-settings" pattern and the standing convention that QoS knobs are env-tunable.
- **Atomic waiter counter** introduced as a process-wide `WaiterCounter` value-object (one class per file → `apps/backend/app/features/inference/service/waiter_counter.py`). Internally holds an `int` and an `asyncio.Lock` (NOT the inference semaphore — independent lock guarding the counter increments/decrements). Exposes:
    - `async def reserve(self, max_waiters: int) -> int` — atomically increments the counter; if the post-increment value exceeds `max_waiters`, decrements back and raises `QueueFullError(max_waiters=max_waiters, current_waiters=<post-increment value before decrement>)`. Otherwise returns the post-increment value.
    - `async def release(self) -> None` — atomically decrements the counter. Always called in the `finally` of a try-block.
    - `def current(self) -> int` — synchronous read of the counter for state-inspection (E006-F002 will consume).
    - `lock_held(self) -> AsyncContextManager[int]` — `@asynccontextmanager` that yields the current count while holding the internal `_lock`. Lets external callers perform atomic check-then-act over the counter without exposing `_lock` directly. The idle watchdog (E005-F002) uses this to atomically verify counter==0 and trigger SIGTERM, eliminating a TOCTOU window between the check and the kill.
- **`InferenceService.run()` body modified** — between F001-of-E004's pre-checks (registry / capability / merge) and the `async with self._semaphore`, F002 inserts the reserve/release pair:
    1. Pre-checks (registry, capability, merge) — unchanged from F001-of-E004.
    2. **`waiter_count = await self._waiter_counter.reserve(self._max_waiters)`** — may raise `QueueFullError` (HTTP 503).
    3. **`try:`** wrap the rest:
        4. `wait_start = perf_counter()`
        5. `async with self._semaphore:` — F001-of-E004's existing wait.
        6. `queue_wait_ms = int((perf_counter() - wait_start) * 1000)`
        7. `result = await self._client.chat(...)`
    8. **`finally: await self._waiter_counter.release()`** — guarantees decrement even if the adapter raises.
- **`InferenceService` constructor extended** to receive `WaiterCounter` and `max_waiters: int`. Updated signature:  
  `__init__(self, registry, client, semaphore, waiter_counter: WaiterCounter, max_waiters: int)`.
- **`get_waiter_counter()` Depends factory** in `app/api/deps.py` with `@lru_cache(maxsize=1)` — process-wide singleton. Factory returns `WaiterCounter()` constructed with no args (the counter has no per-process configuration; `max_waiters` is passed at runtime, not at construction).
- **`get_inference_service()` factory updated** to also `Depends(get_waiter_counter)` and `Depends(get_settings)` — pulls `settings.max_inference_waiters` and passes it to `InferenceService`.
- **`QUEUE_FULL` error wiring**: F002 imports `QueueFullError` from `app.exceptions` (defined by F004-of-E004). The error carries params `{max_waiters: int, current_waiters: int}`; the FastAPI exception handler (also F004) renders it as RFC 7807 with HTTP status 503.

## Out of scope

- **Per-request timeout / 504** — `LIP-E004-F003`. F002 caps queue depth in waiters; F003 caps wait time per request. Independent dimensions.
- **Per-consumer rate limiting** — F002 is a global queue limit, not per-caller. Two consumers each issuing 3 requests = 6 waiters → 5th and 6th rejected; F002 cannot distinguish whose 5th request it is. G4 architectural foundation only.
- **Priority queue / fairness lanes** — F002 + the underlying FIFO `asyncio.Semaphore` produce strict FIFO with rejection at the head boundary. No priority lanes.
- **`Retry-After` HTTP header** — out of v1. The disambiguated-idea Quality Attributes don't require it; consumers can implement their own retry interval. Future addition is purely additive (no contract change).
- **Streaming responses** — out of v1.
- **Adaptive queue sizing** based on inference latency — out of v1; static `max_waiters` is sufficient.
- **Observability of the rejection rate** — `LIP-E006-F002` (state-inspection endpoints) reads `waiter_counter.current()` for queue depth; rejection rate is a future structured-log metric.
- **Dead-letter handling for rejected requests** — out of v1; rejection is fire-and-forget, the consumer owns retry.
- **Adjusting `max_waiters` at runtime without restart** — out of v1; setting changes require process restart.
- **Distinguishing "in-flight" from "waiting"** in the counter — F002 counts both as "waiters." The semantically purer split (waiters = waiting; in_flight = currently in the lock) is achievable with two counters but adds complexity for no v1 benefit; deferred.

## Acceptance criteria

- A single `POST /v1/inference` succeeds with `metadata.queue_wait_ms` ≈ 0 (no contention).
- 4 simultaneous `POST /v1/inference` calls all succeed with HTTP 200 (the 4-waiter limit is inclusive; 4 waiters fit, 5 don't).
- 5 simultaneous `POST /v1/inference` calls: 4 succeed with HTTP 200; 1 returns HTTP 503 with `Content-Type: application/problem+json`, body code `"QUEUE_FULL"`, params `{"max_waiters": 4, "current_waiters": 5}`.
- The 503 response is *immediate* (under 5 ms wall-clock) — F002 rejects without queuing on the semaphore.
- After a 503 rejection, a subsequent (non-overlapping) request succeeds — verified by issuing 5 calls, observing the 503, then issuing a 6th call after the 4 successes complete; the 6th gets 200.
- Rejected requests do not leak slots: after a 503, `waiter_counter.current() <= 4` (the rejection's increment was rolled back via decrement before the raise).
- Successful requests release their slot even on adapter failure: with Ollama down, fire 4 calls (all 502), then fire a 5th — the 5th gets 502 (Ollama still down), not 503 (counter was decremented in `finally`).
- `MAX_INFERENCE_WAITERS=2` env var: 2 simultaneous requests succeed; 3rd returns 503 with `max_waiters=2`. Verified that the env var path works end-to-end.
- `MAX_INFERENCE_WAITERS=0`: rejected at config validation by Pydantic (`gt=0`); FastAPI fails to start with a clear error. (The `gt=0` Field constraint catches misconfiguration at startup.)
- `WaiterCounter.current()` is synchronously readable without acquiring the internal lock — used by E006-F002's state endpoint without serializing it behind inference traffic.
- A registry-not-found request (HTTP 404) does NOT increment the waiter counter — F002's pre-checks fast-fail before `reserve()`. Verified: hold 4 in-flight inference requests; issue a `model="nonexistent"` request; counter remains at 4 (not 5); the 404 returns immediately.
- Pyright strict passes; `task check` passes.
- The `WaiterCounter` class lives in its own file (`waiter_counter.py`) — one class per file rule.
- `lock_held()` async context manager yields the current count while holding the internal lock — verified by holding the context manager in one task while a concurrent `reserve()` blocks until the context manager exits. The yielded count matches `current()` at entry.

**`WaiterCounter` implementation:**

```python
import asyncio
from collections.abc import AsyncIterator
from contextlib import asynccontextmanager

from app.exceptions import QueueFullError


class WaiterCounter:
    def __init__(self) -> None:
        self._count: int = 0
        self._lock: asyncio.Lock = asyncio.Lock()

    async def reserve(self, max_waiters: int) -> int:
        async with self._lock:
            self._count += 1
            if self._count > max_waiters:
                current_waiters = self._count
                self._count -= 1
                raise QueueFullError(
                    max_waiters=max_waiters,
                    current_waiters=current_waiters,
                )
            return self._count

    async def release(self) -> None:
        async with self._lock:
            self._count -= 1

    def current(self) -> int:
        return self._count

    @asynccontextmanager
    async def lock_held(self) -> AsyncIterator[int]:
        """Yield current count while holding the internal lock.

        Used by the idle watchdog (E005-F002) to atomically check the counter
        and act (SIGTERM) without a race against new reserve() calls.
        Keep the with-block body short and non-blocking — it serializes
        reserve() and release() calls.
        """
        async with self._lock:
            yield self._count
```

**Why a separate `asyncio.Lock` rather than a CAS-on-int:** Python's GIL makes `int` increment/decrement atomic in CPython, but the increment-check-decrement-raise sequence is *not* atomic across `await` points. The lock guards the entire transaction. The lock contention is microseconds; under v1's ≤4-concurrent workload, immeasurable. Synchronous `current()` reads outside the lock are safe — Python's GIL ensures a consistent integer read; readers don't need to serialize.

**`InferenceService.run()` modified body:**

```python
async def run(
    self,
    envelope: InferenceRequest,
    request_id: str,
) -> InferenceResponse:
    start = time.perf_counter()
    model_info = self._registry.get(envelope.model)
    check_envelope_capabilities(envelope, model_info)
    merged = merge_sampling_params(envelope.params, model_info.sampling_defaults)

    await self._waiter_counter.reserve(self._max_waiters)
    try:
        wait_start = time.perf_counter()
        async with self._semaphore:
            queue_wait_ms = int((time.perf_counter() - wait_start) * 1000)
            result = await self._client.chat(
                model_tag=model_info.backend_tag,
                messages=envelope.messages,
                params=merged,
            )
    finally:
        await self._waiter_counter.release()

    latency_ms = int((time.perf_counter() - start) * 1000)
    return InferenceResponse(
        content=result.content,
        metadata=ResponseMetadata(
            model=envelope.model,
            prompt_tokens=result.prompt_tokens,
            completion_tokens=result.completion_tokens,
            request_id=request_id,
            latency_ms=latency_ms,
            queue_wait_ms=queue_wait_ms,
            finish_reason=result.finish_reason,
            backend="ollama",
        ),
    )
```

**`reserve` raises *before* the `try/finally` wraps it.** The `try` only wraps the post-acquire body; if `reserve()` raises `QueueFullError`, no `release()` runs (correct — the increment was already rolled back inside `reserve()` itself). This is the central correctness invariant: every successful `reserve()` is paired with exactly one `release()`; every failed `reserve()` rolls itself back atomically.

**`Settings` field with `pydantic-settings`:**

```python
# app/core/config.py
class Settings(BaseSettings):
    # existing fields …
    max_inference_waiters: int = Field(default=4, gt=0)
```

`pydantic-settings` reads `MAX_INFERENCE_WAITERS` from env. Negative or zero values rejected at app-construction time (`gt=0`). Default 4 matches the project's ≤4-consumer Quality Attribute.

**Counter is process-wide singleton** via `@lru_cache(maxsize=1)` on `get_waiter_counter()` — same pattern as semaphore and registry. The counter must outlive request scope; per-request counters defeat the bound.

**No leaked permits** — every successful `reserve()` is followed by `release()` via `try/finally`. Every failed `reserve()` self-rolls-back inside the lock before raising. Every adapter exception (`AdapterConnectionFailureError`, etc.) propagates through the `finally` → release runs → counter decrements → next request can use the slot.

**`waiter_counter.current()` is sync, not async.** Reading an `int` is GIL-atomic in CPython; no await, no lock. This makes E006-F002 trivial — it can read queue depth without serializing on the inference path.

**Error pipeline (CLAUDE.md mandate):** the code `QUEUE_FULL` (HTTP 503) is defined in [packages/error-contracts/errors.yaml](packages/error-contracts/errors.yaml) under E004-F004's contract with params `{max_waiters: int, current_waiters: int}`. Per CLAUDE.md "Never add an error code without editing errors.yaml, running task errors:generate," before implementing F002 confirm the entry exists in `errors.yaml`, run `task errors:generate`, and run `task check:errors`. F002 imports `QueueFullError` from `app.exceptions`.

## Open questions

None. All decisions resolved within thickening: counter as a separate class with a dedicated `asyncio.Lock` (clearer than fold-into-Semaphore); reserve-then-roll-back-on-overflow pattern (atomic, no race); `max_inference_waiters` configurable via `pydantic-settings` to match CLAUDE.md's no-`os.environ` rule (default 4); `WaiterCounter.current()` synchronous (GIL-atomic int read; safe for state-inspection without lock contention); `try/finally` placement guarantees cleanup on any exception path; pre-check fast-fails (`RegistryNotFoundError`, `ModelCapabilityNotSupportedError`) skip `reserve()` entirely (verified by AC).

## Unit test scenarios

- `WaiterCounter().current() == 0` at construction.
- `await counter.reserve(max_waiters=4)` increments `_count` to 1 and returns 1.
- `await counter.reserve(4)` four times sequentially produces returned values 1, 2, 3, 4.
- The fifth `await counter.reserve(4)` raises `QueueFullError(max_waiters=4, current_waiters=5)` and rolls back the increment — `counter.current() == 4` after the raise.
- After 4 reserves and 4 releases, `counter.current() == 0` — invariant pairing.
- `counter.release()` from `_count == 0` decrements to `-1` (test acknowledges the bug-prone case; a future hardening could clamp at 0, but v1 trusts the `try/finally` pairing).
- Concurrent reserves: 5 tasks calling `await counter.reserve(4)` via `asyncio.gather(...)` produce exactly 4 successes and 1 `QueueFullError` — verified by gather + exception inspection.
- `counter.current()` is synchronous (no `await`); verified by inspecting the method signature.
- `Settings.max_inference_waiters` defaults to 4 when env unset.
- `MAX_INFERENCE_WAITERS=2` env var produces `Settings.max_inference_waiters == 2`.
- `MAX_INFERENCE_WAITERS=0` raises Pydantic `ValidationError` at `Settings()` construction (`gt=0`).
- `MAX_INFERENCE_WAITERS=-1` similarly raises.
