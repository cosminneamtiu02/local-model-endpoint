## Short description

`asyncio.wait_for` wraps each inference call with a 180-second timeout; on expiry, the consumer receives HTTP 504 with a structured "inference timeout" DomainError.

## Problem

Without a per-request timeout, a single pathological inference (prompt-injection-induced infinite generation, an Ollama bug producing a runaway loop, a degenerate model state, or simply a 50,000-token prompt that takes minutes to prefill on the M4 Mini) holds the inference semaphore indefinitely. F001-of-E004 serializes; without F003, that one slow request blocks all others until it completes — minutes or hours of accumulated wait.

F003 caps total time-in-flight per request at 180 seconds. On expiry the inference is *cancelled* — `asyncio.wait_for` propagates `asyncio.CancelledError` into `client.chat()`, which cancels the in-flight `httpx` request, releases the semaphore, decrements the waiter counter, and returns HTTP 504 with `INFERENCE_TIMEOUT` DomainError carrying `{timeout_seconds}`. The consumer learns "this inference is dead" within a bounded wall-clock budget instead of waiting forever.

The 180s ceiling is calibrated against measured Gemma 4 E2B throughput (52 tok/s decode, ~6.6s cold load on M4 Mini). 180 - 6.6 ≈ 173 s of decoding, ≈ 9,000 tokens. Cognitive-task responses target <2k tokens; 180s is a "definitely broken" threshold, not a "slow but normal" threshold.

*Stakeholder:* Cosmin (operator + consumer-side dev). *Need:* every inference returns within a bounded wall-clock budget. Failures are programmable (`error.code == "INFERENCE_TIMEOUT"`), retries on the consumer side are predictable, and one stuck request can never starve others.

## Scope

- **Configurable `inference_timeout_seconds` setting** added to `Settings`: `inference_timeout_seconds: int = Field(default=180, gt=0)`. Reads from `LIP_INFERENCE_TIMEOUT_SECONDS` env var. Same configurability rationale as F002's `max_inference_waiters` — pydantic-settings convention; default matches the disambiguated-idea Quality Attribute.
- **`InferenceService.run()` body modified** — the existing post-F002 sequence has its `client.chat(...)` call wrapped in `asyncio.wait_for`:

    ```
    BEFORE F003 (after F001+F002):
        async with self._semaphore:
            queue_wait_ms = ...
            result = await self._client.chat(...)

    AFTER F003:
        async with self._semaphore:
            queue_wait_ms = ...
            try:
                result = await asyncio.wait_for(
                    self._client.chat(...),
                    timeout=self._timeout_seconds,
                )
            except asyncio.TimeoutError as exc:
                raise InferenceTimeoutError(
                    timeout_seconds=self._timeout_seconds
                ) from exc
    ```

- **Timeout placement: *inside* the semaphore lock, *not* outside.** The timeout caps how long the *running* inference can take, not how long it can wait + run. Queue wait is bounded by F002 (max 4 waiters × ≤180s each = 12 minutes worst case, but practically ≤ minutes given Gemma-4-E2B normal latency). The wait_for budget governs the *active* inference only.
- **`InferenceService` constructor extended** with `timeout_seconds: int`. Updated signature:
  `__init__(self, registry, client, semaphore, waiter_counter, max_waiters: int, timeout_seconds: int)`.
- **`get_inference_service()` factory updated** to read `settings.inference_timeout_seconds` and pass it through.
- **`asyncio.TimeoutError` → `InferenceTimeoutError(timeout_seconds=...)` mapping.** The `try/except` is the only place F003 catches an exception. Cancellation behavior of `wait_for` triggers `httpx`'s own cancellation path (request abort, connection cleanup), then raises `asyncio.TimeoutError` at the `wait_for` site. F003 catches and rewraps.
- **Cancellation reaches the adapter cleanly.** `asyncio.wait_for` schedules the cancellation by sending `CancelledError` into the wrapped coroutine; `httpx.AsyncClient` is documented to handle `CancelledError` by closing the underlying connection. No leaked sockets, no zombie requests against Ollama.
- **`finish_reason` not set to `"timeout"` here** — that field exists on the `OllamaChatResult`/`InferenceResponse` for *successful* responses with a length cap; F003's path raises an exception before any `OllamaChatResult` is constructed, so the consumer never sees a 504 response with `finish_reason`. (The `Literal["stop", "length", "timeout"]` Literal includes `"timeout"` for symmetry / future use, but v1 produces only `"stop"` and `"length"` on the success path.)
- **`InferenceTimeoutError` wiring.** F003 imports from `app.exceptions`; F004 of E004 defines the class with HTTP 504 + RFC 7807 rendering.

## Out of scope

- **Streaming-response timeout semantics** — out of v1.
- **Per-request timeout overrides via the envelope** — out of v1; the `params` field has no `timeout` (a deliberate omission in F001-of-E001's envelope schema). Tunable per-deployment via env var, not per-request.
- **Adaptive / dynamic timeout based on prompt length** — out of v1; flat 180s.
- **Different timeout for warm-up vs production traffic** — `LIP-E005-F001` (warm-up) uses its own minimal-prompt request and is unaffected. F003's wrapper catches both equally; warm-up's intentionally-tiny inference completes in under a second.
- **Timeout that includes queue wait** — explicitly *not* what F003 does. The 180s is wall-clock from `wait_for` start (which is *after* `async with self._semaphore` acquires) to the inference's completion or cancellation. Queue-wait time does not count against the budget. (Argument: the semaphore wait is bounded by F002's 4-waiter cap; conflating the two would punish a request that waited but then ran fast.)
- **Retry-After or Retry-Hint headers on 504** — out of v1.
- **Distinguishing "Ollama is hung" from "real long inference"** — F003 doesn't try; both produce a 504 after 180s. Operator triage uses logs / Ollama state.
- **Cancellation of the *waiting* coroutine when timeout would have already expired** — out of v1; a request waits for the semaphore as long as it takes (capped only by F002's queue limit). After acquiring, it gets a fresh 180s window. The naming `INFERENCE_TIMEOUT` (not `REQUEST_TIMEOUT`) reflects this — the timer is on the inference body specifically.

## Acceptance criteria

- A single `POST /v1/inference` with a normal prompt completes in well under 180 s and returns HTTP 200. (Gemma 4 E2B normal latency on M4 Mini: <2 s for a typical cognitive-task response.)
- An inference that exceeds 180 s returns HTTP 504, `Content-Type: application/problem+json`, body code `"INFERENCE_TIMEOUT"`, params `{"timeout_seconds": 180}`. Verified via mock-adapter test: the adapter mock awaits `asyncio.sleep(200)`; the route returns 504 within ~180.x seconds.
- `LIP_INFERENCE_TIMEOUT_SECONDS=2`: a mock-adapter test where Ollama's response takes 5s yields a 504 within ~2.x seconds; a 1s mock yields HTTP 200. End-to-end env-var path verified.
- `LIP_INFERENCE_TIMEOUT_SECONDS=0` is rejected at startup (`gt=0` Field constraint).
- A 504 response releases the semaphore — verified by: hold semaphore via long-running mock; trigger 504; observe a second waiter immediately acquires the semaphore (no leaked permit).
- A 504 response decrements the waiter counter — verified by: with 4 in-flight requests, force one to time out; `waiter_counter.current()` drops to 3; a new request can acquire (5th-waiter rejection no longer fires).
- Cancellation propagates: the `httpx` request to Ollama is *aborted* mid-flight (verified by checking the test fixture's `httpx.AsyncClient` records a cancelled request, not a completed one).
- The 504 response's `metadata` is *not* present (no `InferenceResponse` is composed; the 504 is an `INFERENCE_TIMEOUT` `ProblemDetails` body, not a typed response with `finish_reason="timeout"`).
- A successful inference's `metadata.finish_reason` is `"stop"` or `"length"` — never `"timeout"`. (The Literal includes `"timeout"` for forward compatibility / Pydantic strict-mode comfort, but v1 never produces it on success.)
- The timeout doesn't count semaphore wait: with `LIP_MAX_INFERENCE_WAITERS=2` and `LIP_INFERENCE_TIMEOUT_SECONDS=10`, fire 4 simultaneous requests where each inference takes 5s. The third request waits ~10s for two predecessors then runs successfully (5s) — total wall-clock 15s, well within the per-call timer's window because the 10s is *only* counting the inference body. Verified via mock.
- `Pyright strict` passes; `task check` passes.

**`InferenceService.run()` final body (post-F003, post-F002, post-F001 of E004):**

```python
import asyncio
import time

from app.exceptions import InferenceTimeoutError
# ... other imports unchanged from prior thickenings ...


class InferenceService:
    def __init__(
        self,
        registry: ModelRegistry,
        client: OllamaClient,
        semaphore: asyncio.Semaphore,
        waiter_counter: WaiterCounter,
        max_waiters: int,
        timeout_seconds: int,
    ) -> None:
        self._registry = registry
        self._client = client
        self._semaphore = semaphore
        self._waiter_counter = waiter_counter
        self._max_waiters = max_waiters
        self._timeout_seconds = timeout_seconds

    async def run(
        self,
        envelope: InferenceRequest,
        request_id: str,
    ) -> InferenceResponse:
        start = time.perf_counter()
        model_info = self._registry.get(envelope.model)
        check_envelope_capabilities(envelope, model_info)
        merged = merge_sampling_params(envelope.params, model_info.sampling_defaults)

        await self._waiter_counter.reserve(self._max_waiters)
        try:
            wait_start = time.perf_counter()
            async with self._semaphore:
                queue_wait_ms = int((time.perf_counter() - wait_start) * 1000)
                try:
                    result = await asyncio.wait_for(
                        self._client.chat(
                            model_tag=model_info.backend_tag,
                            messages=envelope.messages,
                            params=merged,
                        ),
                        timeout=self._timeout_seconds,
                    )
                except asyncio.TimeoutError as exc:
                    raise InferenceTimeoutError(
                        timeout_seconds=self._timeout_seconds,
                    ) from exc
        finally:
            await self._waiter_counter.release()

        latency_ms = int((time.perf_counter() - start) * 1000)
        return InferenceResponse(
            content=result.content,
            metadata=ResponseMetadata(
                model=envelope.model,
                prompt_tokens=result.prompt_tokens,
                completion_tokens=result.completion_tokens,
                request_id=request_id,
                latency_ms=latency_ms,
                queue_wait_ms=queue_wait_ms,
                finish_reason=result.finish_reason,
                backend="ollama",
            ),
        )
```

**Why `asyncio.wait_for` and not `asyncio.timeout`:** Both work in Python 3.13. `wait_for` is the older idiom and slightly more explicit ("call this coroutine; cap it at N seconds"). `asyncio.timeout` is a context manager that's nicer for multi-statement guarded blocks — overkill here for a single `await`. Either is fine; locking `wait_for` for clarity.

**Cancellation correctness:** When `wait_for` fires:
1. `wait_for` sends `CancelledError` into `client.chat()`.
2. `client.chat()` is mid-`await` on `_request("POST", "/api/chat", ...)` → the `httpx` request's underlying socket is cancelled, connection released back to pool (or closed).
3. The exception unwinds out of `chat()`, out of `wait_for`'s scope.
4. `wait_for` re-raises as `asyncio.TimeoutError`.
5. F003's `except` catches and rewraps as `InferenceTimeoutError`.
6. Exception propagates out of `async with self._semaphore` → semaphore released.
7. Exception propagates out of the `try/finally` → `waiter_counter.release()` runs → counter decrements.
8. Exception propagates out of `run()` to the FastAPI exception handler → RFC 7807 504 response.

Each step is documented behavior of the relevant module; no custom cleanup logic. `async with` and `try/finally` carry the cleanup discipline for free.

**Why timeout *inside* the semaphore, not wrapping it:** wrapping `async with self._semaphore` plus the chat call would force the timer to count semaphore-wait time. Under contention, slow waiters would time out *before reaching the inference at all*, defeating both fairness (FIFO under-served) and `INFERENCE_TIMEOUT`'s semantic clarity ("the inference itself took too long"). The chosen placement keeps the timer scoped to the inference body, making the 504 a precise diagnostic signal. F002's queue cap (4 waiters) plus per-call timeout (180s) bound the worst-case end-to-end at ~12 minutes — already useful, and the actual measured Gemma-4-E2B latency means the 90th percentile is seconds, not minutes.

**Pyright strict typing:** `asyncio.wait_for(...)` is fully typed; `asyncio.TimeoutError` is `concurrent.futures.TimeoutError`'s alias since 3.11 — Pyright accepts both spellings. `try: ... except asyncio.TimeoutError` is the canonical form.

## Open questions

None. All decisions resolved within thickening: timeout 180 s default (calibrated to Gemma-4-E2B + cognitive-task workload, configurable via env); `asyncio.wait_for` over `asyncio.timeout` (single-await idiom); timeout *inside* semaphore lock so it scopes the inference body, not the wait; `INFERENCE_TIMEOUT` is a separate error from `ADAPTER_CONNECTION_FAILURE` — daemon being down is a different failure than inference-runs-too-long; cancellation propagation carries the cleanup discipline for free via `async with` + `try/finally`.

## Unit test scenarios

- `InferenceService.run()` with a fast mock adapter (returns immediately) does NOT raise `InferenceTimeoutError`.
- `InferenceService.run()` with a slow mock adapter (`asyncio.sleep(timeout_seconds + 1)`) and `timeout_seconds=2` raises `InferenceTimeoutError(timeout_seconds=2)`.
- The raised `InferenceTimeoutError`'s `__cause__` is an `asyncio.TimeoutError` instance (verified via `from exc` chain).
- `Settings.inference_timeout_seconds` defaults to 180 when env unset.
- `LIP_INFERENCE_TIMEOUT_SECONDS=2` env var produces `Settings.inference_timeout_seconds == 2`.
- `LIP_INFERENCE_TIMEOUT_SECONDS=0` raises Pydantic `ValidationError` at `Settings()` construction (`gt=0`).

- The timeout *only* wraps the `client.chat()` call — verified that the registry lookup, capability check, and merge happen *before* the `asyncio.wait_for` and are not subject to the timeout (mock that delays each step in turn and observes which one triggers the timeout).
- Successful inference's `metadata.finish_reason` is one of `"stop"`, `"length"` — never `"timeout"` (timeout produces a 504, not a 200 with finish_reason).
