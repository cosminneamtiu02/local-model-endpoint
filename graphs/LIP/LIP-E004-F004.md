## Short description

DomainError base class with subclasses (queue-full, inference-timeout, adapter-connection-failure, validation-error, registry-not-found, etc.); a FastAPI exception handler maps each subclass to a structured JSON response with the appropriate HTTP status code.

## Problem

The DomainError hierarchy is the single mechanism by which every failure mode in LIP becomes a typed, contract-documented HTTP response that consumer projects can program against. Without it, every error path either returns a raw FastAPI `HTTPException` (forbidden by CLAUDE.md's "Never raise HTTPException" rule) or produces an unstructured 500 response that consumers cannot distinguish between adapter connection failure, queue full, model not found, validation error, etc. F004 is the structured-error half of G3 (single observability point) — DomainError-shaped responses are the consumer-visible part of observability even before structured logging arrives in a future milestone. It is also load-bearing for E004-F002 (queue+503), E004-F003 (timeout+504), E003-F003 (Ollama failure mapping), and E001-F002 (orchestration), all of which raise specific DomainError subclasses defined here.

*Stakeholder:* Cosmin (consumer-side developer + maintainer + operator). *Need:* every failure mode produces a predictable JSON response with a stable typed error code, so consumer code can pattern-match (`if error.code == "QUEUE_FULL": retry_with_backoff()`) and so future-Cosmin can audit error responses without parsing free-text.

## Scope

- **Five new error codes added to `packages/error-contracts/errors.yaml`** (regenerated via `task errors:generate` into `apps/backend/app/exceptions/_generated/`):
    - `QUEUE_FULL` — HTTP 503, params `{max_waiters: integer, current_waiters: integer}`. Raised by E004-F002 when the bounded waiter count is exceeded.
    - `INFERENCE_TIMEOUT` — HTTP 504, params `{timeout_seconds: integer}`. Raised by E004-F003 when `asyncio.wait_for` fires.
    - `ADAPTER_CONNECTION_FAILURE` — HTTP 502, params `{backend: string, reason: string}`. Raised by E003-F003 for connection refused, non-2xx, malformed JSON, or any Ollama-specific failure caught at the adapter boundary. Single umbrella code with `reason` carrying the specific failure mode (no contract change to add new failure modes).
    - `REGISTRY_NOT_FOUND` — HTTP 404, params `{model: string}`. Raised by E002-F001 when the consumer's logical model name doesn't exist in the registry.
    - `MODEL_CAPABILITY_NOT_SUPPORTED` — HTTP 422, params `{model: string, requested_capability: string}`. Raised by E001-F002 (orchestrator) using E002-F003's capability flag exposure when a consumer's request includes content the model doesn't support.
- **Wire shape switched to RFC 7807 (`application/problem+json`).** Response body becomes:
    ```json
    {
      "type": "urn:lip:error:queue-full",
      "title": "Queue Full",
      "status": 503,
      "detail": "Inference queue at capacity (5 waiters, max 4)",
      "instance": "/api/v1/inference",
      "code": "QUEUE_FULL",
      "request_id": "abc-123",
      "max_waiters": 4,
      "current_waiters": 5
    }
    ```
   Five RFC 7807 standard fields (`type`, `title`, `status`, `detail`, `instance`), plus two project extensions (`code`, `request_id`), plus the error's typed params at root level. `Content-Type: application/problem+json`. Replaces the post-bootstrap `{"error": {...}}` envelope.
- **DomainError base class extended.** New `ClassVar` fields: `type_uri: str`, `title: str`. New `detail()` method that formats the per-instance human-readable explanation from params via the template. Existing `code: str`, `http_status: int`, `params` are unchanged.
- **Existing schemas replaced.** `app/schemas/error_body.py`, `error_response.py`, and `error_detail.py` are deleted and replaced by `app/schemas/problem_details.py` (RFC 7807 + LIP extensions) and `app/schemas/validation_error_detail.py` (renamed from `error_detail.py` for clarity since "detail" overloads with RFC 7807's `detail` field). `app/schemas/__init__.py` re-export updated.
- **Codegen extended (`packages/error-contracts/scripts/generate.py`).** Reads new `title` and `detail_template` fields from each errors.yaml entry; auto-derives `type_uri` from code as `urn:lip:error:` + `code.lower().replace('_', '-')`; generates `type_uri: ClassVar[str]`, `title: ClassVar[str]`, and a `detail() -> str` method (template substitution via `str.format(**params)` if params, else returns title as fallback) on each error class. The codegen tests in `packages/error-contracts/tests/test_generate.py` are updated to verify the new fields.
- **FastAPI exception handler rewritten (`app/api/errors.py`).** Three handlers (DomainError, RequestValidationError, generic Exception) now produce RFC 7807-shaped responses with `Content-Type: application/problem+json`. The validation-error handler adds a `validation_errors: [{field: string, reason: string}]` extension array since RFC 7807 has no standard validation-error field. All five existing generic error codes (NOT_FOUND, CONFLICT, VALIDATION_FAILED, INTERNAL_ERROR, RATE_LIMITED) get `title` and `detail_template` added in errors.yaml so they fit the new shape without breakage.

## Out of scope

- **Structured-log emission of error events** — out of v1 per Project Boundary; v1 maps to JSON, no structlog event at error time beyond request-middleware fallout.
- **Error code translation to user-facing messages** — no i18n layer (stripped during bootstrap); `title` and `detail` are hard-coded English strings. Consumers pattern-match on `code` and construct their own user-facing messages if needed.
- **`Retry-After` HTTP header on `RATE_LIMITED` or `QUEUE_FULL`** — out of v1; useful future addition. The `retry_after_seconds` param exists on RATE_LIMITED but no HTTP header is set.
- **The actual semaphore / queue-depth enforcement** — E004-F002. F004 only defines the `QUEUE_FULL` type.
- **The actual `asyncio.wait_for` timeout enforcement** — E004-F003.
- **Ollama-specific exception catching** — E003-F003. F004 only defines `ADAPTER_CONNECTION_FAILURE`; the adapter raises it.
- **Per-project rate-limiting logic** — out of v1; `RATE_LIMITED` exists but nothing in v1 raises it. G4 architectural foundation only.
- **Tool-call error codes** — out of v1.
- **Streaming error semantics** — out of v1.
- **Stack-trace inclusion in error responses** — out of v1; stack traces (for `INTERNAL_ERROR`) are logged structlog-side but never sent to consumers (no PII leak).
- **Resolvable `type` URIs (hosted docs pages)** — out of v1; `type` is a non-resolvable URN per RFC 7807 §3.1's "consumers SHOULD NOT auto-dereference" guidance. If hosted docs are added later, a stable URI mapping can be introduced without breaking the URN format.

- Each generated error class has `code: ClassVar[str]`, `http_status: ClassVar[int]`, `type_uri: ClassVar[str]` (URN derived from code), `title: ClassVar[str]`, and a `detail()` method that returns the rendered template with params substituted.
- The 5 existing generic codes (NOT_FOUND, CONFLICT, VALIDATION_FAILED, INTERNAL_ERROR, RATE_LIMITED) regenerate cleanly with the new `title`/`detail_template` fields after they're added to errors.yaml.
- Raising `QueueFullError(max_waiters=4, current_waiters=5)` from a route produces an HTTP 503 response with `Content-Type: application/problem+json` and the RFC 7807 + extension body shape.
- Response body has all five RFC 7807 standard fields populated: `type` (`"urn:lip:error:queue-full"`), `title` (`"Queue Full"`), `status` (503), `detail` (rendered template with params substituted), `instance` (the request path).
- Response body has the two project extensions populated: `code` (`"QUEUE_FULL"`), `request_id` (the UUID from the RequestIdMiddleware).

- An unhandled `Exception` produces an HTTP 500 RFC 7807 response with code `"INTERNAL_ERROR"`, no params, and a generic static `detail` (no PII leak; no stack trace exposure).
- The auto-generated OpenAPI schema served at `/openapi.json` references the `ProblemDetails` schema in `responses=` declarations on the future inference endpoint (LIP-E001-F002 wires this; F004 ships the schema definition).
- Existing tests in `tests/unit/exceptions/test_error_handler.py` and `tests/integration/test_health.py` are updated to assert the RFC 7807 shape; new tests cover all 5 LIP-specific codes and the `validation_errors` extension array.
- Codegen tests in `packages/error-contracts/tests/test_generate.py` verify the new `title`/`detail_template`/`type_uri` fields are produced correctly, including URN derivation (`QUEUE_FULL` → `urn:lip:error:queue-full`).
- After the codegen change, `task check:errors` passes (regenerates and runs `git diff --exit-code` cleanly).

## Technical constraints

**RFC 7807 conformance:**

- Response `Content-Type: application/problem+json` (not `application/json`) for every error response, per RFC 7807 §3.
- Body must contain all five RFC 7807 standard fields (`type`, `title`, `status`, `detail`, `instance`); each is required.
- `type` is a URN of the form `urn:lip:error:<code-kebab>` — a non-resolvable identifier per RFC 7807 §3.1.
- Project extensions: `code: str` (SCREAMING_SNAKE for ergonomic pattern matching), `request_id: str` (UUID from the RequestIdMiddleware).
- The error's typed params are spread at the root level of the response body (not nested). This matches RFC 7807's extension-field convention.
- For `VALIDATION_FAILED` only, an additional extension: `validation_errors: list[{field: str, reason: str}]` — carries per-field error details since RFC 7807 has no standard validation-error field.
- The `instance` field is the request URL path (e.g., `/api/v1/inference`); not the full URL, not method-prefixed, not a unique-per-occurrence URI.

**File layout (one class per file, per CLAUDE.md):**

```
apps/backend/app/schemas/
├── problem_details.py            # ProblemDetails (replaces error_body + error_response)
└── validation_error_detail.py    # ValidationErrorDetail (renamed from error_detail.py)

apps/backend/app/exceptions/_generated/
├── queue_full_error.py           # one per error class, regenerated
├── queue_full_params.py
├── inference_timeout_error.py
├── inference_timeout_params.py
├── adapter_connection_failure_error.py
├── adapter_connection_failure_params.py
├── registry_not_found_error.py
├── registry_not_found_params.py
├── model_capability_not_supported_error.py
└── model_capability_not_supported_params.py
# (plus regenerated existing 5 generic codes)
```

**`errors.yaml` schema extension:**

```yaml
QUEUE_FULL:
  http_status: 503
  description: Inference queue at capacity, request rejected   # internal-only docs
  title: Queue Full                                            # RFC 7807 title (short, summary)
  detail_template: "Inference queue at capacity ({current_waiters} waiters, max {max_waiters})"
  params:
    max_waiters: integer
    current_waiters: integer
```

**Codegen extensions** (`packages/error-contracts/scripts/generate.py`):

- Read new fields `title`, `detail_template` from each error entry; both required.
- Auto-derive `type_uri` from `code`: `urn:lip:error:` + `code.lower().replace('_', '-')`.
- Generate `type_uri: ClassVar[str]`, `title: ClassVar[str]` on each error class.
- Generate a `detail(self) -> str` method: `self.detail_template.format(**self.params.model_dump())` if params, else returns `self.title` as fallback.

**Pydantic config on `ProblemDetails`:**

- `model_config = ConfigDict(extra="allow")` — RFC 7807 explicitly permits extension fields, so the schema must accept arbitrary additional fields. (Intentional asymmetry with F001's request-side `extra="forbid"` — request schemas catch consumer bugs by rejecting unknowns; response schemas invite extension by design.)

**Existing-file rewrites required (not just additions):**

- `app/schemas/error_body.py`, `app/schemas/error_response.py`: deleted.
- `app/schemas/error_detail.py`: renamed to `validation_error_detail.py` (one-class-per-file, name reflects content).
- `app/schemas/__init__.py`: re-exports updated.
- `app/api/errors.py`: handler functions rewritten to construct `ProblemDetails` responses with `media_type="application/problem+json"`.
- `tests/unit/exceptions/test_error_handler.py`: assertions updated for RFC 7807 shape.
- `tests/integration/test_health.py`: any error-shape assertions updated.
- `packages/error-contracts/tests/test_generate.py`: codegen tests updated for new fields.
- `packages/error-contracts/errors.yaml`: existing 5 generic entries get `title` and `detail_template` added.

## Open questions

- *[UNRESOLVED]* **Exact wording of each error's `title` and `detail_template`.** These are user-facing strings (well, consumer-developer-facing) that a UX-aware human should review during implementation. Codegen passes through whatever `errors.yaml` provides verbatim; F004 implementation should walk through each title/template and tighten the wording before committing. Example: `QUEUE_FULL` title `"Queue Full"` might be clearer as `"Inference Queue Full"`.
- *[UNRESOLVED]* **Stable ordering of `validation_errors` extension array items.** Pydantic's `errors()` iteration order is deterministic but field-declaration-order-dependent, which makes test snapshots fragile across model edits. Sorted-by-field is more stable across schema changes but reorders related errors away from the order the consumer's request listed them. Decision deferred to feature-dev based on what ergonomics test snapshots reveal.

## Unit test scenarios

- `task errors:generate` against an `errors.yaml` containing exactly the 5 LIP codes + 5 generic codes produces 10 generated `*_error.py` files plus param files where params are declared.
- Each generated error class has `code: ClassVar[str]`, `http_status: ClassVar[int]`, `type_uri: ClassVar[str]`, `title: ClassVar[str]` populated from `errors.yaml` (verified per code).
- The `type_uri` derivation rule produces `urn:lip:error:queue-full` from `QUEUE_FULL`, `urn:lip:error:adapter-connection-failure` from `ADAPTER_CONNECTION_FAILURE`, etc. (kebab-case lowercase).
- A generated error class with params (`QueueFullError(max_waiters=4, current_waiters=5)`) instantiates without error and `detail()` returns the `detail_template` with both params substituted.
- A generated error class without params (`InternalServerError()`) instantiates without args and `detail()` returns the `title` as the fallback string.
- Calling `detail()` on a partially-constructed error (missing required param) raises a `KeyError` (template substitution fails) — exception ordering test.
- The `ProblemDetails` Pydantic schema accepts the canonical 5 RFC 7807 fields + `code` + `request_id` extensions + arbitrary additional fields per `extra="allow"`.
- The `ProblemDetails` schema rejects when `status` is missing (required field).
- The `ValidationErrorDetail` schema accepts `{"field": "messages.0.role", "reason": "Input should be 'user', 'assistant' or 'system'"}` and rejects unknown keys.
- The codegen tests in `packages/error-contracts/tests/test_generate.py` verify each of the 10 codes' generated output matches a fixture file character-for-character (snapshot test).
- `task check:errors` regenerates `_generated/` and `git diff --exit-code` passes when nothing has changed since the last regeneration.
- `task check:errors` fails (non-zero exit) when `errors.yaml` is edited but `task errors:generate` has not been re-run — drift detection.

## Integration test scenarios

- A FastAPI route raising `QueueFullError(max_waiters=4, current_waiters=5)` produces an HTTP 503 response with `Content-Type: application/problem+json` and a body matching the RFC 7807 + extensions shape.
- The 503 response body contains all 5 RFC 7807 fields populated: `type="urn:lip:error:queue-full"`, `title="Queue Full"`, `status=503`, `detail` with substituted params, `instance` set to the request's URL path.
- The 503 response body contains both project extensions: `code="QUEUE_FULL"`, `request_id` matching the `X-Request-ID` response header (RequestIdMiddleware integration).
- The 503 response body contains the typed params spread at root: `max_waiters=4`, `current_waiters=5`.
- A FastAPI route receiving an envelope with an invalid `role` produces an HTTP 422 RFC 7807 response with `code="VALIDATION_FAILED"` and a `validation_errors` extension array listing each failed field with `{"field": "...", "reason": "..."}`.
- A FastAPI route raising an unhandled `RuntimeError` produces an HTTP 500 RFC 7807 response with `code="INTERNAL_ERROR"`, no params, generic `detail` (no PII or stack-trace leak).
- The exception handler chain orders DomainError before RequestValidationError before generic Exception (verified by raising each from a test route and observing correct mapping).
- Each of the 5 LIP codes (`QUEUE_FULL`, `INFERENCE_TIMEOUT`, `ADAPTER_CONNECTION_FAILURE`, `REGISTRY_NOT_FOUND`, `MODEL_CAPABILITY_NOT_SUPPORTED`) maps to its declared HTTP status when raised from a test route — verified per code.
- Each of the 5 generic codes (`NOT_FOUND`, `CONFLICT`, `VALIDATION_FAILED`, `INTERNAL_ERROR`, `RATE_LIMITED`) maps correctly when raised from a test route.

## E2E test scenarios

Not applicable. End-to-end serialization of the RFC 7807 wire shape is already exercised by two concrete flows specified elsewhere, both of which traverse the same `app/api/errors.py` handlers F004 ships:

- The inference router's *integration* scenarios (LIP-E001-F002) raise real `RegistryNotFoundError` and `ModelCapabilityNotSupportedError` instances, which already assert the full `application/problem+json` body shape — `type`, `title`, `status`, `detail`, `instance`, `code`, `request_id`, plus typed params at root — against `REGISTRY_NOT_FOUND` and `MODEL_CAPABILITY_NOT_SUPPORTED` raised through the real handler stack. This covers the DomainError handler path end-to-end.
- The QoS-layer E2E flows (LIP-E004-F002, LIP-E004-F003) raise `QueueFullError` and `InferenceTimeoutError` against a running app under simulated load and assert the same RFC 7807 shape. This covers the second branch of the DomainError handler path under realistic concurrency.

A standalone F004 E2E test would either duplicate those flows or invent a synthetic route that raises a DomainError purely for testing — adding no marginal coverage of the handler logic that isn't already exercised by features which raise these errors as part of their own user-facing behavior.
