## Short description

A FastAPI startup hook performs a single minimal inference using the registry's default model, ensuring Ollama loads the model into memory before readiness is signalled to consumers.

## Problem

On a cold-started Ollama, the first inference call pays a ~6.6s model-load cost (Gemma 4 E2B on M4 Mini, measured) before producing a single token. Without warm-up, the *first consumer's* first request absorbs that latency, manifesting as a 6+ second response time that violates the project's "≤2s normal latency" Quality Attribute. The consumer cannot distinguish "model is loading" from "service is broken," and the synchronous-load failure mode silently inflates first-call latency.

F001 moves the cold-load cost from "first consumer call" to "service startup," and gates `/health`'s readiness signal on warm-up completion. Consumers issuing requests during the warm-up window receive a clear not-ready signal from `/health` (handled by E006-F001) and can wait or back off. By the time `/health` returns ready, the model is in Ollama's GPU memory and the next inference call has its expected sub-second prefill + sub-second decode latency.

The warm-up call itself goes through the *full production path* — registry lookup, capability check, parameter merge, adapter call — so any mis-wiring is detected at startup, not at first user request. This is the most aggressive version of "fail at the right time": startup failures are operator-visible and cheap to debug; first-request failures look like consumer bugs.

*Stakeholder:* Cosmin (operator + consumer-side dev). *Need:* the first inference any consumer makes is fast, predictable, and indistinguishable from any subsequent inference. Cold-load latency is moved off the consumer-visible critical path.

## Scope

- **`async def warm_up_inference(client, registry, logical_name="default-task", timeout_seconds=30) -> None`** function in `apps/backend/app/features/inference/service/startup_warmup.py`. Pure async function — no class, no state. Performs:
    1. `model_info = registry.get(logical_name)` — verifies the registry entry exists and resolves the backend tag. Raises `RegistryNotFoundError` if missing (caught at the lifespan level).
    2. `await asyncio.wait_for(client.chat(...), timeout=timeout_seconds)` — sends a minimal Ollama call wrapped in a wall-clock budget. The 2-character user message ensures prompt prefill is essentially free; `max_tokens=1` ensures decode is a single token. The `wait_for` wrapper bounds total warm-up time; without it, a stuck or slow-loading Ollama could hang lifespan startup indefinitely (uvicorn never reaches `yield` → listen socket never opens → no traffic served, with no exception for the broad lifespan-level `except` to catch).
    3. Returns `None` on success. Raises `AdapterConnectionFailureError` on backend failure or `asyncio.TimeoutError` on `timeout_seconds` exceeded — both fall into the lifespan's broad `except Exception` and produce a `warmup_failed` structured-log event with the service running unready.
- **Default logical-name parameter `"default-task"`** matches the v1 registry entry from `LIP-E002-F001`. The function is parametric to avoid hardcoding inside the lifespan; future multi-model deployments add a setting (`warmup_logical_name`) that flows through. v1 ships with the default value never overridden.
- **Configurable `warmup_timeout_seconds` setting** added to `Settings`: `warmup_timeout_seconds: int = Field(default=30, gt=0)`. Reads from `WARMUP_TIMEOUT_SECONDS`. The 30s default gives ~5x margin over the measured ~6.6s cold-load on the M4 Mini, enough for system noise without letting startup hang for minutes if Ollama wedges. Separate from `INFERENCE_TIMEOUT_SECONDS` (180s default for production traffic) — warm-up is single-token and shouldn't share the production wall-clock budget.
- **Lifespan integration in `app/main.py`** — the existing `lifespan` from F002-of-E001 (which constructs the `OllamaClient`) is extended to: (a) initialize `app.state.is_ready = False` immediately, (b) construct the `OllamaClient`, (c) construct a `ModelRegistry` from `DEFAULT_MODELS`, (d) call `warm_up_inference(client, registry)`, (e) on success, set `app.state.is_ready = True`, (f) on failure, log a structured `"warmup_failed"` event with the exception details and *leave `is_ready` False* — the service still listens, but `/health` reports not-ready until the operator intervenes (restart of LIP and/or Ollama).
- **Readiness state mechanism** — single boolean on `app.state.is_ready: bool`. No dedicated `ReadinessState` class in v1; the failure scenarios are unary ("warm-up succeeded" vs "didn't"), and the `/health` endpoint (E006-F001) only needs the bool. If future health-check requirements grow (multi-model warm-up, partial readiness states), introducing a class is a refactor; v1 keeps it minimal.
- **Warm-up runs inside the existing lifespan, before the FastAPI route is open to traffic.** No race condition between warm-up completion and serving real requests — `lifespan`'s `yield` is what un-blocks the server, and we set `is_ready=True` *before* yielding. (Technically the route accepts connections from the moment `lifespan` returns the `yield`, but the readiness flag is set in the *same* statement-block before yield, so any request in that microsecond sees a ready service.)
- **Exception handling at lifespan**: `try/except Exception` wraps `warm_up_inference`. Logs `logger.exception("warmup_failed", logical_name="default-task")`. Does *not* re-raise — startup continues. The service binds, accepts connections, and responds to `/health` with not-ready. Inference requests during this window will hit the not-ready guard in `/health` (E006-F001) before reaching the orchestrator; if they reach the orchestrator anyway (e.g., `/v1/inference` is not gated by readiness — by design, so consumers can still attempt requests), they retry the Ollama call at request time, which either succeeds (and Ollama is now reachable) or returns the same `AdapterConnectionFailureError` the warm-up saw. Either way, no zombie state.
- **Warm-up does not invoke the orchestrator (`InferenceService.run`).** Calling `run()` from lifespan would force-construct the entire DI graph at startup *and* exercise the semaphore + waiter counter, polluting their state with a synthetic call. Instead, warm-up calls `client.chat()` directly with a constructed `ModelInfo` from the registry — bypassing the QoS layer that hasn't started serving real traffic yet. The semantics are "warm Ollama," not "exercise the full request pipeline."

## Out of scope

- **Idle-shutdown timer** — `LIP-E005-F002`. Independent feature; F002 manages process exit, F001 manages process startup.
- **Ollama `launchd` plist** — `LIP-E005-F003`. F001 of E005 assumes Ollama is reachable at `OLLAMA_HOST` when warm-up runs; if not, the failure path applies.
- **Multi-model warm-up** — out of v1; warms only the `"default-task"` entry. Adding a `warmup_logical_names: list[str]` setting later is purely additive.
- **Warm-up retry on failure** — out of v1; one-shot. Operator restart re-tries.
- **Warm-up during runtime (not just startup)** — out of v1. Idle shutdown after 10 minutes (F002 of E005) means the next consumer's request triggers cold-load; no mid-runtime re-warm.
- **Configurable warm-up prompt or token count** — out of v1; hardcoded `Message(role="user", content="ok")` + `ModelParams(max_tokens=1)`. The exact bytes don't matter for warming purposes.
- **Warm-up timing observability** — out of v1; the structured-log event for warm-up duration is a future addition. The warm-up's wall-clock time is logged at the end via a `logger.info("warmup_complete", duration_ms=...)` event for operator visibility, but no structured metrics.
- **Warm-up status reflected in `/openapi.json`** — out of scope; `is_ready` is operational state, not contract.
- **Synchronous warm-up that blocks the FastAPI process from binding** — F001 runs warm-up *during* lifespan startup; the listen socket is already bound by the time lifespan begins, but the route handlers don't run until `yield`. This is FastAPI's standard pattern; nothing to add.

## Acceptance criteria

- After clean startup (Ollama up, model not loaded), `app.state.is_ready` becomes `True` within 10 seconds. (Cold-load measured at ~6.6s on M4 Mini; allow 50% overhead for variability.)
- After clean startup, `app.state.is_ready` is `True` *before* the FastAPI server begins accepting requests on the listen socket. (Verified via integration test that connects to the server and immediately checks `app.state.is_ready` from inside a route — the route should never see `False` once the server is reachable.)
- A `POST /v1/inference` issued *immediately* after the server reports listening returns sub-second wall-clock latency (model already loaded; not the cold-load 6.6s).
- With Ollama down at startup, the FastAPI server still binds and accepts connections; `app.state.is_ready` remains `False`; a structured-log event `warmup_failed` is recorded with the underlying exception (`AdapterConnectionFailureError(reason="connection_refused")`).
- With Ollama reachable but the model-load hung (mocked via a non-responding endpoint), warm-up exits within `WARMUP_TIMEOUT_SECONDS` (default 30s) raising `asyncio.TimeoutError`; the lifespan's broad `except Exception` catches it, logs `warmup_failed`, and the server still binds with `app.state.is_ready=False`. Verified that lifespan never blocks past `warmup_timeout_seconds + small overhead`.
- `WARMUP_TIMEOUT_SECONDS=0` is rejected at startup (`gt=0`).
- Warm-up calls the same `OllamaClient` instance that production traffic uses — verified by `id(app.state.ollama_client) == id(client_passed_to_warmup)`.
- Warm-up does NOT acquire `get_inference_semaphore()` — verified by `get_inference_semaphore()._value == 1` immediately after lifespan completes (semaphore is unused; full permit available).
- Warm-up does NOT increment the waiter counter — verified by `get_waiter_counter().current() == 0` after lifespan completes.
- Warm-up emits a structured-log `warmup_complete` event with `duration_ms: int` on success — verified by capturing structlog output during a startup test.
- A logical-name typo (e.g., setting the warm-up to call `"default-tsak"`) raises `RegistryNotFoundError`, the lifespan catches it, logs `warmup_failed`, and the server still starts. (No silent failures from misconfigurations; structured-log carries the exact exception.)
- The warm-up function `warm_up_inference` is a module-level `async def`, not a class method. (Per CLAUDE.md "one class per file" applies to *classes*; pure functions live as module-level.)
- Pyright strict passes; `task check` passes.

**`startup_warmup.py` implementation:**

```python
import asyncio

import structlog

from app.features.inference.model.message import Message
from app.features.inference.model.model_params import ModelParams
from app.features.inference.repository.ollama_client import OllamaClient
from app.features.inference.service.model_registry import ModelRegistry

logger = structlog.get_logger(__name__)


async def warm_up_inference(
    client: OllamaClient,
    registry: ModelRegistry,
    logical_name: str = "default-task",
    timeout_seconds: int = 30,
) -> None:
    """Issue a single minimal inference call to load the default model into Ollama's memory.

    Raises RegistryNotFoundError if the logical name is unknown; raises
    AdapterConnectionFailureError if the Ollama daemon is unreachable or returns
    an unexpected shape; raises asyncio.TimeoutError if the call exceeds
    timeout_seconds (caps total lifespan startup time so a stuck Ollama cannot
    hang the listen socket from opening). Caller (lifespan) is responsible for
    catching these and deciding the resulting readiness state.
    """
    model_info = registry.get(logical_name)
    await asyncio.wait_for(
        client.chat(
            model_tag=model_info.backend_tag,
            messages=[Message(role="user", content="ok")],
            params=ModelParams(max_tokens=1),
        ),
        timeout=timeout_seconds,
    )
```

**`main.py` lifespan modified:**

```python
import time
from contextlib import asynccontextmanager
from collections.abc import AsyncIterator

import structlog
from fastapi import FastAPI

from app.api.health_router import router as health_router
from app.core.config import get_settings
from app.features.inference.repository.ollama_client import OllamaClient
from app.features.inference.router.inference_router import router as inference_router
from app.features.inference.service.default_models import DEFAULT_MODELS
from app.features.inference.service.model_registry import ModelRegistry
from app.features.inference.service.startup_warmup import warm_up_inference

logger = structlog.get_logger(__name__)


@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncIterator[None]:
    settings = get_settings()
    client = OllamaClient(base_url=settings.ollama_host)
    app.state.ollama_client = client
    app.state.is_ready = False

    registry = ModelRegistry(DEFAULT_MODELS)
    start = time.perf_counter()
    try:
        await warm_up_inference(
            client, registry, timeout_seconds=settings.warmup_timeout_seconds
        )
        duration_ms = int((time.perf_counter() - start) * 1000)
        app.state.is_ready = True
        logger.info("warmup_complete", duration_ms=duration_ms, logical_name="default-task")
    except Exception:
        logger.exception("warmup_failed", logical_name="default-task")
        # app.state.is_ready stays False; service still starts.

    try:
        yield
    finally:
        await client.close()


def create_app() -> FastAPI:
    app = FastAPI(title="Local Inference Provider", lifespan=lifespan)
    app.include_router(health_router)
    app.include_router(inference_router)
    return app
```

**Why a module-level function not a class:** the warm-up has no state, no configuration beyond the parameters, and is called exactly once. A class would just add ceremony. Per CLAUDE.md "one class per file" governs classes, not functions; module-level functions in `service/` files are an established pattern (see `sampling_merge.py`, `capability_check.py`).

**Why `app.state.is_ready` not a `ReadinessState` class:** v1 has unary readiness — either warm-up succeeded or it didn't. A bool captures that exactly. A class with `is_ready: bool, last_check: datetime, reason: str | None` over-models for cases that don't exist in v1. If multi-model warm-up or partial-readiness scenarios appear, a refactor introduces the class then; v1 stays minimal.

**Why warm-up doesn't go through `InferenceService.run()`:** the orchestrator threads a request through registry → capability check → semaphore → waiter counter → adapter. At startup, the QoS layer (semaphore, counter) hasn't started serving real traffic and shouldn't be polluted by a synthetic call. Calling `client.chat()` directly bypasses the QoS layer cleanly. The integrity check ("registry lookup works, adapter responds") is preserved by going through the registry to resolve `backend_tag`.

**Why catch `Exception` (broad) at lifespan level, not specific DomainErrors:** the lifespan is *the* recovery boundary for startup failures. Granularity at this level adds nothing — any failure becomes "service starts but unready." A narrower catch would just let an unexpected error type kill the process, which is *worse* operator UX (debug a stopped service rather than a not-ready one). The structured-log event preserves the exception type and traceback for diagnosis.

**Why the broad catch doesn't violate CLAUDE.md's "Never write a `try/except` that silently swallows errors":** the catch *logs* via `logger.exception(...)`. Silent swallowing means catching without logging or re-raising. F001 logs structlog `warmup_failed` with the full exception traceback. The `is_ready` flag itself is the consumer-visible signal; no error is hidden.

**Pyright strict typing:** `app.state` is dynamically typed in FastAPI/Starlette (`Any`-ish). Setting `app.state.is_ready = False` doesn't typecheck statically; reads require `cast(bool, app.state.is_ready)` or per-feature accessor wrappers. v1 lives with the dynamic typing; the readiness boolean is read in exactly one place (E006-F001's `/health` endpoint), so a single `cast` there is acceptable. (If the surface grows, a `ReadinessState` Protocol can be introduced.)

**Lifespan ordering invariant:** `is_ready = False` is set *before* the warm-up call, ensuring any race-prone read sees False until warm-up explicitly sets True. Setting True only after the warm-up await completes is a strict happens-before relationship.
