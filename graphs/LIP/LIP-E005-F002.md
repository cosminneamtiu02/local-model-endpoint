## Short description

An internal timer tracks the timestamp of the last inference; when idle exceeds 10 minutes, the FastAPI process exits cleanly so the next consumer who needs inference re-wakes it.

## Problem

LIP runs on the user's M4 Mac Mini base — a workstation, not a dedicated server. Idle FastAPI processes plus the loaded Gemma 4 E2B model in Ollama's GPU memory consume baseline RAM (the model alone occupies ~3 GB depending on quant) for hours after the last consumer call. Without F002, a single dev-session's worth of consumer requests leaves LIP and Ollama "warm" indefinitely, eating system memory the user wants for other work.

F002 implements G6 (on-demand lifecycle) directly: after 10 minutes with no inference traffic, the FastAPI process exits cleanly via `SIGTERM`-to-self, triggering the lifespan's shutdown hooks (`await client.close()`). The Ollama daemon stays running (it's `launchd`-managed; F003 of E005 owns that), but its `OLLAMA_KEEP_ALIVE=300s` setting (also F003 of E005) means Gemma 4 E2B is unloaded from GPU memory ~5 minutes after LIP exits. Net effect: ~15 minutes after the last inference, the system is back to baseline memory; the *next* consumer that needs LIP wakes the FastAPI process (consumer-side responsibility per the disambiguated idea), which warm-loads the model again and serves traffic.

The 10-minute threshold is a balance: long enough that interactive dev sessions (think-then-act-then-think) don't trigger shutdown mid-flow; short enough that an abandoned dev session doesn't waste resources for hours. Configurable via env var; default 600 seconds (10 minutes).

*Stakeholder:* Cosmin (operator hat). *Need:* the Mac Mini's resources free up automatically when LIP isn't being used. The on-demand posture works without operator discipline.

## Scope

- **`LastRequestTracker` class** in `apps/backend/app/features/inference/service/last_request_tracker.py`. Holds a single monotonic-clock timestamp marking the last *completed* inference. Methods:
    - `record() -> None` — synchronous; sets `self._last_finish = time.monotonic()`. No lock — float writes are GIL-atomic in CPython.
    - `last_finish() -> float` — synchronous; returns the stored monotonic timestamp.
    - `__init__()` — initializes timestamp to `time.monotonic()` so the timer starts counting from app-startup, not from epoch (otherwise `time.monotonic() - 0.0` is ~1 second worth of uptime and the watchdog fires immediately).
- **Configurable `idle_shutdown_seconds` setting** added to `Settings`: `idle_shutdown_seconds: int = Field(default=600, gt=0)`. Reads from `IDLE_SHUTDOWN_SECONDS`. Default 600 (10 minutes); env-tunable.
- **Watchdog coroutine** `async def idle_watchdog(tracker, waiter_counter, idle_seconds)` in `idle_watchdog.py` (separate file per the file-layout block below; one pure function per module matches project convention). Runs forever in a loop:
    1. `await asyncio.sleep(60)` — poll once per minute. Granularity: shutdown fires within 60 s of crossing the threshold (1% over for default).
    2. `idle_for = time.monotonic() - tracker.last_finish()`. If `idle_for < idle_seconds`, `continue`.
    3. Acquire `waiter_counter.lock_held()` (E004-F002's `@asynccontextmanager`) — yields the current count while holding the counter's internal lock. Re-check `count > 0` inside the lock; if so, `continue`. Otherwise log `idle_shutdown_triggered` and send `os.kill(os.getpid(), signal.SIGTERM)` to self while still holding the lock. Return after the kill fires. Holding the lock through the kill prevents the TOCTOU race where a new request arrives between the idle check and the SIGTERM signal — `reserve()` blocks on the same lock until the watchdog releases it (which never happens, because lifespan-level shutdown unwinds the watchdog task as part of SIGTERM-driven cleanup).
- **Watchdog spawned in lifespan** *after* the warm-up completes (whether warm-up succeeded or failed; an unready service should still self-shut after idle). Spawn pattern: `watchdog_task = asyncio.create_task(idle_watchdog(tracker, waiter_counter, settings.idle_shutdown_seconds))`. The `lifespan` `finally` block cancels: `watchdog_task.cancel(); await asyncio.gather(watchdog_task, return_exceptions=True)`.
- **`InferenceService.run()` modified** — at the *end* (in a `finally` to cover both success and DomainError paths), call `self._tracker.record()`. The recording happens after the inference completes — successful or failed — meaning "10 minutes since last completion."
- **`InferenceService` constructor extended** to receive `tracker: LastRequestTracker`. Updated signature:
  `__init__(self, registry, client, semaphore, waiter_counter, max_waiters, timeout_seconds, tracker)`.
- **`get_last_request_tracker()` Depends factory** in `app/api/deps.py` with `@lru_cache(maxsize=1)` — process-wide singleton. Same pattern as semaphore / waiter counter / registry.
- **`get_inference_service()` factory updated** to also `Depends(get_last_request_tracker)`.
- **SIGTERM-to-self pattern**: `os.kill(os.getpid(), signal.SIGTERM)` is the canonical "graceful shutdown" trigger. uvicorn (FastAPI's ASGI runner) handles SIGTERM by initiating lifespan shutdown — the `lifespan` `finally` block runs, `await client.close()` closes the httpx pool, the process exits with code 0. No `os._exit`, no skipping cleanup.
- **`record()` in pre-checks too**: even fast-fail paths (registry-not-found, capability mismatch) count as "consumer is using the service." `record()` is called at the *start* of `run()` AND at the end (in `finally`), so any code path through `run()` updates the timestamp. This prevents a steady stream of bad-name requests from triggering idle shutdown.

## Out of scope

- **Wake-up wrapper script** — explicitly *out of scope* per the disambiguated idea (consumer-side responsibility).
- **Auto-start at Mac boot** — explicitly *out of scope* (contradicts G6).
- **Coordinated shutdown of Ollama** — Ollama is `launchd`-managed independently. F002 of E005 only shuts down LIP; Ollama's `OLLAMA_KEEP_ALIVE=300s` (F003 of E005) handles its own model unloading.
- **Dynamic adjustment of idle threshold based on load** — out of v1; flat 600s.
- **Different thresholds per consumer** — out of v1.
- **Pre-shutdown warning notifications** — out of v1; no Slack ping or operator email when shutdown fires.
- **Pause / extend idle timer via API** — out of v1; no `/state/keepalive` endpoint to push back the timer.
- **Tracking last request *start* (vs finish) for the timer** — F002 tracks last *completion* (or pre-check fast-fail point). Long-running inferences are captured by the `waiter_counter.current() > 0` skip, not by tracking start times.
- **Counting `/health` checks as activity** — F002 records inside `InferenceService.run()` only. Health probes don't reset the timer; they're polling, not consumption. (If consumers poll `/health` continuously while idle, LIP still shuts down — preventing pathological "keep-alive via health-check" patterns.)
- **Counting `/state` inspection as activity** — same; only inference reaches `run()`.

## Acceptance criteria

- After 600 seconds of no `/v1/inference` traffic, the process exits with code 0 (clean SIGTERM shutdown). Verified via end-to-end test that starts the app, waits 11 minutes (or 1 minute with `IDLE_SHUTDOWN_SECONDS=60`), confirms process exit + lifespan shutdown hooks fired.
- The `idle_shutdown_triggered` structured-log event is emitted with the actual `idle_for_seconds: int` value just before SIGTERM is sent.
- Lifespan shutdown hooks run on idle exit — verified by capturing `await client.close()`'s log/side-effects (the httpx client's connection pool teardown event).
- A `/v1/inference` call resets the timer: with `IDLE_SHUTDOWN_SECONDS=120`, fire one inference at t=60s, observe shutdown happens at t≥180s (60+120), not t=120s.
- A failed `/v1/inference` (registry not found) also resets the timer: with `IDLE_SHUTDOWN_SECONDS=120`, fire `model="nonexistent"` at t=60s, observe shutdown at t≥180s.
- An in-flight inference defers shutdown: with `IDLE_SHUTDOWN_SECONDS=60`, start an inference at t=10s that takes 90s (mock adapter); the watchdog sees `waiter_counter.current()=1` at t=70s and skips; shutdown fires only after the inference finishes + 60s of true idle. (Total wall-clock ~160s+.)
- `IDLE_SHUTDOWN_SECONDS=0` rejected at startup (`gt=0`).
- The watchdog's polling interval is 60 seconds — verified by observing exactly one `await asyncio.sleep(60)` per loop iteration in a test that mocks `asyncio.sleep` to count calls.
- The watchdog's idle check + SIGTERM is atomic under `waiter_counter.lock_held()`: a unit test that holds `lock_held()` while the watchdog has crossed the idle threshold blocks the watchdog at the lock acquisition; releasing the held lock allows the watchdog to proceed. A concurrent `reserve()` issued between the idle check and the kill blocks until the kill fires (request never serviced — uvicorn graceful shutdown handles cleanup) or the watchdog observes `count > 0` inside the lock and skips this round.
- `last_request_tracker.last_finish()` is initialized to `time.monotonic()` at construction time (not 0.0 / not epoch); shutdown does not fire prematurely after app startup.
- `last_request_tracker.last_finish()` updates on inference attempt — verified that calling `service.run(...)` at t=0 makes `tracker.last_finish() ≈ time.monotonic()` immediately afterward.
- Watchdog is cancelled cleanly in lifespan `finally` — verified that the `asyncio.CancelledError` propagates from `watchdog_task.cancel()` and `await asyncio.gather(watchdog_task, return_exceptions=True)` returns without raising.
- Pyright strict passes; `task check` passes.
- The `LastRequestTracker` class lives in its own file (`last_request_tracker.py`) — one class per file rule.

**`LastRequestTracker` implementation:**

```python
import time


class LastRequestTracker:
    def __init__(self) -> None:
        self._last_finish: float = time.monotonic()

    def record(self) -> None:
        self._last_finish = time.monotonic()

    def last_finish(self) -> float:
        return self._last_finish
```

No lock. CPython float assignments are atomic under the GIL. Even if a `record()` and `last_finish()` race, the read sees either the old value or the new — never a torn value.

**`idle_watchdog` implementation:**

```python
import asyncio
import os
import signal
import time

import structlog

from app.features.inference.service.last_request_tracker import LastRequestTracker
from app.features.inference.service.waiter_counter import WaiterCounter

logger = structlog.get_logger(__name__)


async def idle_watchdog(
    tracker: LastRequestTracker,
    waiter_counter: WaiterCounter,
    idle_seconds: int,
) -> None:
    """Poll once per minute; SIGTERM the process when idle exceeds the threshold.

    The check-and-kill is performed atomically under WaiterCounter's lock so
    a request arriving between the idle check and the kill is blocked at
    reserve() until the watchdog releases the lock — which it never does on
    the kill path, because lifespan unwinds the watchdog task during the
    SIGTERM-driven shutdown.
    """
    while True:
        await asyncio.sleep(60)
        idle_for = time.monotonic() - tracker.last_finish()
        if idle_for < idle_seconds:
            continue
        async with waiter_counter.lock_held() as count:
            if count > 0:
                continue
            logger.info(
                "idle_shutdown_triggered",
                idle_for_seconds=int(idle_for),
                threshold_seconds=idle_seconds,
            )
            os.kill(os.getpid(), signal.SIGTERM)
            return
```

**`InferenceService.run()` modified body** (post-F002-of-E005, full final form):

```python
async def run(
    self,
    envelope: InferenceRequest,
    request_id: str,
) -> InferenceResponse:
    self._tracker.record()  # idle reset on entry — covers fast-fail paths too
    start = time.perf_counter()
    try:
        model_info = self._registry.get(envelope.model)
        check_envelope_capabilities(envelope, model_info)
        merged = merge_sampling_params(envelope.params, model_info.sampling_defaults)

        await self._waiter_counter.reserve(self._max_waiters)
        try:
            wait_start = time.perf_counter()
            async with self._semaphore:
                queue_wait_ms = int((time.perf_counter() - wait_start) * 1000)
                try:
                    result = await asyncio.wait_for(
                        self._client.chat(
                            model_tag=model_info.backend_tag,
                            messages=envelope.messages,
                            params=merged,
                        ),
                        timeout=self._timeout_seconds,
                    )
                except asyncio.TimeoutError as exc:
                    raise InferenceTimeoutError(
                        timeout_seconds=self._timeout_seconds,
                    ) from exc
        finally:
            await self._waiter_counter.release()

        latency_ms = int((time.perf_counter() - start) * 1000)
        return InferenceResponse(
            content=result.content,
            metadata=ResponseMetadata(
                model=envelope.model,
                prompt_tokens=result.prompt_tokens,
                completion_tokens=result.completion_tokens,
                request_id=request_id,
                latency_ms=latency_ms,
                queue_wait_ms=queue_wait_ms,
                finish_reason=result.finish_reason,
                backend="ollama",
            ),
        )
    finally:
        self._tracker.record()  # idle reset on exit — covers DomainError paths
```

**Why record at *both* entry and `finally`:** entry-record ensures fast-fail paths (registry-not-found, capability mismatch) reset the timer — the consumer is *trying* to use the service, that's activity. The `finally`-record covers slow inferences whose entry-record was minutes ago by the time they finish. The two cover all paths through `run()` cleanly.

**Lifespan spawning the watchdog:**

```python
# app/main.py — lifespan post-F002-of-E005

@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncIterator[None]:
    settings = get_settings()
    client = OllamaClient(base_url=settings.ollama_host)
    app.state.ollama_client = client
    app.state.is_ready = False

    registry = ModelRegistry(DEFAULT_MODELS)
    tracker = get_last_request_tracker()
    waiter_counter = get_waiter_counter()

    start = time.perf_counter()
    try:
        await warm_up_inference(
            client, registry, timeout_seconds=settings.warmup_timeout_seconds
        )
        duration_ms = int((time.perf_counter() - start) * 1000)
        app.state.is_ready = True
        logger.info("warmup_complete", duration_ms=duration_ms, logical_name="default-task")
    except Exception:
        logger.exception("warmup_failed", logical_name="default-task")

    watchdog_task = asyncio.create_task(
        idle_watchdog(tracker, waiter_counter, settings.idle_shutdown_seconds)
    )

    try:
        yield
    finally:
        watchdog_task.cancel()
        await asyncio.gather(watchdog_task, return_exceptions=True)
        await client.close()
```

**Why SIGTERM not `os._exit`:** `os._exit(0)` skips Python's atexit hooks, asyncio loop cleanup, and FastAPI's lifespan shutdown — meaning `await client.close()` doesn't run. This leaves the httpx connection pool with sockets in `CLOSE_WAIT` state and Ollama wondering why the connection went away. SIGTERM-to-self is documented in uvicorn's docs as the canonical "graceful self-shutdown" pattern; uvicorn's signal handler catches it, asks the FastAPI app to shut down via lifespan, runs all cleanup, exits with code 0.

**Why a separate `LastRequestTracker` class instead of folding into `WaiterCounter`:** they have different semantics (counter tracks active count; tracker tracks last-touch timestamp) and different atomicity needs (counter mutations need a lock for the increment-check pair; tracker writes are atomic GIL-protected float assignments). Combining them would force the lock onto the timer reads, defeating the no-contention property. One class per concern; per CLAUDE.md.

**Watchdog cleanup discipline**: `asyncio.create_task(...)` returns a Task that runs in background. On lifespan shutdown, `task.cancel()` sends `CancelledError` into the task; the `await asyncio.gather(..., return_exceptions=True)` awaits the cancellation completing without re-raising. This is the canonical asyncio pattern for "fire-and-forget background task with clean teardown."

**Polling interval (60s) hardcoded vs configurable:** the threshold (600s) is configurable; the poll interval (60s) is hardcoded at 1/10 of the default threshold. Making it configurable adds two env vars where one suffices; the 1-minute granularity is plenty for a 10-minute threshold. If someone sets `IDLE_SHUTDOWN_SECONDS=30`, the 60s poll means shutdown fires within 60+30=90s instead of 30s — degraded precision but still functional. Acceptable.

**Pyright strict typing**: `asyncio.create_task`, `os.kill`, `signal.SIGTERM` are all typed in stdlib; no `# type: ignore` needed. `asyncio.gather(..., return_exceptions=True)` return type is `list[Any | BaseException]`; the result is unused so no narrowing required.
