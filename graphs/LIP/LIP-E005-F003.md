## Short description

A `launchd` plist file configuring the always-on Ollama daemon with `OLLAMA_KEEP_ALIVE=300s`, `OLLAMA_NUM_PARALLEL=1`, `OLLAMA_MAX_LOADED_MODELS=1`, `OLLAMA_FLASH_ATTENTION=1`, and `OLLAMA_KV_CACHE_TYPE=q8_0`, plus its installation procedure.

## Problem

Ollama, when launched from the Ollama.app GUI, runs as a foreground app — closing the dock icon stops the daemon, model is unloaded, and the next LIP request that reaches the adapter sees `connection_refused`. Manually running `ollama serve` from a terminal works but requires that terminal to stay open. Neither posture matches LIP's "always-on Ollama, on-demand FastAPI" lifecycle (G6).

F003 ships the canonical `launchd` plist that runs Ollama as a user-level always-on agent, configured with the five env vars that calibrate Ollama for LIP's workload (single-model, single-stream, KV-cache-quantized for the 16 GB Mini's memory envelope). The plist gets installed to `~/Library/LaunchAgents/`, bootstrapped via `launchctl`, and from then on Ollama starts at user login and survives terminal closures, GUI app exits, and idle sessions. LIP's FastAPI side stays on-demand; Ollama is the always-on substrate (with its own short `KEEP_ALIVE=300s` model-unload timer to free RAM during quiet periods).

The five env vars aren't optional. `OLLAMA_KEEP_ALIVE=300s` sets the model-unload window to 5 minutes (matches the 10-minute LIP idle shutdown — by the time LIP exits and a fresh wake-up arrives, Ollama has unloaded the model and reloads it on demand); this also happens to be Ollama's current upstream default, so the explicit pin doubles as defense-in-depth against upstream default drift. `OLLAMA_NUM_PARALLEL=1` pins Ollama's parallel-request slots to 1 — defense in depth for LIP's `asyncio.Semaphore(1)`, so even if the service-layer serialization breaks (or upstream raises the default) Ollama still won't run two streams concurrently. `OLLAMA_MAX_LOADED_MODELS=1` enforces the 16 GB memory envelope (Ollama's default `0` auto-derives from VRAM and may pick 2–3 on a 16 GB Mini and OOM; pinning to `1` is meaningful). `OLLAMA_FLASH_ATTENTION=1` and `OLLAMA_KV_CACHE_TYPE=q8_0` cut KV-cache memory for the 128K-context model — without them, Gemma 4 E2B's full context burns RAM that the Mini doesn't have.

*Stakeholder:* Cosmin (operator hat). *Need:* Ollama runs reliably without operator intervention, with the right env config baked in. Reproducible from the repo via `task ollama:install`.

## Scope

- **Plist file** at `infra/launchd/com.lip.ollama.plist` — XML plist following Apple's `launchd.plist(5)` schema. Contents:
    - `Label`: `com.lip.ollama` — globally unique reverse-DNS-style identifier; used as the launchd agent ID.
    - `ProgramArguments`: `["/opt/homebrew/bin/ollama", "serve"]` — Apple-Silicon Homebrew default. Documented as customizable for non-Homebrew installs (e.g., `/usr/local/bin/ollama` on Intel Macs, `~/bin/ollama` for manual installs).
    - `EnvironmentVariables`: dict containing the five canonical env vars verbatim (`OLLAMA_KEEP_ALIVE=300s`, `OLLAMA_NUM_PARALLEL=1`, `OLLAMA_MAX_LOADED_MODELS=1`, `OLLAMA_FLASH_ATTENTION=1`, `OLLAMA_KV_CACHE_TYPE=q8_0`).
    - `RunAtLoad`: `true` — start when the agent is loaded (i.e., at user login after install).
    - `KeepAlive`: `true` — restart if Ollama exits unexpectedly.
    - `StandardOutPath`: `/Users/cosminneamtiu/Library/Logs/ollama/stdout.log` — Ollama's stdout. Documented as user-customizable; the path is hardcoded to Cosmin's machine since the plist is part of *this* repo for *this* deployment, not a generic distribution.
    - `StandardErrorPath`: `/Users/cosminneamtiu/Library/Logs/ollama/stderr.log` — same logic.
    - `ProcessType`: `Background` — launchd hint; reduces priority of the daemon vs interactive apps.
- **Documentation file** at `docs/ollama-launchd.md` — a short markdown doc explaining:
    - What the plist does and which env vars it sets.
    - Why those specific env-var values (mapped to G5 / G6 / 16 GB Mini constraints).
    - Install procedure: `task ollama:install` (which copies the plist to `~/Library/LaunchAgents/`, ensures `~/Library/Logs/ollama/` exists, then runs `launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.lip.ollama.plist`).
    - Uninstall procedure: `task ollama:uninstall` (`launchctl bootout`, then remove the file).
    - Status check: `task ollama:status` (`launchctl print gui/$(id -u)/com.lip.ollama`).
    - How to customize for non-Homebrew Ollama installs (replace `/opt/homebrew/bin/ollama` in the plist).
    - How to verify the env vars made it through (`launchctl print gui/$(id -u)/com.lip.ollama` shows the resolved environment).
- **Taskfile entries** for the three commands above. The repo's existing `Taskfile.yml` gets a new `ollama:` namespace.
    - `task ollama:install` — `mkdir -p ~/Library/Logs/ollama; mkdir -p ~/Library/LaunchAgents; cp infra/launchd/com.lip.ollama.plist ~/Library/LaunchAgents/com.lip.ollama.plist; launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.lip.ollama.plist`. Idempotency: if already loaded, `bootstrap` errors; document this behavior in the doc.
    - `task ollama:uninstall` — `launchctl bootout gui/$(id -u)/com.lip.ollama; rm -f ~/Library/LaunchAgents/com.lip.ollama.plist`.
    - `task ollama:status` — `launchctl print gui/$(id -u)/com.lip.ollama`.
- **Plist validation in CI** — `task check` extended to run `plutil -lint infra/launchd/com.lip.ollama.plist` against the source-of-truth plist. Catches a broken plist before the operator copies it to `~/Library/LaunchAgents/` and `launchctl` complains.
- **Pre-existing Ollama installation assumed.** F003 doesn't ship Ollama; it ships only the launchd config. The doc references `brew install ollama` as the canonical way to get the binary; that's the operator's prerequisite.

## Out of scope

- **Ollama installation itself** — `brew install ollama` is the operator's prereq. F003 doesn't ship the binary or shell scripts to fetch it.
- **macOS-side health-checks** — `task ollama:status` reports launchd state, not whether `/api/tags` returns 200. A separate Ollama liveness check is out of v1; LIP's `/health` (E006-F001) reports its own readiness, which implicitly depends on Ollama-up.
- **Auto-install at repo clone** — the install procedure is *manual* and *one-time*. F003 ships the plist + commands; the operator runs `task ollama:install` once after cloning and once after any plist edit.
- **Linux / Windows equivalents** — out of v1 scope per the macOS-only deployment-target constraint.
- **Cross-developer-portability** — the plist's `StandardOutPath`/`StandardErrorPath` hardcode `/Users/cosminneamtiu/Library/Logs/ollama/`. This is the *repo's* config for *this* deployment. If a second operator ever uses LIP, they edit those two paths. Not a v1 deliverable to make this generic.
- **Templating the plist via Jinja or env-substitution** — overkill for a config that gets edited once. Hardcoded paths plus a short doc-section explaining what to swap for non-default setups.
- **Per-environment plists** (dev / prod) — there's only one environment (the Mac Mini).
- **Backup / rollback of plist edits** — operator's responsibility via git.
- **Coordinating Ollama upgrades** — `brew upgrade ollama` doesn't touch the plist; `launchctl kickstart -k gui/$(id -u)/com.lip.ollama` restarts the agent if needed. Documented in the doc but not automated.
- **Sentry / Datadog / log forwarding** — out of v1; logs land in `~/Library/Logs/ollama/`, operator tails them when needed.
- **Plist signing or notarization** — not required for `~/Library/LaunchAgents/` user agents.
- **Modifying Ollama's data dir** (default `~/.ollama/`) — fine as-is; not configured here.
- **Enforceable per-test memory-footprint assertions:** idle <200 MB and active ~7.5 GB with the Gemma 4 E2B model loaded are observed targets from the M4 Mac Mini calibration (`docs/disambigued-idea.md:142-144`), not gated via tests in v1. The five env vars (`OLLAMA_KEEP_ALIVE=300s`, `OLLAMA_NUM_PARALLEL=1`, `OLLAMA_MAX_LOADED_MODELS=1`, `OLLAMA_FLASH_ATTENTION=1`, `OLLAMA_KV_CACHE_TYPE=q8_0`) are calibrated to produce these footprints, but the numbers themselves are calibration data — operator-observed via Activity Monitor / `ps`, not pytest-asserted. Adding RSS-based test assertions is hostile in CI (no GPU, no Ollama) and noisy on the dev Mini (other apps consume RAM). If memory-regression detection becomes valuable, a future feature can add an operator-runnable benchmark task; v1 stays out of this scope per the disambiguation framing.

**Taskfile entries (sketch):**

```yaml
ollama:install:
  desc: "Install the Ollama launchd agent (one-time setup)."
  cmds:
    - mkdir -p ~/Library/Logs/ollama
    - mkdir -p ~/Library/LaunchAgents
    - cp infra/launchd/com.lip.ollama.plist ~/Library/LaunchAgents/com.lip.ollama.plist
    - launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.lip.ollama.plist

ollama:uninstall:
  desc: "Remove the Ollama launchd agent."
  cmds:
    - launchctl bootout gui/$(id -u)/com.lip.ollama
    - rm -f ~/Library/LaunchAgents/com.lip.ollama.plist

ollama:status:
  desc: "Report Ollama launchd agent status."
  cmds:
    - launchctl print gui/$(id -u)/com.lip.ollama

check:plist:
  desc: "Validate launchd plist schema."
  cmds:
    - plutil -lint infra/launchd/com.lip.ollama.plist
```

The existing `task check` aggregate target gets `check:plist` added to its dependency list.

**Why `launchctl bootstrap` over `launchctl load`:** Apple deprecated `load`/`unload` in macOS 10.10+ in favor of the domain-aware `bootstrap`/`bootout` commands. `bootstrap` produces clearer error messages, supports session-aware loading (`gui/<uid>` for user-session agents), and is the documented modern command. `load` still works but isn't the recommended target.

**Why `gui/$(id -u)`:** the `gui` domain ties the agent to the logged-in user's GUI session — exactly the right scope for "starts when Cosmin logs in." Alternatives (`user/<uid>`, `system`) either don't tie to login session or require root. The `gui` domain matches the always-on-during-user-session intent.

**Why `KeepAlive=true`** (not just `RunAtLoad=true`): if Ollama crashes mid-session, launchd auto-restarts it. F003 of E001 (and the rest of the QoS layer) treats Ollama-down as `AdapterConnectionFailureError`; auto-restart means subsequent requests succeed without operator intervention. `KeepAlive` does have a backoff: launchd won't auto-restart faster than every ~10 seconds if Ollama keeps crashing immediately. Acceptable.

**Why `ProcessType=Background`:** launchd's `ProcessType` informs the OS scheduler. `Background` lowers Ollama's priority versus interactive apps — when Cosmin is using the Mini for desktop work, Ollama yields CPU naturally. Subtle but useful for the "Mini stays usable for other work" goal (G6).

**Why hardcode `/Users/cosminneamtiu/...` paths:** this is *the* deployment plist for *this* repo's *this* Mac Mini. Templating adds complexity without value for a single-user, single-machine deployment. If a second operator ever uses LIP, they edit two lines in the plist. Documented in the doc.

**Why `plutil -lint` in `task check`:** the plist is shipped as repo artifact. A typo (missing closing tag, wrong key type) would be discovered only at `launchctl bootstrap` time — late, post-CI, in an interactive operator session. Catching at CI is one `plutil -lint` call; the validator is a stock macOS utility, no extra dependency. Adding it costs nothing and protects against breakage.

**Why no plist signing:** `~/Library/LaunchAgents/` doesn't require signed plists for user-scope agents. macOS Gatekeeper / notarization apply to apps and binaries, not to launchd config files. (System-scope agents in `/Library/LaunchDaemons/` get more scrutiny; we're not using that scope.)

**Idempotency note for `task ollama:install`:** running `bootstrap` against an already-loaded service errors with "Service already loaded." The doc explains that `task ollama:uninstall && task ollama:install` is the idempotent reload pattern. Or `launchctl kickstart -k gui/$(id -u)/com.lip.ollama` to restart in place after editing the plist.
