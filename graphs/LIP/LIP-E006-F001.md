## Short description

Liveness returns 200 if the process is alive; readiness returns 200 only after the warm-up dummy inference has completed, otherwise 503 — letting consumers wait until the service is truly ready to serve real inference.

## Problem

Consumer projects launching LIP need a deterministic signal for "the service is reachable AND ready to serve real inference." Without F001, a consumer that fires `POST /v1/inference` immediately after starting LIP gets either (a) a connection-refused error if Ollama hasn't been probed yet, (b) a multi-second cold-load latency on the first call, or (c) an `AdapterConnectionFailureError` if Ollama happens to be down at that moment. None of these are programmable in a clean retry loop.

F001 ships a single `GET /health` endpoint that returns three observable states:

- **Connection refused / timeout** → process is down. Consumer's wake-up wrapper restarts it.
- **HTTP 200** → process is up AND warm-up completed. Inference traffic safe to send.
- **HTTP 503** → process is up but warm-up hasn't completed (still loading the model, or warm-up failed and operator hasn't intervened). Consumer waits.

The HTTP status alone drives the consumer's poll loop; the JSON body adds context (`is_ready: bool`, `reason: str | None`) for human readers and structured-log scrapers. No separate `/health/live` and `/health/ready` paths — the single endpoint conveys both states cleanly. (Process-up is implicit in "got an HTTP response at all"; the 200/503 distinction conveys readiness.)

*Stakeholder:* Cosmin (consumer-side developer + operator hats). *Need:* a deterministic wake-up signal that consumer projects can program against, plus a quick health-check that a human can curl mid-session to triage "is the service alive."

## Scope

- **`GET /health` endpoint** at the root path (no `/v1` prefix — health is operational, not versioned API). Implementation in `apps/backend/app/api/health_router.py` — modifies the existing router from the post-bootstrap template.
- **Two response shapes**, both validated by a single Pydantic model `HealthResponse`:

    ```python
    class HealthResponse(BaseModel):
        status: Literal["ready", "not_ready"]
        is_ready: bool
        reason: str | None = None
    ```
- **HTTP status mapping:**
    - `200` + `HealthResponse(status="ready", is_ready=True, reason=None)` when `app.state.is_ready == True`.
    - `503` + `HealthResponse(status="not_ready", is_ready=False, reason="warmup_pending")` when `app.state.is_ready == False`.
- **Body uses plain JSON (`application/json`), not RFC 7807.** F004-of-E004's `application/problem+json` is for application errors raised via the `DomainError` hierarchy; `/health`'s 503 is a *status signal*, not an error. The body is a `HealthResponse` Pydantic model serialized as plain JSON. The `/health` route returns a `fastapi.responses.JSONResponse` directly with the chosen status code, bypassing the global RFC 7807 exception handler entirely.
- **No Ollama probe.** F001 reads `app.state.is_ready` (set by F001-of-E005's lifespan warm-up) and returns the answer. No network call to Ollama; no chance of `/health` itself blocking on Ollama latency. Probing Ollama for "is the model loaded?" is `LIP-E006-F002`'s concern.
- **`async def health(request)` handler** with one statement of branching logic (read `app.state.is_ready`, return one of two `JSONResponse` constructions). Per CLAUDE.md, business logic in handlers is forbidden, but reading-and-mapping a single bool is not business logic — it's the entire purpose of the route.
- **Does NOT call `LastRequestTracker.record()`.** F001-of-E006 explicitly does not tick the idle-timer (F002 of E005). Consumer poll-loops on `/health` would otherwise keep LIP "active" indefinitely, defeating idle shutdown.
- **OpenAPI integration**: `@router.get("/health", responses={200: {"model": HealthResponse}, 503: {"model": HealthResponse}})`. Both response shapes documented in `/openapi.json` so consumer-side codegen sees the contract. The handler does not declare `response_model=HealthResponse` (which would only document the 200 shape); the explicit `responses` dict covers both.

## Out of scope

- **Separate `/health/live` and `/health/ready` paths** (kubelet convention) — Epic Q2 resolved: single `/health` with HTTP-status-as-readiness-signal is simpler and equally expressive for v1's local-network single-machine deployment.
- **Ollama probe** — `LIP-E006-F002` (state inspection) covers "is the model loaded right now"; F001 just reflects the warm-up flag from `app.state.is_ready`.
- **Authentication on `/health`** — out of v1 (local-network trust).
- **Returning structured timing data** (uptime, last-warmup duration) — out of v1; F002 of E006 surfaces operational state.
- **Configurable readiness criteria** — `app.state.is_ready` is the single criterion in v1. Future criteria (Ollama health, queue health, recent error rate) become refactor work; v1 keeps the unary signal.
- **HEAD method support** — FastAPI generates HEAD automatically for GET routes, but the body is what carries `reason`. Curl `-I` returns 200/503 with no body — useful for quick "is it up" checks. No special handling needed; FastAPI's default is correct.
- **Caching the response** — the answer is a single bool read; caching is irrelevant.
- **Prometheus / OpenTelemetry exposition** — out of v1 scope per Project Boundary.
- **Web-UI / HTML rendering** — out of v1; structured JSON only.
- **Active liveness checks** (e.g., crashing the process if /health hasn't been called in N seconds) — out of v1; explicit no-auto-recovery stance.
- **`/health` resetting the idle-shutdown timer** — out of v1, by design. Health probes are passive observation, not active use.
- **Per-consumer rate limiting on `/health`** — out of v1.

## Acceptance criteria

- After clean startup with successful warm-up, `GET /health` returns HTTP 200, `Content-Type: application/json`, body `{"status": "ready", "is_ready": true, "reason": null}`.
- During the warm-up window (after process bind but before warm-up completes), `GET /health` returns HTTP 503, body `{"status": "not_ready", "is_ready": false, "reason": "warmup_pending"}`.
- After a failed warm-up (Ollama down at startup), `GET /health` returns HTTP 503 with the same body — the lifespan caught the warmup_failed exception but `app.state.is_ready` stayed False.
- `GET /health` responds in under 50 ms wall-clock (no Ollama probe; just an in-process bool read).
- `GET /health` is `async def`; the handler is a single bool read + a branched `JSONResponse` construction (no business logic, no try/except, no service-layer call).
- `/openapi.json` documents `GET /health` with `responses` containing both `200` and `503` referencing `HealthResponse` schema.
- The 503 response body is plain JSON (`application/json`), NOT RFC 7807 (`application/problem+json`). Verified by inspecting `Content-Type` header.
- `GET /health` does not call `LastRequestTracker.record()`; verified by issuing 100 `/health` requests during a quiet window and observing `tracker.last_finish()` does not advance.
- `GET /health` does not require any of the inference-feature DI (registry, semaphore, waiter counter, OllamaClient). Verified by depending only on the FastAPI `Request` for `app.state` access.
- A consumer's wake-up loop pattern works: launch LIP; poll `GET /health` every 1s; first 200 received indicates the service is ready for inference. End-to-end test verifies the loop completes within ~10s on a clean Ollama.
- The existing post-bootstrap `/health` endpoint's behavior is *replaced* — no longer returns the bootstrap's `{"status": "ok"}` shape; the new `HealthResponse` shape is the single source of truth.
- Pyright strict passes; `task check` passes.

## Technical constraints

**File layout** (modifies existing post-bootstrap files):

```
apps/backend/app/api/
├── health_router.py            # MODIFY: replace bootstrap stub with readiness-aware /health
└── health_response.py          # NEW: HealthResponse Pydantic model
```

**`health_response.py`:**

```python
from typing import Literal

from pydantic import BaseModel, ConfigDict


class HealthResponse(BaseModel):
    model_config = ConfigDict(extra="forbid")

    status: Literal["ready", "not_ready"]
    is_ready: bool
    reason: str | None = None
```

**`health_router.py`:**

```python
from fastapi import APIRouter, Request
from fastapi.responses import JSONResponse

from app.api.health_response import HealthResponse

router = APIRouter()


@router.get(
    "/health",
    responses={
        200: {"model": HealthResponse},
        503: {"model": HealthResponse},
    },
)
async def health(request: Request) -> JSONResponse:
    is_ready = bool(getattr(request.app.state, "is_ready", False))
    if is_ready:
        body = HealthResponse(status="ready", is_ready=True, reason=None)
        return JSONResponse(status_code=200, content=body.model_dump())
    body = HealthResponse(
        status="not_ready",
        is_ready=False,
        reason="warmup_pending",
    )
    return JSONResponse(status_code=503, content=body.model_dump())
```

**Why `JSONResponse(status_code=...)` instead of `response_model=HealthResponse`:** FastAPI's `response_model=` works for the success case but doesn't drive non-200 status. The dual-response shape requires explicit `JSONResponse` construction with the chosen `status_code`. The `responses=` dict on the decorator handles OpenAPI documentation; `response_model=` would conflict by claiming a single shape.

**Why `getattr(request.app.state, "is_ready", False)` instead of direct attribute access:** during the brief moment between FastAPI accepting connections and lifespan running its first statement, `app.state.is_ready` may not yet be set. A defensive `getattr(..., default=False)` returns "not ready" for any pre-init state — strictly correct (the warm-up hasn't run yet), no `AttributeError`. Once the lifespan's `is_ready = False` line runs, the attribute is always present; `getattr` becomes redundant but harmless.

**Why no `Depends()` injection:** `/health` deliberately depends on no inference-feature DI. Reading `app.state.is_ready` directly via `Request` makes `/health` answerable even if the registry, semaphore, or OllamaClient construction failed — the operator can still curl `/health` and observe "503 not_ready" rather than getting a 500 from a broken DI graph. Robustness over purity here.

**Why HTTP 503 specifically:** RFC 7231 §6.6.4 — Service Unavailable — semantically matches "process is up but cannot currently serve the workload." 503 is also the standard kubelet liveness/readiness signal, so consumer libraries treat 503-from-/health as a "wait and retry" trigger out-of-the-box. 425 (Too Early) is closer in spirit but less widely understood; 503 wins on convention.

**`HealthResponse` as a shared schema candidate**: in v1 it's used in exactly one place (the `/health` route). Putting it in `app/api/health_response.py` keeps it physically close to the route and respects "one class per file." If a future feature returns `HealthResponse` (state-inspection might), the file stays where it is and the import is shared.

**Pyright strict on `app.state` typing:** Starlette types `app.state` as `State`, an `Any`-bag. Reading via `getattr(..., default=False)` returns `Any`, then `bool(...)` narrows to `bool`. Pyright accepts this without `# type: ignore`. (If we wanted stricter typing, a `cast(bool, ...)` would also work, but `bool(getattr(...))` reads cleaner and provides a runtime-safe fallback.)

**OpenAPI documentation:** the `responses` dict ensures `/openapi.json` lists both 200 and 503 with their shapes. Consumer-side OpenAPI codegen (if any consumer ever uses it) generates a discriminated-union return type or treats them as alternative response codes; either way, the wire contract is documented.

**Bootstrap legacy:** the post-bootstrap `health_router.py` returned `{"status": "ok"}`. F001 *replaces* that. No backward-compatibility concern — no consumer is yet built against the bootstrap stub.

## Open questions

None. All decisions resolved within thickening: single `/health` path with HTTP-status-as-readiness signal (Epic Q2 resolved — kubelet's separate-paths convention is over-engineered for a personal local-network service); plain JSON body, not RFC 7807 (status signals aren't application errors); 503 for not-ready (RFC 7231 + kubelet convention); no Ollama probe (Epic Q3 deferred to F002 of E006); no idle-timer tick (preserves F002-of-E005's "10 min since last *inference*" semantics); defensive `getattr` for app.state read; module-level `HealthResponse` Pydantic schema in `app/api/`.

## Unit test scenarios

- `HealthResponse(status="ready", is_ready=True, reason=None)` constructs without error.
- `HealthResponse(status="not_ready", is_ready=False, reason="warmup_pending")` constructs without error.
