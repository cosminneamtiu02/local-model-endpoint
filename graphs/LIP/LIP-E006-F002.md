## Short description

Structured-JSON endpoints exposing live operational state — current semaphore queue depth, whether Ollama has the model loaded, and the timestamp of the most recent inference — without log parsing.

## Problem

Mid-session operator questions ("is LIP busy?", "did Ollama unload the model already?", "when was the last inference?") have no answer in v1 without F002 — `/health` (F001 of E006) reports only the unary `is_ready` flag, and the only other surface is structlog output (which requires tail+grep). Without F002, triage during an unexplained latency or a hung consumer is "ssh in, run `ps`, run `lsof`, tail logs" — operator-hostile and slow.

F002 ships a single consolidated `GET /state` endpoint that returns all live operational state as structured JSON in one round-trip:

- **`queue_depth: int`** — current waiter count from the F002-of-E004 atomic counter (how many requests are queued or in flight).
- **`max_waiters: int`** — the configured limit (echoes the `LIP_MAX_INFERENCE_WAITERS` env var) so the operator sees both axes of the bound.
- **`last_request_seconds_ago: int`** — wall-clock seconds since the most recent activity touching `InferenceService.run()` (computed from F002-of-E005's `LastRequestTracker.last_finish()`).
- **`ollama_reachable: bool`** — whether a probe to Ollama's `/api/ps` succeeded.
- **`ollama_loaded_models: list[str]`** — backend tags currently in Ollama's memory (from `/api/ps`). Empty list if Ollama is up but has unloaded the model (post-`KEEP_ALIVE` window).
- **`is_ready: bool`** — same flag as `/health` for consistency; operator gets the full picture from one call.

The endpoint is best-effort: if Ollama is unreachable, `/state` still returns 200 with `ollama_reachable: false` + `ollama_loaded_models: []` rather than 5xx. The endpoint's purpose is to *report* state including degraded states; failing because the state is degraded would be perverse.

*Stakeholder:* Cosmin (operator hat). *Need:* mid-session operational truth in one curl, without log parsing or process inspection. The "is it stuck or just busy" decision becomes a 50 ms `/state` round-trip instead of a multi-step diagnostic.

## Scope

- **Single consolidated `GET /state` endpoint** at the root path (no `/v1` prefix — operational, not contract API). Returns the full state in one structured-JSON response. Epic Q1 resolved: one endpoint, not multiple `/state/queue` + `/state/model` paths. Reasons: single round-trip is operator-friendly; ≤4-consumer load doesn't benefit from selective fetching; one OpenAPI entry vs three.
- **`StateResponse` Pydantic model** (in `apps/backend/app/api/state_response.py` — one class per file):

    ```python
    class StateResponse(BaseModel):
        model_config = ConfigDict(extra="forbid")

        is_ready: bool
        queue_depth: int
        max_waiters: int
        last_request_seconds_ago: int
        ollama_reachable: bool
        ollama_loaded_models: list[str]
    ```
- **`OllamaClient.list_loaded_models() -> list[str]` async method** added to the existing `OllamaClient` class (F001 of E003). Calls `GET /api/ps`, parses the response, returns the `name` field from each entry in `models[]`. Wraps the call in the same try/except → `AdapterConnectionFailureError` translation as F003-of-E003 (consistency with F002-of-E003's `chat()` failure handling). Raises `AdapterConnectionFailureError` on failure; the `/state` route catches and translates to `ollama_reachable: false`.
- **`/state` route handler** in `apps/backend/app/api/state_router.py` (new file). Wires:
    - `Depends(get_waiter_counter)` for `queue_depth` (`waiter_counter.current()`).
    - `Depends(get_settings)` for `max_waiters` (`settings.max_inference_waiters`).
    - `Depends(get_last_request_tracker)` for `last_request_seconds_ago` (`int(time.monotonic() - tracker.last_finish())`).
    - `Depends(get_ollama_client)` for the Ollama probe.
    - `Request` for `app.state.is_ready`.
- **Best-effort Ollama probe**: the route's body wraps `await client.list_loaded_models()` in `try/except AdapterConnectionFailureError`. On exception → `ollama_reachable=False, ollama_loaded_models=[]`. On success → `ollama_reachable=True, ollama_loaded_models=<probe-result>`.
- **Always returns HTTP 200**: `/state` reports state, including degraded states. A 5xx from the state endpoint itself defeats its operational purpose. The endpoint never raises a `DomainError`.
- **No TTL caching of the Ollama probe.** Every `/state` call probes `/api/ps`. Ollama's `/api/ps` is sub-100ms locally; ≤4 consumers + occasional operator curls produce nowhere near enough load to warrant caching. Epic Q3 resolved.
- **No idle-timer tick.** Like `/health`, `/state` doesn't call `tracker.record()` — operator inspections must not keep LIP awake. (F002 of E005's record-on-entry-and-exit happens only inside `InferenceService.run()`.)
- **Router registration** in `app/main.py`'s `create_app` factory: `app.include_router(state_router)`.

## Out of scope

- **Multiple endpoints (`/state/queue`, `/state/ollama`, `/state/last-request`)** — Epic Q1 resolved: single consolidated endpoint.
- **TTL caching of the Ollama probe** — Epic Q3 resolved: no cache in v1.
- **Authentication on `/state`** — out of v1 (local-network trust).
- **Metrics output (Prometheus, OpenMetrics)** — out of v1 per Project Boundary.
- **Historical state (last N requests, queue-depth-over-time)** — out of v1; current state only.
- **Per-model state when v1 has only one model** — `ollama_loaded_models` is a list anticipating future multi-model deployments; v1 always returns 0 or 1 entry.
- **State writes / control endpoints** — `/state` is read-only. No `PUT /state`, no `DELETE /state/queue`, no admin operations.
- **Streaming of state changes** — out of v1; no WebSocket / SSE.
- **Active health checks of Ollama beyond `/api/ps`** — `/api/version`, `/api/embeddings` round-trip, GPU memory probes are out of v1.
- **Reporting of recent error rate or latency percentiles** — out of v1; structured logs are the source for that, in a future structured-log milestone.
- **Alerting / paging on degraded state** — out of v1; operator polls manually.
- **Graceful exposure of the registry's full contents** — `ollama_loaded_models` shows what Ollama has, not what's registered. If exposing registry contents is needed, a future `/state/registry` endpoint can be added; v1 keeps `/state` minimal.

## Acceptance criteria

- `GET /state` returns HTTP 200 with `Content-Type: application/json` and a body matching `StateResponse`.
- Steady-state response (no in-flight inference, model loaded, post-warmup): `{"is_ready": true, "queue_depth": 0, "max_waiters": 4, "last_request_seconds_ago": <int>, "ollama_reachable": true, "ollama_loaded_models": ["gemma4:e2b"]}` (or whatever's in `/api/ps`).
- During an in-flight inference: `queue_depth >= 1` (at least one waiter — the in-flight call counts).
- After Ollama's `KEEP_ALIVE=300s` window expires (model unloaded): `ollama_loaded_models: []`, `ollama_reachable: true` (Ollama daemon up but no model loaded).
- With Ollama daemon stopped: `ollama_reachable: false`, `ollama_loaded_models: []`. HTTP status still 200 — `/state` reports the degraded state, doesn't propagate the underlying failure as a 502.
- `last_request_seconds_ago` reflects elapsed time since the last `InferenceService.run()` activity. Verified by: fire one inference at t=0, observe `/state` at t=10 returns `last_request_seconds_ago: 10` (±1 due to `int()` rounding).
- `/state` does NOT increment `LastRequestTracker` (idle-shutdown timer not reset by inspection). Verified by 100 successive `/state` calls, then check `tracker.last_finish()` did not advance.
- `/state` responds in under 200 ms wall-clock typical (in-process reads + one Ollama HTTP round-trip on a local connection — measured ~50 ms in practice).
- `/state` does NOT acquire `get_inference_semaphore()` — verified that calling `/state` while an inference is in-flight returns immediately (no blocking on the semaphore).
- `/openapi.json` documents `GET /state` with `responses={200: {"model": StateResponse}}`.
- The `OllamaClient.list_loaded_models()` method calls `GET /api/ps` with the expected URL; verified via test using `httpx.MockTransport`.
- The `OllamaClient.list_loaded_models()` method raises `AdapterConnectionFailureError` on httpx failures, in the same shape as `chat()`. Verified via the same exception-mapping AC battery as F003-of-E003.
- `LIP_MAX_INFERENCE_WAITERS=2` env var: `/state` reports `max_waiters: 2`. End-to-end env-var path verified.
- A registry-not-found request followed by a `/state` call: `last_request_seconds_ago` reflects the failed-but-attempted request (because F002-of-E005's `record()` is called at `run()` entry).
- Pyright strict passes; `task check` passes.

**`state_router.py`:**

```python
import time

from fastapi import APIRouter, Depends, Request

from app.api.deps import (
    get_last_request_tracker,
    get_ollama_client,
    get_settings,
    get_waiter_counter,
)
from app.api.state_response import StateResponse
from app.core.config import Settings
from app.exceptions import AdapterConnectionFailureError
from app.features.inference.repository.ollama_client import OllamaClient
from app.features.inference.service.last_request_tracker import LastRequestTracker
from app.features.inference.service.waiter_counter import WaiterCounter

router = APIRouter()


@router.get(
    "/state",
    responses={200: {"model": StateResponse}},
)
async def state(
    request: Request,
    settings: Settings = Depends(get_settings),
    waiter_counter: WaiterCounter = Depends(get_waiter_counter),
    tracker: LastRequestTracker = Depends(get_last_request_tracker),
    client: OllamaClient = Depends(get_ollama_client),
) -> StateResponse:
    is_ready = bool(getattr(request.app.state, "is_ready", False))
    last_seconds_ago = int(time.monotonic() - tracker.last_finish())
    try:
        loaded = await client.list_loaded_models()
        ollama_reachable = True
    except AdapterConnectionFailureError:
        loaded = []
        ollama_reachable = False
    return StateResponse(
        is_ready=is_ready,
        queue_depth=waiter_counter.current(),
        max_waiters=settings.max_inference_waiters,
        last_request_seconds_ago=last_seconds_ago,
        ollama_reachable=ollama_reachable,
        ollama_loaded_models=loaded,
    )
```

**`OllamaClient.list_loaded_models()` addition:**

```python
class OllamaClient:
    # __init__, close, _request, chat — unchanged from F001/F002/F003 of E003

    async def list_loaded_models(self) -> list[str]:
        """Probe Ollama's /api/ps for currently-loaded backend tags.

        Raises AdapterConnectionFailureError on any Ollama-specific failure,
        consistent with the chat() method's mapping.
        """
        import json  # already imported at module top in real code

        try:
            response = await self._request("GET", "/api/ps")
            response.raise_for_status()
            data = response.json()
            return [m["name"] for m in data.get("models", [])]
        except httpx.ConnectTimeout as exc:
            raise AdapterConnectionFailureError(
                backend="ollama", reason="connect_timeout"
            ) from exc
        except httpx.ConnectError as exc:
            raise AdapterConnectionFailureError(
                backend="ollama", reason="connection_refused"
            ) from exc
        except httpx.HTTPStatusError as exc:
            raise AdapterConnectionFailureError(
                backend="ollama", reason="non_2xx"
            ) from exc
        except (json.JSONDecodeError, KeyError) as exc:
            raise AdapterConnectionFailureError(
                backend="ollama", reason="malformed_response"
            ) from exc
        except httpx.RequestError as exc:
            raise AdapterConnectionFailureError(
                backend="ollama", reason="unknown"
            ) from exc
```

**Why a separate router file (`state_router.py`) instead of folding into `health_router.py`:** consistency with the existing pattern (each cross-cutting endpoint in its own router). Future feature evolution — e.g., splitting `/state` into multiple paths if the consolidated endpoint grows too large — keeps router edits localized.

**Why `last_request_seconds_ago` as a delta, not a timestamp:** `time.monotonic()` produces a relative tick-count, not a wall-clock timestamp. Returning a raw monotonic value would be meaningless to a consumer. The delta is the meaningful quantity ("X seconds since last activity"). If a future need for wall-clock surfaces (e.g., correlating with consumer logs), a separate `last_request_at: datetime` field could be added — but v1 keeps it simple.

**Why best-effort Ollama probe** rather than propagating the failure: the operator's mental model for `/state` is "tell me what's going on." If Ollama is down, that *is* what's going on, and the report should say so explicitly (`ollama_reachable: false`). A 502 from `/state` would be a worse signal — it conflates "the state endpoint is broken" with "Ollama is broken." Keeping the endpoint always-200 makes diagnostics unambiguous.

**Why `int()` truncation on `last_request_seconds_ago`:** sub-second precision is operationally meaningless ("how long ago was the last inference" with millisecond precision adds noise). `int()` produces clean integer seconds.

**`OllamaClient.list_loaded_models()` follows F003-of-E003's failure-mapping pattern**: same five `except` clauses, same `AdapterConnectionFailureError(backend="ollama", reason=...)` shape. The duplication is intentional — refactoring out a shared decorator/helper is a future cleanup; v1 keeps the explicit pattern visible.

**Pyright strict typing:** all standard library + Pydantic types are fully annotated. No `# type: ignore` needed.
