## Short description

Human-written documentation of the request envelope, model registry, sampling-default-and-override semantics, error response shapes, semaphore + queue + timeout behavior, on-demand lifecycle (launch-and-poll-`/health`), and Ollama daemon prerequisites.

## Problem

OpenAPI documents *shapes* — field names, types, status codes, response schemas. It does not document *meaning*: why `params: ModelParams` defaults to all-`None`, what consumer-supplied `temperature=None` does (opt-out, not "use default"), why `model: str` accepts logical names like `"default-task"` instead of Ollama tags, what HTTP 503 from `/v1/inference` implies the consumer should do (retry-with-backoff vs surface to user), why `/health` reports 503 right after launch (warm-up window) and how long to expect, how the 10-minute idle-shutdown interacts with consumer retry logic, what prerequisites must be in place for any inference to work at all (Ollama running as a launchd agent).

Without F001, future-Cosmin starting a new consumer project would either re-read every feature's source code or re-derive these semantics by trial and error. F001 is the single human-written document that captures the conceptual model so a 30-minute read produces a working consumer integration.

The consumer-integration snippet (in F002 of E007) is the executable counterpart — copy-paste-able starting code that materializes everything F001 explains. Together they implement G8 (consumers integrate without reading the service's source).

*Stakeholder:* Cosmin in the consumer-side-developer hat, *one to twelve months from now*, starting a new consumer project. *Need:* a conceptual model that lets him integrate against LIP without remembering implementation details, plus a set of operational expectations (when can I send my first request? what happens if I send too many? what if Ollama goes down mid-call?).

## Scope

- **`README.md` at the project root** (replaces the post-bootstrap generic README). Project-root location is the canonical landing page for the repo; consumers cloning land here. Epic Q2 resolved: project root over feature-slice root because LIP has effectively one consumer-facing surface (the inference endpoint), and a feature-scoped README would split discoverability.
- **Section structure** (every section explicit and titled — no "miscellaneous"):
    1. **Title + one-paragraph summary** — what LIP is, who runs it, on what hardware, in 3–5 sentences.
    2. **Prerequisites** — Ollama installed via Homebrew (`brew install ollama`), `gemma4:e2b` model pulled (`ollama pull gemma4:e2b`), Ollama configured as a launchd agent (link to `docs/ollama-launchd.md`), Python 3.13 + `uv`. Every later section assumes these.
    3. **Quick start** — three commands: `task ollama:install`; `task dev`; consumer-side `curl -X POST -H "Content-Type: application/json" -d '{"messages":[{"role":"user","content":"hello"}],"model":"default-task"}' http://localhost:8000/v1/inference`. Plus the link to `/docs` (Swagger UI) and `/redoc`.
    4. **Conceptual model** subsections:
        - **Request envelope.** Fields: `messages: list[Message]` (role + content; multimodal-ready), `model: str` (logical name, never a backend tag), `params: ModelParams` (sampling overrides), `metadata: dict[str, Any]` (passthrough for future per-project controls). Inline JSON example.
        - **Model registry.** Logical names map to backend tags; v1 ships one entry, `"default-task"` → `"gemma4:e2b"`. Capabilities (`text|image|audio`) and `context_max=131072` documented. Why this layer exists (G1, G2).
        - **Sampling defaults and overrides.** Registry's `sampling_defaults` (`temperature=0.0` for cognitive-task workload) merge with consumer-supplied `params`: consumer-set fields override; consumer-unset fields fall through. Explicit `None` from consumer = opt-out. Inline merge example showing the three cases.
        - **Capability validation.** If a consumer sends an `ImageContent` part to a text-only model, the service rejects with `MODEL_CAPABILITY_NOT_SUPPORTED` *before* any Ollama call. (v1's `default-task` supports all three modalities, so this triggers only with future text-only entries.)
    5. **Response envelope.** `content: str` plus 8-field `metadata` (`model` = logical name returned, `prompt_tokens`, `completion_tokens`, `request_id`, `latency_ms`, `queue_wait_ms`, `finish_reason`, `backend`). Why each field is there. Inline example.
    6. **Error responses (RFC 7807).** All errors are `application/problem+json` with five RFC 7807 fields plus `code` + `request_id` extensions plus typed params. Table mapping error code → HTTP status → meaning → recommended consumer action:
        - `VALIDATION_FAILED` (422) — fix request, retry never (deterministic).
        - `REGISTRY_NOT_FOUND` (404) — fix the `model` field; logical name unknown.
        - `MODEL_CAPABILITY_NOT_SUPPORTED` (422) — content type unsupported by model; either change content or change model.
        - `QUEUE_FULL` (503) — retry with backoff; queue is at `max_waiters`.
        - `INFERENCE_TIMEOUT` (504) — retry once; if it persists, investigate (likely a degenerate prompt).
        - `ADAPTER_CONNECTION_FAILURE` (502) — retry with backoff; Ollama may be down or restarting.
        - `INTERNAL_ERROR` (500) — file a bug; this should never happen in steady state.
    7. **Backpressure & timeouts.** `asyncio.Semaphore(1)` serializes inference. `MAX_INFERENCE_WAITERS=4` caps queue depth. `INFERENCE_TIMEOUT_SECONDS=180` caps per-request inference duration. What these mean for a consumer issuing N concurrent requests: 4 will wait, the 5th gets immediate 503; once running, every request has 180 s to complete or 504. Both env-tunable per `pydantic-settings`.
    8. **Lifecycle.** LIP is on-demand. Consumer is responsible for *launching* the FastAPI process (`task dev` or however the consumer wraps process management). After launch, consumer polls `GET /health` until 200; on 503, wait 1 s and retry. Once 200, send inference. After 10 minutes of no inference traffic, LIP self-shuts via SIGTERM; the next consumer that needs inference re-launches. Ollama stays running independently (launchd-managed, KEEP_ALIVE=300 s for model unload).
    9. **Operational endpoints.** `GET /health` (liveness + readiness); `GET /state` (queue depth, max waiters, last-request seconds-ago, Ollama reachable, loaded models). `GET /docs` (Swagger UI), `GET /redoc` (Redoc UI). All on the same `0.0.0.0:8000` port (or whatever `task dev` binds).
    10. **Stack & conventions.** Python 3.13 + `uv`; FastAPI + Pydantic v2 + structlog + httpx; Ollama as v1 backend, replaceable per G1; vertical-slice architecture per CLAUDE.md; one class per file; TDD; pyright strict; `task check` before any commit.
    11. **Related docs.** Link to `docs/ollama-launchd.md` (launchd setup), `graphs/LIP/` (Project + Epic + Feature spec tree for anyone wanting to understand the design rationale), `CLAUDE.md` (AI-assisted-dev discipline contract).
- **Inline code examples**: every conceptual model subsection has at least one minimal JSON request/response or Python httpx snippet. Examples are *real* (correct field names, correct envelope shape, correct response shape) — verified by being copy-paste runnable against the running service.
- **Tone**: matter-of-fact, no marketing language, no emojis. Written for one reader (future-Cosmin). Treats the reader as already familiar with FastAPI / Pydantic / asyncio basics.
- **OpenAPI exposure decision** (Epic Q1 resolved): keep FastAPI defaults — both `/docs` (Swagger UI) and `/redoc` (Redoc UI) are exposed. README documents both URLs. No production-disabling logic in v1 (no production/dev distinction; single deployment trusts local-network callers).
- **Length target**: 600–1200 lines including code blocks. Long enough to be complete; short enough to be readable in 30 minutes. The detailed feature specs in `graphs/LIP/` cover everything in finer detail; README is the synthesis.

**Markdown structure conventions:**

- H1 (one) for the title — "Local Inference Provider (LIP)".
- H2 for top-level sections (Prerequisites, Quick start, Conceptual model, etc.).
- H3 for subsections within Conceptual model (Request envelope, Model registry, Sampling defaults and overrides, Capability validation).
- H3 for subsections within other sections only when the section length warrants it.
- Inline code spans for paths, identifiers, env vars, status codes (e.g., `/health`, `MAX_INFERENCE_WAITERS`, `503`).
- Triple-backtick code blocks for JSON examples (with `json` language tag) and Python snippets (with `python` language tag) and shell commands (with `bash` or `shell`).
- Tables for the error-code table; markdown's pipe-syntax format.
- Internal links via section-anchor references (`[Backpressure & timeouts](#backpressure--timeouts)`) for navigation.
- External links inline; no separate references section.

**Code-example correctness:**

Every JSON request/response example in the README is generated by hand but verified against the live service during implementation. The acceptance criterion that says "all field names match the actual schemas" means: during F001's implementation, the developer runs each example and pastes the actual response, ensuring no name drift between docs and code.

**No README content generation from code:** the README is hand-written. FastAPI's `/openapi.json` is the canonical machine-readable contract; the README is the human-readable conceptual narrative. Auto-generation tooling (e.g., scraping `errors.yaml` to build the error-code table) is out of v1 scope. Drift risk is mitigated by short feedback loops (one developer, one repo) and by the linting gates (Pyright strict catches schema renames before they reach prod).

**No CHANGELOG / version history**: git log + commit messages are sufficient for a one-developer project. Adding a CHANGELOG.md is a future concern if external consumers ever care about release notes.

**No CONTRIBUTING.md / CODE_OF_CONDUCT**: one-developer project; out of v1.

**Link-rot mitigation**: external links to FastAPI / Pydantic / Ollama docs use the project's homepage (e.g., `https://docs.ollama.com/`) rather than deep-linking to a specific anchor that may move. Internal links use markdown's `#section-anchor` syntax which is stable as long as section titles don't change.

**Tone discipline**: no emojis (per CLAUDE.md). No marketing-style superlatives ("blazingly fast", "world-class"). Plain technical English. Active voice.
