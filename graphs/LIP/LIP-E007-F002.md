## Short description

Concrete Python code a consumer project can copy as a starting point — wakes the service, polls `/health`, issues an inference call, handles errors per the documented response shape.

## Problem

The README (F001 of E007) explains LIP's conceptual model and error contract narratively. F002 ships the *executable* counterpart: a single self-contained Python file that demonstrates every piece of the consumer-integration story end-to-end. A new consumer project starts by copying `examples/consumer.py`, adapting the prompt-construction logic and response-handling to its domain, and removing what it doesn't need. The README explains *what* to do; F002 of E007 shows *exactly how* in working code.

Without F002, a consumer-side developer integrating against LIP must transliterate prose into code — inferring the right `httpx` configuration (base URL, timeouts), the right wake-up loop, the right retry strategy per error code. Each consumer would re-derive the same patterns slightly differently. F002 is the canonical reference implementation that makes those patterns concrete and copy-pasteable.

The snippet is *intentionally complete and small* — every consumer pattern in one file (~80 lines of code), no abstractions, no separate modules. A consumer project can lift the file wholesale or cherry-pick the pieces it needs.

*Stakeholder:* Cosmin in the consumer-side-developer hat, *one to twelve months from now*. *Need:* a starting point that compiles and runs, demonstrates the canonical wake-up + retry + error-handling patterns, and is short enough to read in under five minutes.

## Scope

- **Single Python file at `examples/consumer.py`** (project root `examples/` directory). One file, no submodules, no shared utilities, no `__init__.py`.
- **Prerequisites in module docstring**: LIP running at the configured base URL, Ollama running (referenced by README link), Python 3.13 + `httpx` available (the LIP repo's `pyproject.toml` already includes httpx, so `uv run python examples/consumer.py …` just works).
- **Demonstrated patterns:**
    - **`wait_for_ready(client, max_attempts=30)`** — polls `GET /health`, returns when 200, raises after `max_attempts` retries (default 30 = 30 s if 1 s sleep per attempt). Catches `httpx.RequestError` (connection refused = LIP not yet listening) and 503 (warm-up pending) symmetrically — both trigger a 1 s sleep and retry.
    - **`infer(client, prompt)`** — sends `POST /v1/inference` with the canonical envelope (single user message, `model="default-task"`, default params). Reads HTTP status; on 200 returns `response.content`; on retryable codes (`QUEUE_FULL`, `INFERENCE_TIMEOUT`, `RATE_LIMITED`, `ADAPTER_CONNECTION_FAILURE`) retries with exponential backoff (2^attempt seconds) up to 3 attempts; on any other error parses the RFC 7807 `code` and raises `RuntimeError` with the code + detail.
    - **`main()`** — entrypoint. Reads prompt from `sys.argv[1]` (or default), constructs `httpx.AsyncClient`, calls `wait_for_ready`, calls `infer`, prints the response content.
    - **Module-as-script invocation**: `if __name__ == "__main__": asyncio.run(main())`.
- **Configuration**: hardcoded `LIP_URL = "http://localhost:8000"` at module top, with a comment noting "edit for your deployment." `LIP_TIMEOUT_SECONDS = 200` (client timeout slightly larger than the server's `INFERENCE_TIMEOUT_SECONDS=180` default — so the *server's* 504 reaches the client before the client's own timeout fires).
- **Error handling**: covers all 10 error codes from `errors.yaml` (5 generic + 5 LIP-specific):
    - `VALIDATION_FAILED`, `NOT_FOUND`, `CONFLICT`, `INTERNAL_ERROR`, `REGISTRY_NOT_FOUND`, `MODEL_CAPABILITY_NOT_SUPPORTED` — surfaced as `RuntimeError` with `code` and `detail` (consumer programs decide what to do; for an example snippet, surfacing is enough).
    - `QUEUE_FULL`, `INFERENCE_TIMEOUT`, `RATE_LIMITED`, `ADAPTER_CONNECTION_FAILURE` — retried with exponential backoff (2^attempt seconds: 1, 2, 4) for up to 3 attempts; if all retries fail, surfaces `RuntimeError`.
- **No tests for this file.** Per CLAUDE.md "TDD always", every implementation file has a paired test. F002 of E007 is *demonstration code*, explicitly not part of the running service. The file lives in `examples/` (outside `apps/backend/app/`), is excluded from `pyproject.toml`'s `tool.pyright.include` and pytest collection, and is documented as such in a comment at the top. (CLAUDE.md's TDD discipline applies to the *service*; example code in `examples/` is repository documentation in code form, not service code.)
- **Pyright strict still applies** — the snippet is included in the strict typing scope. Demonstration code that doesn't typecheck would fail `task check` and is unacceptable for a reference implementation.
- **README cross-link**: F001 of E007 references `examples/consumer.py` from the Quick start section. F002 ensures the file exists at that path.

## Out of scope

- **Multi-language examples** (Go, Rust, etc.) — Python only in v1 per Project Boundary.
- **Async-first vs sync example** — async only. The snippet uses `httpx.AsyncClient` because LIP's own code is async; a sync example would diverge from the project's idiom and be more complex (sync httpx in a sync `def main()` plus a separate retry loop).
- **A reusable client library / package** — F002 is a script, not a library. Future "LIP Python client" packaging is out of v1 (and explicitly rejected per Project Rejected Directions).
- **Multi-turn conversation example** — the snippet sends a single user message. Multi-turn (conversation history with assistant turns) is consumer-decided; the envelope supports it, but the example doesn't demonstrate it.
- **Multimodal example** (image / audio content) — out of v1 example scope. The envelope supports `ContentPart` lists, but the example uses simple `content: str`. Adding a multimodal example would double the file size for a use-case most consumer projects won't touch initially.
- **Retry strategy beyond exponential backoff** — out of v1; no jitter, no circuit breakers, no per-error-code-tuned strategies. Exponential backoff with 3 attempts is the demonstration; consumers customize for their own SLOs.
- **Connection pool tuning** — `httpx.AsyncClient`'s defaults are fine for a one-shot script. Long-running consumer services tune pool size based on their own concurrency; that's their decision.
- **Auth / API keys** — out of v1 (local-network trust).
- **Streaming response handling** — out of v1 (no streaming).
- **Tool-call response handling** — out of v1.
- **Performance / latency measurement** — the snippet doesn't time itself. Consumers add their own measurement.
- **Logging integration** — uses `print()` for output. Production consumer code would integrate structlog or the consumer's existing logger; the snippet stays minimal.
- **Configuration via environment / config file** — module-level constants. Consumers swap to whatever config mechanism their project uses.
- **CLI argument parsing beyond `sys.argv[1]`** — no `argparse`; one positional argument (prompt) and that's it. Real consumers use `argparse` or `click`; not the demonstration's job.
- **Wake-up *launching* logic** (e.g., `subprocess.Popen("task dev")`) — out of v1; the consumer is responsible for *how* LIP is started (fork+exec, ssh, ansible, manual `task dev` in a separate terminal, etc.). The snippet only demonstrates the *post-launch* wake-up wait. Per the disambiguated idea, "consumers are responsible for launching the process and polling /health" — F002 demonstrates the polling, not the launching.

**Implementation pattern** (canonical form for AC verification):

```python
"""Example LIP consumer integration.

This is a complete, runnable example demonstrating consumer-side integration
with the Local Inference Provider (LIP) service. Copy as a starting point.

Prerequisites:
  - LIP running locally at LIP_URL (default http://localhost:8000).
  - Ollama running (see docs/ollama-launchd.md).

Run:
  uv run python examples/consumer.py "What is 2+2?"
"""
import asyncio
import sys

import httpx

LIP_URL = "http://localhost:8000"
# Slightly larger than the server's INFERENCE_TIMEOUT_SECONDS=180 default
# so the server's 504 reaches the client before the client's own timeout.
LIP_TIMEOUT_SECONDS = 200

RETRYABLE_CODES = {
    "QUEUE_FULL",
    "INFERENCE_TIMEOUT",
    "RATE_LIMITED",
    "ADAPTER_CONNECTION_FAILURE",
}


async def wait_for_ready(
    client: httpx.AsyncClient,
    *,
    max_attempts: int = 30,
) -> None:
    """Poll /health until LIP returns 200 (ready)."""
    for attempt in range(max_attempts):
        try:
            response = await client.get("/health", timeout=5.0)
            if response.status_code == 200:
                return
        except httpx.RequestError:
            pass  # LIP not yet listening; retry
        await asyncio.sleep(1)
    raise RuntimeError(
        f"LIP did not become ready within {max_attempts} seconds"
    )


async def infer(client: httpx.AsyncClient, prompt: str) -> str:
    """Send one inference request, return the model's response content."""
    body: dict[str, object] = {
        "messages": [{"role": "user", "content": prompt}],
        "model": "default-task",
    }
    for attempt in range(3):
        response = await client.post(
            "/v1/inference",
            json=body,
            timeout=LIP_TIMEOUT_SECONDS,
        )
        if response.status_code == 200:
            return response.json()["content"]
        problem = response.json()
        code = problem.get("code", "UNKNOWN")
        if code in RETRYABLE_CODES and attempt < 2:
            await asyncio.sleep(2 ** attempt)  # 1s, 2s, 4s
            continue
        raise RuntimeError(
            f"LIP error {code}: {problem.get('detail', '<no detail>')}"
        )
    raise RuntimeError("LIP exhausted retries")


async def main() -> None:
    prompt = sys.argv[1] if len(sys.argv) > 1 else "Say hello in one sentence."
    async with httpx.AsyncClient(base_url=LIP_URL) as client:
        await wait_for_ready(client)
        answer = await infer(client, prompt)
        print(answer)


if __name__ == "__main__":
    asyncio.run(main())
```

**Why a single file, not a package**: any consumer copying this code as a starting point will adapt heavily — extracting "their own" client class, integrating with their own logging, replacing `print` with their own output handling. Keeping the example as one file means one `cp` is the entire integration. A package-shaped example would imply F002 is a *library* consumers depend on, but per Project Rejected Directions, generated client libraries are out of v1 — and a hand-written one is the same shape.

**Why hardcoded `LIP_URL` instead of env-var lookup**: example simplicity. Real consumer projects use whatever configuration mechanism they already have (`pydantic-settings`, `os.environ`, config files); the example shouldn't impose its own. A comment near the constant documents that consumers swap it.

**Why exponential backoff at `2 ** attempt`** (1, 2, 4 seconds) and 3 attempts: matches typical retry-with-backoff defaults seen in `tenacity` and similar libraries. After 3 attempts (1+2+4=7 seconds total wait + 3 round-trips), if any of the four retryable codes (`QUEUE_FULL`, `INFERENCE_TIMEOUT`, `RATE_LIMITED`, `ADAPTER_CONNECTION_FAILURE`) is still firing, something is genuinely broken (queue is permanently full, Ollama is permanently down, or upstream rate limits are exhausted) and surfacing the error is the right move. Tunable per consumer; demonstration is the default.

**Why `LIP_TIMEOUT_SECONDS=200` (>180)**: see Insight at top of this thickening — client timeout must exceed the server's per-request timeout so the server's 504 (`INFERENCE_TIMEOUT`) reaches the client before the client's own timeout fires. With `client_timeout=200, server_timeout=180`, every server-issued 504 lands as a parseable RFC 7807 504 the consumer can program against. Reversed (`client_timeout=120, server_timeout=180`), the client gives up first and gets a `httpx.ReadTimeout` exception, losing the structured-error semantics.

**Why excluded from pytest**: the file is demonstration, not service code. Running it as a test would require LIP itself to be running (it'd be an integration test of the example, which is over-engineering for v1). Excluded via `pyproject.toml`:

```toml
[tool.pytest.ini_options]
testpaths = ["apps/backend/tests", "packages"]   # not "examples/"
```

(The exact exclusion mechanism may differ based on the repo's existing test config; the point is `examples/` is not a test root.)

**Why pyright-strict still applies to the example**: a typed example is documentation that the type system has accepted. A consumer reading the file should be able to copy it into a strict-typed project and have it pass. If `examples/consumer.py` had type errors, it would mislead consumers about the right shape. The `pyproject.toml` `[tool.pyright]` `include` adds `examples/`.

**Pyright strict considerations:**

- `httpx.AsyncClient(base_url=LIP_URL)` is fully typed.
- `response.json()` returns `Any` per httpx's typing — the `.get()` calls and `["content"]` access produce `Any`-typed values; the function's return annotation `-> str` requires either a `cast(str, …)` or accepting that strict-mode tolerates the implicit-Any flow. The example lives with `Any` for now; a real consumer typing this up would add Pydantic models for the response shapes (which is exactly what F001 of E001 already does on the server side; consumers can pull those types via OpenAPI codegen if they want).
- `body: dict[str, object]` — `object` is the safest broad type; Pyright accepts heterogeneous dict construction.
